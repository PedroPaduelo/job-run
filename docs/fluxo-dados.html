Fluxo de Dados e Informação

🔄 Como o dado flui no sistema

O JobSync processa jobs assíncronos de ponta a ponta. O ciclo completo inclui: criação → fila → processamento → callback → logging. Cada etapa é registrada no banco e pode ser monitorada em tempo real.

1. Fluxo Completo de um Job

1
📥 Cliente envia job

Usuário ou sistema externo envia uma requisição POST /api/jobs com:

  • queue: nome da fila destino
  • webhookUrl: onde o worker vai chamar
  • payload: dados (max 1MB, validado)
  • callbackUrl (opcional): onde notificar conclusão
2
🔐 Middleware valida

O onRequest hook do Fastify valida:

  • JWT token (ou API Key no header Authorization)
  • Organization ID (do JWT ou header X-Organization-Id)
  • Rate limit (por organization ou API Key)
  • IP allowlist (se configurado)
  • Schema do payload com Zod
3
🗄️ Job e fila são criados

jobService.create():

  1. Upsert da fila (cria se não existe)
  2. Cria registro Job no PostgreSQL (status PENDING ou SCHEDULED)
  3. Adiciona job no BullMQ queue (Redis)
4
📡 Worker consome da fila

Worker BullMQ (por organização) pega job da fila Redis:

  • Concorrência configurável por fila
  • Retry com backoff exponencial
  • Timeout configurável (default 5min)
5
🌐 Worker chama webhook

worker.ts faz HTTP request para webhookUrl:

  • Method configurável (GET, POST, PUT, PATCH, DELETE)
  • Body em JSON para métodos com payload
  • GET: payload vira query params
  • Resposta salva em Job.result
6
✅ Job é atualizado

PostgreSQL update com:

  • status: COMPLETED ou FAILED
  • result ou error
  • completedAt
  • Log em JobLog
7
📤 Callback (opcional)

Se callbackUrl fornecido, worker faz POST com:

  • jobId, status
  • result ou error
  • duration, attempts
  • Timeout de 5s (não crítico)

⏰ Fluxo de Recurring Jobs

Scheduler Automático

A
Scheduler roda a cada 30s

RecurringJobScheduler usa lock distribuído no Redis (60s TTL) para garantir apenas 1 instância executando em ambiente multi-worker.

B
Busca jobs due

SELECT * FROM recurring_jobs WHERE is_active = true AND next_run_at ≤ agora.

C
Aplica regras

Verifica: maxRuns atingido?, recurrenceEndDate passou?, continueOnFailure = false e último job falhou? Se sim, desativa.

D
Cria job + atualiza nextRunAt

Atomica: UPDATE recurring_jobs SET next_run_at = ? WHERE id = ?, então INSERT INTO jobs com recurrence_id.

E
Job entra na fila normalmente

O job criado segue o mesmo caminho dos jobs manuais (passos 3–7).

💾 Onde cada dado é persistido

Entidade Armazenamento Tempo de vida Propósito
Job PostgreSQL (jobs) Permanente (até cleanup admin) Registro histórico, auditoria, consulta
JobLog PostgreSQL (job_logs) Permanente Log de cada tentativa, debugging
Queue PostgreSQL (queues) + Redis (BullMQ) Config persistente, fila volátil Configuração + fila em memória
BullMQ Job Redis (bull:jobs-${org}-${queue}) Até ser processado (ou fail com limpeza) Fila de mensagens, estado em memória
AuditLog PostgreSQL (audit_logs) Permanente Compliance, rastreamento de ações
API Key PostgreSQL (api_keys) + cache Redis Permanente (até revoke) Autenticação M2M, rate limit por chave
RecurringJob PostgreSQL (recurring_jobs) Permanente Configuração de jobs periódicos
Scheduler Lock Redis (scheduler:lock) 60s TTL Evita múltiplos workers executando scheduler

✅ Validação de dados

Validação no Recebimento

  • • URL safety: Bloqueia localhost, IPs privados, requer HTTPS em prod
  • • Payload size: Máximo 1MB (JSON.stringify length)
  • • Headers: Authorization, Content-Type, X-Organization-Id
  • • Rate limit: Token bucket por org/API key
  • • IP allowlist: Pode bloquear IPs não autorizados
  • • Queue ativa: Fila deve estar isActive

Validação no Processamento

  • • Timeout: Worker aborta após timeout configurado
  • • Retry backoff: Exponential com delay configurável
  • • Max attempts: Padrão 3, configurável por fila
  • • Callback timeout: 5s, falha não crítica
  • • Response check: !response.ok → throw
  • • Content-Type: JSON aceito, texto salvo como { text }

🛡️ Tratamento de Erros

🇵 Fluss de erro durante processamento

❌
Webhook falha (5xx, timeout): Worker loga, atualiza status para PROCESSING (se vai retry) ou FAILED/DEAD (última tentativa).
🔄
Retry automático: BullMQ gerencia retry com backoff exponencial (delay configurável). JobLog registra cada tentativa.
📞
Callback falha: Timeout 5s, erro ignorado (não crítico).
✅
Sucesso: Job atualizado para COMPLETED, resultado salvo, callback enviado (se houver).

📈 Estados do Job

PENDING — aguardando na fila SCHEDULED — agendado para futuro PROCESSING — worker pegou COMPLETED — sucesso FAILED — falhou (pode retry) CANCELLED — cancelado pelo usuário DEAD — falhou todas tentativas

🔐 Auditoria

Toda ação relevante (auth, criação, cancelamento, erro) gera um AuditLog no PostgreSQL, com userId, organizationId, action, entityType, entityId, IP, user-agent e status.
