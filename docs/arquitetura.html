◾ Os Dois Processos Node.js

O sistema é composto por dois processos independentes que se comunicam exclusivamente via Redis (BullMQ). Eles podem ser escalados, reiniciados e monitorados de forma separada.

1

Processo: API Server

src/index.ts → app.ts

▶ Inicia servidor Fastify na porta 3003
▶ Registra plugins: cors, jwt, swagger, rateLimit
▶ Autentica via JWT externo (Auth Service) e API Keys M2M
▶ Enfileira jobs no Redis via BullMQ Queue.add()
▶ Persiste metadados de jobs no PostgreSQL via Prisma
▶ Expõe endpoints REST: /jobs, /queues, /recurring-jobs, /api-keys
▶ Usa 2 conexões Redis: uma para cache/rate-limit, outra exclusiva para BullMQ
Entrypoint: npm run start:api
Arquivo: src/index.ts
2

Processo: Worker

src/worker.ts

▶ Escuta filas BullMQ no Redis (Worker.process())
▶ Para cada job: envia HTTP POST para a callbackUrl do cliente
▶ Atualiza status do job no PostgreSQL: pending → processing → completed/failed
▶ Suporta retry automático com backoff configurável
▶ Executa o RecurringJobScheduler a cada 30s com lock distribuído Redis
▶ O scheduler usa SET NX EX no Redis para garantir execução única (sem duplicatas em múltiplas instâncias)
Entrypoint: npm run start:worker
Arquivo: src/worker.ts

◾ Padrão de Filas por Organização

Cada organização tem seu namespace isolado de filas no Redis. O padrão de nomeação garante isolamento multi-tenant sem conflito entre organizações diferentes.

Convenção de Nomeação

# Padrão de nome de fila
jobs-{orgId}-{queueName}
# Exemplos reais:
jobs-org_abc123-email-notifications
jobs-org_abc123-pdf-generation
jobs-org_xyz789-email-notifications
✓ Namespacing por organização garante isolamento total
✓ Uma org não pode acessar filas de outra org
✓ O queueName é escolhido pelo cliente no momento de criar a fila
✓ O Worker escaneia dinamicamente as filas existentes no Redis

◾ Connection Pool: 2 Conexões Redis

O sistema mantém duas conexões Redis separadas por design. BullMQ exige uma conexão dedicada que não pode ser compartilhada com operações gerais.

Conexão Redis #1
redisClient (geral)
▪
Cache de dados
Respostas temporárias, sessões
▪
Rate Limiting
Contadores por API Key / IP
▪
Scheduler Lock
SET NX EX para lock distribuído
Conexão Redis #2
bullmqRedis (dedicada)
▪
BullMQ Queue (API)
Publicar jobs nas filas
▪
BullMQ Worker (Worker)
Consumir e processar jobs
▪
Estado das Filas
Waiting, Active, Completed, Failed
⚠
Por que duas conexões?

BullMQ usa comandos Redis bloqueantes (BLPOP, scripts Lua, transações MULTI/EXEC) que não podem coexistir com operações gerais na mesma conexão. Misturar as conexões causaria deadlocks e comportamento imprevisível. A separação é obrigatória pela documentação do BullMQ.

◾ Fluxo de Autenticação

Dois mecanismos de autenticação coexistem: JWT emitido por serviço externo (para usuários humanos via Frontend) e API Keys M2M (para integrações machine-to-machine).

4
API compara hash, extrai orgId e permissões

◾ RecurringJobScheduler

O scheduler de jobs recorrentes roda dentro do processo Worker. Ele usa lock distribuído Redis para garantir execução única mesmo com múltiplas instâncias do Worker em execução.

⏱
Intervalo de Tick
30s

A cada 30 segundos o scheduler avalia quais recurring jobs precisam ser enfileirados

🔒
Lock Distribuído
SET scheduler:lock
  NX EX 25

Apenas 1 instância adquire o lock por ciclo. As demais pulam silenciosamente.

📄
Lógica de Execução
1.Busca recurring jobs ativos no DB
2.Verifica nextRunAt de cada job
3.Enfileira jobs prontos para execução
4.Calcula e persiste novo nextRunAt
5.Respeita maxRuns e endDate

◾ Decisões Arquiteturais

Por que Fastify e não Express?

Fastify oferece validação de schema nativa via JSON Schema, serialização JSON 2x mais rápida, plugin system com contexto encapsulado e suporte nativo a TypeScript. É o padrão moderno para APIs Node.js de alta performance.

Por que BullMQ e não processamento direto?

BullMQ garante: durabilidade (jobs não se perdem no reinício), retry automático com backoff, rate limiting, jobs agendados, prioridades, visibilidade do estado e escalabilidade horizontal sem código adicional.

Por que 2 processos separados?

Separação de responsabilidades e escalabilidade independente. Se o Worker está sobrecarregado, escala-se só ele. Se a API tem alta demanda, escala-se só ela. Reinicialização de um não afeta o outro.

Por que Auth Service externo?

Centralização de autenticação. O JobSync não gerencia senhas, MFA ou sessões de usuário. Delega ao Auth Service (que pode ser Keycloak, Auth0, ou serviço interno) e apenas consome o JWT resultante.
