🔗 Integrações Externas

Integrações Externas do JobSync

O JobSync depende de seis sistemas externos para funcionar: um serviço de autenticação JWT, webhooks configuráveis pelos clientes, callbacks de status, monitoramento de erros via Sentry, banco de dados PostgreSQL e cache/filas via Redis. Esta página documenta propósito, endpoints, políticas de falha e riscos de cada integração.

Auth Service Webhooks SSRF-safe Callbacks Sentry PostgreSQL Redis / BullMQ

Auth Service Externo

Autenticação JWT / Multi-org
Risco MÉDIO

Propósito

Valida tokens JWT de acesso, busca o perfil do usuário e os dados da organização para montar o contexto de multi-tenancy em cada requisição autenticada. A URL é configurada via AUTH_SERVER_URL (padrão http://localhost:3000 em dev).

Endpoints Consumidos

  • GET
    /.well-known/jwks.json

    Busca chaves públicas para verificar assinatura JWT. Resultado cacheado no Redis por 5 min.

  • GET
    /auth/profile

    Perfil do usuário autenticado (Bearer token obrigatório).

  • GET
    /organizations/

    Lista todas as organizações do usuário.

  • GET
    /organizations/{"{orgId}"}

    Dados completos de uma organização específica.

  • GET
    /organizations/{"{orgId}"}/members/me

    Membership e role do usuário na org (para controle de permissões).

Política de Falha

• JWKS indisponível: fallback para chave cacheada no Redis.

• Org não encontrada: criada localmente com dados mínimos.

• Aceita múltiplos issuers: nommand.com e localhost.

Variáveis de Ambiente

AUTH_SERVER_URL=https://auth.nommand.com

# dev: http://localhost:3000

Riscos

  • ▸ Indisponibilidade bloqueia todos os logins.
  • ▸ Cache JWKS pode retornar chave rotacionada invalidada.
  • ▸ Fallback de org pode criar dados inconsistentes.
🌐

Webhook Targets

URLs externas configuradas pelos clientes
Risco ALTO

Propósito

O worker executa chamadas HTTP para URLs arbitrárias fornecidas pelos clientes como webhookUrl de um job. Suporta todos os métodos (GET, POST, PUT, PATCH, DELETE) e headers customizados. A resposta JSON é salva em Job.result.

Proteção SSRF

Bloqueios ativos:

• 127.x.x.x — loopback

• 10.x.x.x — rede privada A

• 192.168.x.x — rede privada B

• 172.16.x.x – 172.31.x.x — rede privada C

• Produção exige HTTPS

Configuração

timeout: 5min (padrão)

método: configurável por job

headers: customizados por job

resposta: salva em Job.result

Política de Falha

• Erro HTTP 4xx/5xx: job marcado como failed.

• Timeout excedido: job marcado como failed.

• Retry automático via BullMQ conforme configuração da fila.

Riscos

  • ▸ SSRF se a validação de IP falhar por DNS rebinding.
  • ▸ Workers travados por targets lentos (timeout default 5min).
  • ▸ Exfiltração de dados internos via URL maliciosa.
  • ▸ Dados sensíveis em Job.result sem criptografia.
📬

Callback URLs

Notificação de status de jobs (opcional)
Risco BAIXO

Propósito

Ao concluir (com sucesso ou falha) um job que possui callbackUrl, o sistema envia um POST para essa URL com o resultado final. É opcional — jobs sem callbackUrl não disparam nenhuma notificação.

Payload Enviado

POST {"{callbackUrl}"}

{"{"}

"jobId": "...",

"status": "completed|failed",

"result": {...} | null,

"error": "..." | null,

"duration": 1234,

"attempts": 1

{"}"}

Política de Falha

• Timeout: 5 segundos.

• Falha na entrega: ignorada silenciosamente.

• Não há retry de callbacks.

• O status do job não é afetado pela falha do callback.

Riscos

  • ▸ Falha silenciosa — clientes não sabem se callback chegou.
  • ▸ Sem retry: notificações perdidas em outages temporários.
  • ▸ Mesmo vetor SSRF dos webhooks se não compartilhar validação.
🐛

Sentry — Monitoramento de Erros

Rastreamento de exceções em produção
Risco BAIXO

Propósito

Captura exceções não tratadas e erros HTTP 5xx, enriquecendo cada evento com contexto de request e identidade para facilitar o diagnóstico em produção. Integrado em server.ts (API) e worker.ts (processamento de jobs).

Contexto Enviado

url, method, statusCode

orgId, userId, requestId

stack trace completo

job metadata (worker)

Política de Falha

• Sentry indisponível: erro apenas logado localmente.

• Não bloqueia resposta da API.

• captureError é fire-and-forget.

Riscos

  • ▸ PII ou dados sensíveis no contexto de erro (userId, payload).
  • ▸ Quota do Sentry pode esgotar em picos de erro.
  • ▸ Sem impacto em latência ou disponibilidade da API.
🗄️

PostgreSQL

Banco de dados principal via Prisma
Risco ALTO

Propósito

Armazena toda a state persistente do sistema: organizações, usuários, API keys, filas, jobs, agendamentos recorrentes e logs de auditoria. Acesso exclusivo via Prisma ORM com connection pool gerenciado.

Configuração

Host: 217.216.81.188

Porta: 54321

Driver: Prisma (pool)

DATABASE_URL via env

Política de Falha

• Timeout de conexão: Prisma retorna erro imediatamente.

• API responde 503 se pool esgotado.

• Worker pause automático em falha de DB.

• Sem réplica de leitura configurada.

Riscos

  • ▸ SPOF: banco em host único, sem failover.
  • ▸ Pool esgotado em picos causa degradação total.
  • ▸ Migrations sem downtime não estão documentadas.
  • ▸ Sem monitoramento de slow queries ativo.
🔴

Redis

Cache, rate limit, locks e filas BullMQ
Risco ALTO

Propósito

Duas conexões Redis distintas: uma para cache e controle (JWKS, rate limit, scheduler lock) e outra exclusiva para o BullMQ (filas de jobs). Separação garante que falhas no BullMQ não afetam o cache e vice-versa.

Conexões e TTLs

// Conexão de cache

maxRetries: 3

JWKS cache TTL: 5 min

rate limit TTL: 1h

scheduler lock TTL: 60s

// Conexão BullMQ

maxRetries: null (infinito)

filas: job queues

REDIS_URL via env

Política de Falha

• Cache Redis down: JWKS re-buscado no Auth Service a cada request.

• Rate limit Redis down: requests podem passar sem controle.

• BullMQ Redis down: jobs param de ser processados.

• maxRetries=null no BullMQ: reconecta indefinidamente.

Riscos

  • ▸ SPOF: Redis único, sem sentinel/cluster.
  • ▸ Queda do Redis paralisa processamento de jobs.
  • ▸ Eviction de keys pode invalidar locks do scheduler.
  • ▸ Sem monitoramento de memória / eviction policy definida.
