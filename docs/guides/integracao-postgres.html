Integração PostgreSQL

Schema Prisma, connection pooling, indexes e estratégia de migração do JobSync

1. Conexão

Variável de Ambiente

A conexão com PostgreSQL é configurada através da variável DATABASE_URL:

DATABASE_URL=postgres://user:password@localhost:54321/job_assync
ℹ️
Porta padrão

O PostgreSQL compartilhado roda na porta 54321. Esta porta é usada em ambientes de desenvolvimento e staging.

Connection Pooling

O Prisma Client gerencia automaticamente o pool de conexões. Valores padrão (ajustáveis via connection string):

ParâmetroValor PadrãoDescrição
connection_limit20Máximo de conexões simultâneas no pool
pool_timeout10sTempo de espera por conexão disponível
connect_timeout5sTempo limite para estabelecer conexão
idle_timeout10sTempo que conexão ociosa permanece aberta

Para ajustar o pool, adicione parâmetros à connection string:

DATABASE_URL="postgres://...?connection_limit=50&pool_timeout=15"

Migrations

O JobSync usa Prisma db push (não migrations). O schema é sincronizado diretamente com o banco:

$ npx prisma db push
$ npx prisma generate
⚠️
Risco em Produção

db push não track mudanças e pode causar perda de dados ou schema drift. Use com cautela em produção. Considere migrações tradicionais para ambientes críticos.

2. Schema Prisma

O banco possui 8 modelos principais organizados para multi-tenancy, audit trail e processamento de jobs:

📋 User
idString (PK) emailString @unique firstNameString? lastNameString? displayNameString? avatarUrlString? statusUserStatus lastSyncAtDateTime createdAtDateTime updatedAtDateTime

Descrição: Sincronizado do Auth Service. ID é externo (não gerado localmente).

🏢 Organization
idString (PK) slugString @unique nameString logoUrlString? settingsJson? isActiveBoolean lastSyncAtDateTime createdAtDateTime updatedAtDateTime

Descrição: Tenant. Cada organização tem dados isolados. slug usado para identificação humana.

🔗 OrganizationMember
idString @default(cuid()) userIdString FK organizationIdString FK roleMemberRole (OWNER/ADMIN/MEMBER/VIEWER) permissionsString[] (overrides role) joinedAtDateTime lastSyncAtDateTime

Descrição: Join table usuário-organização com papel (role) e permissões customizadas.

🔑 ApiKey
idString @default(cuid()) keyString @unique (jsk_xxx) nameString descriptionString? organizationIdString FK permissionsString[] allowedDomainsString[] allowedIpsString[] rateLimitInt? isActiveBoolean expiresAtDateTime? lastUsedAtDateTime? usageCountInt createdAtDateTime updatedAtDateTime

Descrição: Autenticação M2M. key é única global. Permite rate limit por chave.

📊 Queue
idString @default(cuid()) nameString descriptionString? organizationIdString FK concurrencyInt (default 5) maxRetriesInt (default 3) retryDelayInt ms (default 5000) timeoutInt ms (default 300000) rateLimitInt? isActiveBoolean isPausedBoolean createdAtDateTime updatedAtDateTime

Descrição: Configuração da fila BullMQ. Unique constraint em (organizationId, name).

📦 Job
idString @default(cuid()) externalIdString? organizationIdString FK queueIdString FK userIdString? webhookUrlString ( destino HTTP) webhookMethodString (GET/POST/PUT/DELETE) callbackUrlString? payloadJson resultJson? errorString? statusJobStatus priorityInt attemptsInt maxAttemptsInt scheduledAtDateTime? startedAtDateTime? completedAtDateTime? metadataJson? tagsString[] createdAtDateTime updatedAtDateTime

Descrição: Entidade principal. Pode ter milhões de registros. FKs: orgId (CASCADE), queueId (CASCADE), userId (SET NULL).

📝 JobLog
idString @default(cuid()) jobIdString FK attemptInt statusString messageString? durationInt (ms) metadataJson? createdAtDateTime

Descrição: Logs por tentativa. jobId com CASCADE delete. Busca por job rápido com filtro createdAt.

🔍 AuditLog
idString @default(cuid()) userIdString? (FK SET NULL) organizationIdString? (FK SET NULL) actionString (ex: "job.created") entityTypeString? entityIdString? metadataJson? ipAddressString? userAgentString? statusAuditStatus createdAtDateTime

Descrição: Trail de auditoria para compliance (LGPD, SOC2). SET NULL preserva logs mesmo se usuário/org deletados.

AuditLog.organizationId Organization.id SET NULL Preserva audit log se organização deletada (compliance).

5. Retenção de Dados

⚠️
Crescimento do Banco

O JobSync não faz auto-purge. O DBA ou operador deve gerenciar retenção manualmente ou via scripts agendados.

TabelaRetençãoMotivoAção recomendada
jobs Ilimitada (manual) Jobs são registros de negócio. Usuário precisa para histórico e troubleshooting. Script periódico para arquivar/purge jobs antigos COMPLETED/FAILED (>1 ano?).
job_logs Ilimitada (manual) BullMQ remove do Redis, mas logs em DB são usados para debug detalhado. Agendar DELETE por job_id de jobs antigos. Ou TTL se Postgres + cron.
audit_logs Ilimitada (manual/compliance) LGPD/SOC2 podem exigir logs por anos. Consulte juridico. Arquivar para storage barato (S3, Glacier). Retenção mínima definida por compliance.
queues Permanente Configuração. Pouco crescimento. N/A
organizations Permanente (soft delete?) Histórico de customer. Considere isActive=false em vez de DELETE. Implementar soft delete. Nunca硬 delete.
api_keys Permanente (com expiração) Auditoria de uso. Revogação precisa de histórico. DELETE apenas após expiração + período de segurança.

Estratégia de Purge (Exemplo SQL)

Para JobLogs de jobs antigos:

-- Remover logs de jobs concluídos há mais de 90 dias
DELETE FROM job_logs jl
USING jobs j
WHERE jl.job_id = j.id
  AND j.status IN ('COMPLETED', 'FAILED', 'DEAD', 'CANCELLED')
  AND j.completed_at < NOW() - INTERVAL '90 days';

-- Remover jobs antigos (após purgar logs)
DELETE FROM jobs
WHERE status IN ('COMPLETED', 'FAILED', 'DEAD', 'CANCELLED')
  AND completed_at < NOW() - INTERVAL '180 days';

6. Estratégia de Migration

db push vs Migrations

O JobSync usa npx prisma db push em vez de prisma migrate dev. Isso significa:

  • Não rastreável: Não há histórico de migrações em prisma/migrations.
  • Instantâneo: O schema atual é aplicado diretamente ao banco.
  • Risco de drift: Se o schema mudar e alguém rodar db push em prod, alterações destrutivas podem ocorrer.
  • Zero-downtime: db push tenta fazer alterações sem bloquear (ADD COLUMN, ADD INDEX), mas DROP pode travar.
🚨
Produção: Use Migrations Tradicionais

Para produção, considere migrar para prisma migrate deploy. Gera migrations versionadas e rollback seguro.

Workflow Recomendado (a partir de db push)

# 1. Alterar schema.prisma
# 2. Aplicar ao banco de dev
$ npx prisma db push

# 3. Se quiser migração versionada, extraia ela:
$ npx prisma migrate diff --from-url $DEV_DB --to-schema-datamodel ./prisma/schema.prisma --script > ./migrations/$(date +%Y%m%d)_alteracao.sql

7. Performance e Escalabilidade

Volume Esperado

O JobSync deve suportar milhões de jobs:

  • Leituras: Dashboard de jobs (paginação, filtros por status/queue).
  • Escritas: Inserção de Job + JobLogs (1+ por retry).
  • Picos: Bulk creation de jobs (ex: importação).

Considerações

ÁreaOtimização
Indexes Todos os FK têm index via @@index. Query principais usam composite indexes (status + orgId + createdAt).
Connection Pool 20 conexões padrão. Ajuste conforme concorrência de workers (2-3x workers).
JSON columns payload, result, metadata são JsonB. Não indexáveis (Postgres 14+ tem GIN indexes opcionais).
TIMESTAMPS Todos os timestamps com default now(). Use UTC para agendamento.
Pagination Use cursor-based paginação (createdAt + id) ao invés de OFFSET para pages grandes.
Partitioning Considere partition por organizationId ou createdAt se >50M rows em jobs/job_logs.

Query Patterns Críticas

-- Dashboard: Listar jobs da org, paginado, por fila/status
SELECT * FROM jobs
WHERE organization_id = $1
  AND (:queueId IS NULL OR queue_id = $2)
  AND (:status IS NULL OR status = $3)
ORDER BY created_at DESC, id DESC
LIMIT $4 OFFSET $5;

-- Ver logs de um job (indexação por jobId + createdAt DESC)
SELECT * FROM job_logs
WHERE job_id = $1
ORDER BY attempt, created_at;
ℹ️
PostgreSQL Compartilhado

O uso do PostgreSQL na porta 54321 pressupõe ambiente compartilhado. Monitore uso de recursos (CPU, IO) e considere instância dedicada se volume for alto.

8. Backups e DR

O JobSync não fornece backups automáticos. A responsabilidade é do operador/cliente:

  • pg_dump: Backup lógico completo (schema + dados).
  • WAL archiving: Point-in-time recovery (PITR) para recover rápida.
  • Frequência: Diária (full) + hourly (WAL/incial).
  • Retenção: 30 dias (ajustável por compliance).
  • Teste restore: Mensalmente, em ambiente de staging.

Exemplo de Backup Script

#!/bin/bash
# backup.sh
BACKUP_DIR="/backups/postgres"
DATE=$(date +%Y%m%d_%H%M%S)
DB_NAME="job_assync"

pg_dump $DB_NAME | gzip > "$BACKUP_DIR/backup_${DATE}.sql.gz"

# Retain last 30 days
find $BACKUP_DIR -name "*.sql.gz" -mtime +30 -delete

High Availability

Para produção crítica, considere:

  • Replicação streaming (1+ réplicas).
  • Failover automático com Patroni ou similar.
  • Backup cross-region.
