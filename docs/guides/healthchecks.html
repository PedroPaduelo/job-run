Health Checks do JobSync

Documentação técnica dos endpoints de health check, liveness e readiness probes para Kubernetes.

Visao Geral

O JobSync implementa dois endpoints de health check para coordenacao com o Kubernetes: liveness probe e readiness probe. Ambos estao implementados no arquivo server.ts (linhas 66-97) como route handlers Fastify.

⚠️
Importante

Os endpoints de health check sao publicos e pulam a verificacao de autenticacao.

GET Liveness Probe

Verifica se o processo do servidor esta rodando. Simples e rapido, sem checar dependencias externas.

/health Sempre retorna 200 se o servidor estiver ativo

Resposta de Sucesso (HTTP 200)

{
  "status": "ok",
  "uptime": 1234.56
}

Caracteristicas

  • Propósito: Indicar ao Kubernetes que o processo esta vivo.
  • Checagem de dependencias: Nenhuma - apenas verifica se o servidor responde.
  • Performance: Ultra rapido, sem I/O externo.
  • Quando falha: Apenas se o processo travou completamente.

Exemplo com curl

curl -s http://localhost:3001/health | jq
{
  "status": "ok",
  "uptime": 456.78
}

GET Readiness Probe

Verifica se o servidor esta pronto para receber trafego, testando todas as dependencias criticas.

/ready Retorna 200 se pronto, 503 se dependencies falharem

Resposta de Sucesso (HTTP 200)

{
  "status": "ready",
  "redis": true,
  "database": true,
  "uptime": 1234.56
}

Resposta de Falha (HTTP 503)

{
  "status": "not_ready",
  "redis": false,
  "database": true,
  "uptime": 1234.56
}

Caracteristicas

  • Propósito: Indicar ao Kubernetes se o pod pode receber trafego.
  • Checagem de dependencias: Redis e PostgreSQL.
  • Performance: Requer duas conexoes externas.
  • Quando falha: Se Redis ou DB estao indisponiveis.

Exemplo com curl (sucesso)

curl -s http://localhost:3001/ready | jq
{
  "status": "ready",
  "redis": true,
  "database": true,
  "uptime": 789.12
}

Exemplo com curl (Redis falhou)

curl -s -w "\nHTTP Status: %{http_code}\n" http://localhost:3001/ready
{
  "status": "not_ready",
  "redis": false,
  "database": true
}
HTTP Status: 503

Dependencias Verificadas

O endpoint /ready verifica as seguintes dependencias criticas:

Redis

Verificado via redis.ping(). Deve retornar 'PONG'. O cliente Redis é instanciado em bullmq.ts como singleton.

BullMQ Queue

PostgreSQL

Verificado via prisma.$queryRaw`SELECT 1`. Testa conexao com o banco de dados.

Prisma ORM

O Que Nao E Verificado

Por design, os health checks nao verificam todas as dependencias do sistema. Isto mantém os probes leves e evita falsos negativos. As seguintes dependencias nao sao verificadas:

  • Auth Service: Servico de autenticacao externo nao é testado.
  • BullMQ Queue Health: Saude das filas (fila cheia, workers ocupados).
  • Worker Status: Estado dos workers de processamento de jobs.
  • Sentry Connectivity: Conectividade com servico de error tracking.
ℹ️
Decisao de Design

Limitamos as verificacoes ao essencial (Redis + DB). Servicos externos tem seu proprio health check.

Implementacao

Os endpoints sao definidos no arquivo server.ts (linhas 66-97). Ambos usam handlers assincronos Fastify e sao registrados antes dos hooks de autenticacao.

Codigo Fonte

// Health check - liveness (sempre retorna ok se server esta up)
fastify.get('/health', async () => {
  return { status: 'ok', uptime: process.uptime() };
});

// Readiness check - verifica dependencias
fastify.get('/ready', async (_, reply) => {
  try {
    const redisOk = (await redis.ping()) === 'PONG';
    const dbOk = await prisma.$queryRaw`SELECT 1`;

    if (!redisOk || !dbOk) {
      return reply.status(503).send({
        status: 'not_ready',
        redis: redisOk,
        database: !!dbOk,
      });
    }

    return {
      status: 'ready',
      redis: redisOk,
      database: !!dbOk,
      uptime: process.uptime(),
    };
  } catch (error) {
    return reply.status(503).send({
      status: 'not_ready',
      error: error instanceof Error ? error.message : 'Unknown error',
    });
  }
});

Autenticacao Ignorada

O hook de autenticacao (linha 100-104) explicitamente pula verificacao para estes endpoints:

fastify.addHook('onRequest', async (request, reply) => {
  // Skip auth for health checks
  if (request.url === '/health' || request.url === '/ready') {
    return;
  }
  // ... resto da autenticacao
});

Kubernetes: Liveness vs Readiness

Entender a diferenca entre estes probes e crucial para deploy em Kubernetes:

Liveness Probe

Determina se o container esta rodando. Se falhar, Kubernetes reinicia o container.

  • Simples e rapido
  • Nao checa dependencias
  • Evita loops de restart

Readiness Probe

Determina se o container esta pronto para receber trafego. Se falhar, pod e removido do service.

  • Checa dependencias
  • Controla trafego de entrada
  • Permite warm-up

Configuracao Kubernetes

livenessProbe:
  httpGet:
    path: /health
    port: 3001
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 3

readinessProbe:
  httpGet:
    path: /ready
    port: 3001
  initialDelaySeconds: 5
  periodSeconds: 5
  timeoutSeconds: 5
  failureThreshold: 3

Comportamento em Falhas

Cenario: Redis cai

  • /health: Continua retornando 200 OK (servidor vivo).
  • /ready: Retorna 503 Service Unavailable.
  • K8s: Pod e removido do servico, novo pod e iniciado.

Cenario: Database cai

  • /health: Continua 200 OK.
  • /ready: Retorna 503.
  • K8s: Pod e removido, novo pod substitui.

Cenario: Servidor trava (deadlock)

  • /health: Nao responde → timeout.
  • K8s: Reinicia container.
✅
Bom Comportamento

Se dependencias criticas caem, o pod para de receber trafego. Isso evita requests falhando e permite auto-recovery via Kubernetes.

Sugestoes de Melhoria

Possiveis expandoes para monitoramento mais completo:

Auth Service Health

Verificar conectividade com servico de autenticacao. Evita requests autenticados falhando.

Seguranca

Worker Pool Health

Verificar se workers de processamento estao ativos e nao bloquados.

Infra

Queue Latency Metrics

Retornar metricas de latencia das filas BullMQ (waiting, active counts).

Monitoramento

Referencia Rapida

Endpoint Metodo Sucesso Falha Purpose

Curl Commands

# Liveness probe
curl -s http://localhost:3001/health

# Readiness probe
curl -s http://localhost:3001/ready

# Com timeout (recomendado para K8s)
curl --max-time 3 http://localhost:3001/health

# Verificar status HTTP apenas
curl -o /dev/null -s -w "%{http_code}" http://localhost:3001/ready
