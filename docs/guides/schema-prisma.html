Modelo completo de dados: entidades, relações, multi-tenancy e constraints

Diagrama de Relacionamentos (ER)

O modelo de dados do JobSync é baseado em multi-tenancy via organizationId. Cada organização (tenant) tem seus dados isolados logicamente no mesmo banco.

Visão Geral dos Modelos

Tabelas principais, propósito e relacionamentos.

👤

User

users

Usuários humanos do sistema. Autenticação via OAuth PKCE. Pode pertencer a múltiplas organizações através de OrganizationMember.

Relações: OrganizationMember (1:N), Job (1:N), AuditLog (1:N)
🏢

Organization

organizations

Tenant do sistema. Isola todos os dados por organização. Usa slug único como identificador público.

Relações: OrganizationMember (1:N), ApiKey (1:N), Queue (1:N), Job (1:N), AuditLog (1:N)
🔗

OrganizationMember

organization_members

Tabela de junção many-to-many entre User e Organization. Armazena role e permissions por tenant.

Unique: [userId, organizationId]
🔑

ApiKey

api_keys

Chaves de autenticação M2M. Permitem acesso machine-to-machine sem usuário. Controlam permissões, domínios, IPs e rate limit.

Relação: Organization (N:1)
📋

Queue

queues

Filas de processamento BullMQ. Configuram concurrency, retry, timeout, rate limit. Nome único por organização.

Relações: Organization (N:1), Job (1:N)
📦

Job

jobs

Unidade de trabalho assíncrono. Armazena payload, webhook URL, status, tentativas. Processado por workers BullMQ.

Relações: Organization (N:1), Queue (N:1), User (N:1 nullable), JobLog (1:N)
📝

JobLog

job_logs

Log de auditoria de cada job. Registra transições de estado, mensagens, duração de processamento.

Relação: Job (N:1)
🔍

AuditLog

audit_logs

Log geral de ações no sistema. Rastreia quem fez o quê, quando, com metadata. Isolado por organização.

Relações: Organization (N:1 nullable), User (N:1 nullable)

Multi-Tenancy

A camada de multi-tenancy é implementada por tenant-scoping: todas as tabelas de dados operacionais (Queue, Job, ApiKey, AuditLog) possuem organizationId e são filtradas por esse campo em todas as queries.

Isolamento por OrganizationId

🔒

Como funciona

  • Cada requisição autenticada traz organizationId (via JWT ou API key).
  • Todos os serviços incluem where: { organizationId } nas queries.
  • Queue tem unique constraint (organizationId, name) garantindo nomes isolados.
  • Organization.isActive permite desativar um tenant (soft blocking).

Roles e Permissões

OrganizationMember armazena o papel (role) e permissões细cased por usuário na organização. Validação de autorização verifica:

  • role (OWNER, ADMIN, MEMBER, VIEWER)
  • permissions[] (ex: ["queue:read", "job:write"])
⚠️
Importante

Não existe row-level security nativa no PostgreSQL. O isolamento é garantido pela aplicação. Sempre incluir organizationId em cada query Prisma.

ApiKey como Autenticação M2M

ApiKey permite que serviços externos acessem a API sem um usuário humano. Funciona como "service account":

Funcionamento

  1. Cliente envia X-API-Key: {key} no cabeçalho.
  2. Middleware consulta ApiKey por key.
  3. Valida: isActive=true, expiresAt > now, allowedDomains, allowedIps.
  4. Extrai organizationId da key.
  5. Contexto de request passa a conter: { organizationId, apiKey }.
  6. Jobs criados por API key podem ter userId = null.

Permissões

ApiKey.permissions é um array de strings no formato "{resource}:{action}":

  • "queue:read" — listar/configurar filas
  • "job:write" — criar jobs
  • "job:read" — consultar jobs
ℹ️
Rastreabilidade

AuditLog registra ações feitas por API key com userId = null mas organizationId válido. ApiKey.createdById rastreia qual usuário criou a chave.

Constraints e Índices

Constraints garantem integridade referencial e unicidade. Índices otimizam queries frequentes.

Unique Constraints

  • User.email único
  • Organization.slug único
  • ApiKey.key único
  • OrganizationMember(userId, organizationId) único
  • Queue(organizationId, name) único

Foreign Keys com Cascade

  • OrganizationMember.userId → User.id (Cascade Delete)
  • OrganizationMember.organizationId → Organization.id (Cascade Delete)
  • ApiKey.organizationId → Organization.id (Cascade Delete)
  • Queue.organizationId → Organization.id (Cascade Delete)
  • Job.organizationId → Organization.id (Cascade Delete)
  • Job.queueId → Queue.id (Cascade Delete)
  • JobLog.jobId → Job.id (Cascade Delete)

Foreign Keys com SetNull

  • Job.userId → User.id (SetNull ao deletar usuário)
  • AuditLog.userId → User.id (SetNull)
  • AuditLog.organizationId → Organization.id (SetNull)

Migrations

O schema é gerenciado pelo Prisma Migrate. Comandos para gerenciar:

# Criar nova migration
npx prisma migrate dev --name nome_da_migration

# Aplicar migrations em produção
npx prisma migrate deploy

# Gerar client após alterar schema
npx prisma generate

# Resetar banco (DESENVOLVIMENTO ONLY)
npx prisma migrate reset
🔴
Nunca modifique banco produtivo diretamente

Toda alteração de schema deve passar por migration do Prisma. Sempre versionar prisma/migrations/ e testar em staging antes de deploy.

Exemplos de Queries Comuns

1. Listar filas de uma organização

const queues = await prisma.queue.findMany({
  where: { organizationId },
  include: { jobs: { take: 5, orderBy: { createdAt: 'desc' } } },
  orderBy: { createdAt: 'desc' },
});

2. Buscar job com logs

const job = await prisma.job.findUnique({
  where: { id: jobId },
  include: {
    logs: { orderBy: { createdAt: 'asc' } },
    queue: true,
    user: true,
  },
});

3. Estatísticas de uma fila (contadores por status)

const stats = await prisma.job.groupBy({
  by: ['status'],
  where: { queueId },
  _count: { status: true },
});

4. Criar job via API key (userId null)

const job = await prisma.job.create({
  data: {
    organizationId,
    queueId,
    userId: null,  // criado por API key
    webhookUrl,
    webhookMethod: 'POST',
    payload,
    status: 'PENDING',
    tags: ['api-created'],
  },
});

5. Registrar log de auditoria

await prisma.auditLog.create({
  data: {
    userId: user?.id,
    organizationId,
    action: 'CREATE_JOB',
    entityType: 'Job',
    entityId: job.id,
    metadata: { before: null, after: job },
    ipAddress: req.ip,
    userAgent: req.headers['user-agent'],
    status: 'SUCCESS',
  },
});

Job Status Transitions

PENDING ──(worker picks)──> PROCESSING ──(success)──> COMPLETED
    │                              │
    │                              └─(failure, retry remaining)──> FAILED ──(retry backoff)──> PENDING
    │                                                                                   │
    └─(cancel)───────────────────────────────────────────────────────────────────────────┘
                                                                                         │
                                                                                         └─(max retries)──> DEAD

Outros:
  SCHEDULED ──(time reached)──> PENDING
  PROCESSING/CANCELLED/DEAD são estados finais ou transitórios controlados.
