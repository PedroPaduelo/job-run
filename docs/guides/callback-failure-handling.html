Callback Failure Handling

O que acontece quando o callbackUrl do cliente falha

🤔 O Problema

Quando o callbackUrl configurado pelo cliente está indisponível, a requisição falha. O código atual trata isso com um try/catch que silencia a falha — apenas loga no console.

⚠️
Silent Failure

O job é marcado como COMPLETED mesmo se o callback falhar. O cliente não sabe que sua notificação não foi entregue.

📊 Cenário de Falha

  1. Job processa com sucesso (webhook retorna 200)
  2. Worker tenta POST para callbackUrl
  3. callbackUrl está down (timeout, DNS, 5xx)
  4. catch log: console.error('Callback failed for job...')
  5. Job permanece COMPLETED — cliente nunca sabe

💥 Impacto

  • Perda de notificações — cliente não recebe resultado
  • Inconsistência de estado — sistema acha que delivered, mas não foi
  • Debug impossível — não há retry, não há métrica de callback failures

🛡️ Mitigações Atuais

// worker.ts (linhas 98-114)
if (callbackUrl) {
  try {
    await fetch(callbackUrl, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        jobId,
        status: 'COMPLETED',
        result,
        duration,
        attempts: job.attemptsMade,
      }),
    });
  } catch (callbackError) {
    console.error(`Callback failed for job ${jobId}:`, callbackError);
  }
}

Apenas log, sem:

  • Retry automático
  • DLQ para callbacks falhados
  • Métrica/alert
  • Atualização no job (callbackStatus)

✅ Recomendações

  1. Adicionar campo callbackStatus no Job (PENDING, SUCCESS, FAILED, RETRYING)
  2. Retry com backoff — usar BullMQ para schedule retry do callback
  3. Dead Letter Queue após maxAttempts, marcar job como CALLBACK_FAILED
  4. Métrica — contador de callback failures por organização
  5. Webhook de alerta — notificar admin quando callback falhar repeatedly
  6. Tempo limite menor — callback timeout deve ser configurável (default 5s)

📈 Diagrama de Fluxo

Job COMPLETED
    │
    ├─► Callback POST
    │   ├─► 200 OK → ✅ callbackStatus: SUCCESS
    │   ├─► Timeout/5xx → 🔄 retry (3x)
    │   │   └─► esgota → ❌ callbackStatus: FAILED
    │   └─► network error → 🛑 log only (ATUAL)
    │
    └─► Job fica COMPLETED independente do callback

🔍 Checklist

  • Callback failures são logados? ✅
  • Callback failures são metricados? ❌
  • Há retry para callbacks? ❌
  • Status do callback é salvo no DB? ❌
  • Alertas configurados? ❌
