Connection Pooling

Prisma e Redis pool tuning para alta concorrência

🔌 O Problema

Cada requisição HTTP abre conexão DB eRedis. Sem pool adequado, threads/processos ficam sem conexões sob carga.

💧
Pool Padrão

Prisma usa connection pool via node-postgres. Redis (ioredis) também tem pool configurável.

📊 Configuração Prisma

// .env
DATABASE_URL="postgresql://...?connection_limit=20"

Parâmetros de connection string:

  • connection_limit — máximo de conexões no pool (default 10)
  • pool_timeout — tempo máximo espera por conexão (default 10s)
  • idle_timeout — fecha conexões ociosas após X ms
  • max_uses — quantas queries uma conexão serve antes de recycle

📊 Configuração Redis (ioredis)

// src/lib/redis.ts
const redis = new Redis({
  host: process.env.REDIS_HOST,
  port: parseInt(process.env.REDIS_PORT || '6379'),
  maxRetriesPerRequest: 20,    // retry se conexão cai
  retryStrategy: (times) => Math.min(times * 100, 3000),
  family: 4,                   // IPv4
  keepAlive: true,
  connectTimeout: 10000,
  commandTimeout: 5000,
  // Não há maxPoolSize exposto no ioredis v5 — usa keepAlive +connection reuse
});

💥 Sintomas de Pool Insuficiente

  • Prisma: Error: Connection pool timed out
  • Redis: Max retries reached for command
  • Latência sobe (requests waiting por conexão)
  • Timeouts aleatórios em load spikes

✅ Recomendações

  1. Sizing do pool DB: 
    connection_limit = min(100, (CPU cores * 2) + effective_spindle_count)
// Ex: 4 cores → 8-12 connections
// Evitar pool muito grande (PostgreSQL default max_connections = 100)
  2. Separar pools por workload:
    • HTTP requests (Fastify) — pool dedicated
    • Workers (processJob) — pool separado? BullMQ workers não usam HTTP, usam Prisma direto
    • Configurar pool por instância (prisma.$extends com datasource diferente)
  3. Redis keepAlive — já está, bom. Ajustar keepAliveTimeout se necessário
  4. Pool reuse — garantir que mesma instância Prisma/Redis é reused (singleton pattern)
  5. Monitor pool metrics — pg_stat_activity para DB, redis.clients() para Redis
  6. Graceful shutdown — no SIGTERM, fechar pool: await prisma.$disconnect()

  • Metrics de pool monitoradas? ❌
