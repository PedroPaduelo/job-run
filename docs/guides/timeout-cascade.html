Timeout Cascade

Timeout de webhook não respeitado em fetch de callback

⏱️ O Problema

O job tem queue.timeout (default 5min = 300000ms). Esse timeout é passado para AbortController no fetch do webhook. Porém, o callback não tem timeout próprio!

⏰
Cascade Failure

Se webhook demora 4min, callback bloco por mais 5min (padrão fetch) → job roda 9min, bloqueando worker.

📊 Fluxo Atual

async function processJob(job) {
  const { timeout } = job.data;  // queue.timeout
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), timeout);

  // 1. Webhook fetch COM timeout
  const response = await fetch( webhookUrl, { signal: controller.signal, ... } );

  clearTimeout(timeoutId);

  // 2. Callback fetch SEM timeout próprio
  if (callbackUrl) {
    await fetch(callbackUrl, {  // ← sem timeout!
      method: 'POST',
      body: JSON.stringify({ jobId, status: 'COMPLETED', ... })
    });
  }
}

💥 Impacto

  • Worker concorrência bloqueada (job demora 2× o esperado)
  • Queue active count fica alto, novos jobs esperam
  • Timeout do BullMQ pode expirar (job timeout) causando retry desnecessário

✅ Recomendações

  1. Timeout de callback separado — adicionar callbackTimeout (default 5s)
  2. AbortController para callback — não reusar o mesmo controller
  3. Não bloquear commit — callback deve ser fire-and-forget com retry async, ou seja, job já foi COMPLETED antes do callback
  4. Log se callback demorar — métrica de callback duration

📈 Diagrama de Tempos

queue.timeout = 300000ms (5min)
    │
    ├─► Webhook fetch (4min OK)
    │   └─► COMPLETED save
    │
    └─► Callback fetch (sem timeout → default fetch 5min?)
        └─► Worker bloqueado por +5min TOTAL = 9min

🔍 Checklist

  • Callback tem timeout? ❌
  • Callback usa AbortController? ❌
  • Callback é fire-and-forget? ❌ (aguarda await)
  • Métrica callback duration? ❌
  • Alerta se callback > 10s? ❌
