Integração Sentry do JobSync

Configuração e uso do Sentry para monitoramento de erros e performance no JobSync.

Configuração da Variável de Ambiente

Adicione o DSN do Sentry ao arquivo .env na raiz do projeto:

# Sentry DSN (obrigatório para habilitar monitoramento)
SENTRY_DSN=https://sua-chave-aqui@o123456.ingest.sentry.io/1234567

# Ambiente (development | production)
NODE_ENV=production
ℹ️
IMPORTANTE

O Sentry só envia erros em produção. Em desenvolvimento, os eventos são capturados mas não enviados (logados no console).

Configuração do initSentry()

Opção Valor Descrição
dsn process.env.SENTRY_DSN Chave DSN do Sentry. Se não definida, Sentry não inicializa.
environment NODE_ENV Ambiente: development ou production.
tracesSampleRate 1.0 dev, 0.1 prod Taxa de amostragem de transações (tracing). 10% em produção.
beforeSend null em NODE_ENV === 'development' Não envia eventos em dev (apenas loga no console).

Funções exportadas

  • initSentry() — Inicializa o Sentry. Deve ser chamado no início da aplicação. 
    export function initSentry() {
  if (!SENTRY_DSN) {
    console.log('[Sentry] DSN not configured, skipping initialization');
    return;
  }

  Sentry.init({
    dsn: SENTRY_DSN,
    environment: process.env.NODE_ENV || 'development',
    tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
    beforeSend(event) {
      if (process.env.NODE_ENV === 'development') {
        console.log('[Sentry] Event captured (not sent in dev):', event.message);
        return null;
      }
      return event;
    },
  });
  console.log('[Sentry] Initialized');
}
  • captureError(error, context?) — Captura erro manualmente, adicionando contexto extra (url, method, organizationId, userId). 
    export function captureError(error: Error, context?: Record<string, unknown>) {
  if (context) {
    Sentry.setContext('additional', context);
  }
  Sentry.captureException(error);
}
  • captureMessage(message, level) — Captura mensagens de log como evento no Sentry.
  • setUser(user) — Define usuário no contexto (parcialmente implementado, não usado no fluxo atual).
  • addBreadcrumb(message, category, data?) — Adiciona breadcrumb para rastrear ações anteriores.
  • sentryErrorHandler(error, request) — Handler de erro do Fastify com contexto da requisição. 
    export function sentryErrorHandler(error: Error, request: unknown) {
  Sentry.withScope((scope) => {
    scope.setExtra('request', request);
    Sentry.captureException(error);
  });
}
  • flushSentry() — Aguarda envio de eventos pendentes no shutdown. 
    export async function flushSentry() {
  await Sentry.close(2000);
}

Error Handler Global

Definido em server.ts (linha 220-256). Captura todos os erros não tratados:

fastify.setErrorHandler((error, request, reply) => {
  // Capturar erro no Sentry com contexto
  captureError(error, {
    url: request.url,
    method: request.method,
    organizationId: request.organizationId,
    userId: request.userId,
  });

  fastify.log.error(error);

  // Log audit
  if (request.organizationId) {
    logAudit({ /* ... audit log ... */ });
  }

  const statusCode = error.statusCode || 500;
  const message = process.env.NODE_ENV === 'production' && statusCode === 500
    ? 'Internal Server Error'
    : error.message;

  reply.status(statusCode).send({ error: message });
});

Shutdown Clean

No shutdown (SIGTERM, SIGINT), o sistema aguarda 2 segundos para enviar eventos pendentes:

process.on('SIGTERM', async () => {
  console.log('SIGTERM received, shutting down...');
  await flushSentry();  // ← aguarda envio
  await fastify.close();
  await prisma.$disconnect();
  process.exit(0);
});

Configuração do initSentry()

Opção Valor Descrição
dsn import.meta.env.VITE_SENTRY_DSN Chave DSN do Sentry via variável Vite.
environment import.meta.env.MODE Ambiente do Vite.
tracesSampleRate import.meta.env.PROD ? 0.1 : 1.0 RUM tracing: 10% prod, 100% dev.
replaysSessionSampleRate 0.1 Session replay: 10% das sessões.
replaysOnErrorSampleRate 1.0 Session replay em erros: sempre.
ignoreErrors Lista de erros ignorados ResizeObserver loop limit exceeded, etc.

Funcionalidades Frontend

  • Error Boundaries — ErrorBoundary exportado e usável para capturar erros de renderização React.
  • Performance Monitoring — Tracing automático de navegação SPA e renderizações.
  • Session Replay — Gravação de sessões em 10% dos usuários (ou sempre em erros).

3. Contexto Capturado

Backend (Fastify errors)

No error handler global, o contexto é capturado via captureError:

  • url — URL da requisição que falhou
  • method — Método HTTP (GET, POST, etc.)
  • organizationId — ID da organização do requester
  • userId — ID do usuário (se autenticado via JWT)

Frontend

  • setUser(id, email) — Implementado na lib mas não chamado (falta integração com Auth Context).
  • Breadcrumbs — addBreadcrumb() disponível mas não utilizado no código atual.
⚠️
GAPS IDENTIFICADOS

Não há chamada a setUser() no frontend apesar da função estar implementada. Pode-se adicionar no AuthContext após login. Breadcrumbs não são adicionados.

4. Eventos Monitorados

Tipos de eventos enviados ao Sentry

Tipo Fonte Descrição
Exception Backend error handler + Frontend ErrorBoundary Exceções não tratadas ou capturadas manualmente via captureError().
Handled Error Qualquer captureError() Erros capturados explicitamente no código (ex: falha de webhook).
Transaction HTTP requests (backend) + navegação (frontend) Transações de tracing com duração, URL, método HTTP.

O que é monitorado

  • API request failures — Todas as requisições que geram erro 4xx/5xx são enviadas ao Sentry.
  • Worker crashes — Não capturado atualmente (worker.ts não usa Sentry).
  • Auth sync failures — Erros no hook onRequest são capturados.
  • Database connection errors — Erros do Prisma caem no error handler.
  • Redis errors — Erros de conexão Redis são capturados.
🔴
WORKER NÃO INTEGRADO COM SENTRY

O arquivo worker.ts não importa ou usa Sentry. Erros de processamento de jobs só são logados no console e no banco (jobLog). Considere adicionar captura de erros no worker.

5. Gaps e Recomendações

Gap Impacto Recomendação
setUser não chamado no frontend Sem identificação de usuário nas exceções frontend Chamar setUser({ id, email }) no AuthContext após login
Sem breadcrumbs adicionados Falta rastreamento de ações anteriores ao erro Adicionar addBreadcrumb() em ações importantes (navegação, cliques)
Sem release version Impossível agrupar erros por versão Setar releases no init com git SHA ou package version
Worker sem Sentry Erros de processamento de jobs invisíveis no Sentry Importar e usar captureError() no catch do worker

Exemplo: Adicionar Release

import { execSync } from 'child_process';

const release = process.env.SENTRY_RELEASE ||
  execSync('git rev-parse --short HEAD').toString().trim();

Sentry.init({
  // ...
  release: `job-run@${release}`,
});

6. Alertas (Recomendação)

Configure alertas no Sentry para notificar sobre:

  • Novas Issues — Quando um novo tipo de erro surge.
  • Increase in Errors — Quando a taxa de erros sobe X% acima da média.
  • Performance Issues — Quando transações ficam lentas.

Alertas podem ser enviados por email, Slack, Discord, PagerDuty, etc.
