Segurança

Sistema de Audit Log

Rastreamento completo de todas as ações no sistema para segurança, compliance e debugging

O que é Audit Logging?

O Audit Log é um sistema de rastreamento que registra todas as ações significativas realizadas no JobSync. Ele responde à pergunta fundamental da segurança: "Quem fez o quê, quando e de onde?".

ℹ️
Importante

Todos os logs de auditoria são escritos diretamente no PostgreSQL. Eles persistirão mesmo que haja falha no processamento de uma requisição, garantindo que a trilha de auditoria seja completa.

Modelo AuditLog

Os logs de auditoria são armazenados na tabela audit_logs do PostgreSQL, definida no schema Prisma:

📋 AuditLog Modelo Prisma
id String (PK)
userId? String
organizationId? String
action String
entityType? String
entityId? String
metadata Json?
ipAddress? String
userAgent? String
status AuditStatus
createdAt DateTime

Relacionamentos

  • user User? @relation(fields: [userId], references: [id], onDelete: SetNull)
  • organization Organization? @relation(fields: [organizationId], references: [id], onDelete: SetNull)

Índices

Para consultas rápidas, os seguintes índices são criados no banco de dados:

ÍndiceCampo(s)Uso
@@index([userId])userIdBuscar logs de um usuário específico
@@index([organizationId])organizationIdFiltrar por organização (multi-tenancy)
@@index([action])actionAgrupar por tipo de ação
@@index([entityType, entityId])entityType + entityIdRastrear mudanças em uma entidade específica
@@index([createdAt])createdAtConsultas por período temporal

O que é Registrado

Vários tipos de eventos são automaticamente capturados pelo sistema. Cada um representa um evento de segurança relevante.

Eventos de Autenticação

🔐 user.auth.success SUCCESS
Descrição Autenticação JWT bem-sucedida
userId Obrigatório
organizationId Obrigatório
action "user.auth.success"
status SUCCESS
🔑 apikey.auth.failed DENIED
Descrição Falha na autenticação por API Key
organizationId Obrigatório
entityType "ApiKey"
entityId ID da API Key
action "apikey.auth.failed"
metadata.reason "inactive" | "expired" | "ip_not_allowed"
status DENIED

Eventos de autorização

🛡️ permission.denied DENIED
Descrição Usuário não tem permissão para uma ação
userId Opcional (API Key pode não ter)
organizationId Obrigatório
action "permission.denied"
metadata.required Permissão necessária (ex: "jobs:create")
metadata.had Array de permissões do usuário
status DENIED

Eventos de Erro

⚠️ error.occurred FAILURE
Descrição Erro não tratado no sistema
userId Opcional
organizationId Opcional
action "error.occurred"
metadata.url URL da requisição
metadata.method Método HTTP
metadata.message Mensagem do erro
status FAILURE

Função logAudit

A função logAudit é responsável por criar registros de auditoria. Ela é projetada como "fire-and-forget" — os erros são capturados silenciosamente para não impactar a requisição principal.

Assinatura da Função

export async function logAudit(params: {
  userId?: string;
  organizationId?: string;
  action: string;
  entityType?: string;
  entityId?: string;
  metadata?: Record;
  ipAddress?: string;
  userAgent?: string;
  status?: 'SUCCESS' | 'FAILURE' | 'DENIED';
}): Promise { ... }

Implementação

export async function logAudit(params: {
  userId?: string;
  organizationId?: string;
  action: string;
  entityType?: string;
  entityId?: string;
  metadata?: Record;
  ipAddress?: string;
  userAgent?: string;
  status?: 'SUCCESS' | 'FAILURE' | 'DENIED';
}): Promise {
  try {
    await prisma.auditLog.create({
      data: {
        userId: params.userId,
        organizationId: params.organizationId,
        action: params.action,
        entityType: params.entityType,
        entityId: params.entityId,
        metadata: params.metadata as Prisma.InputJsonValue | undefined,
        ipAddress: params.ipAddress,
        userAgent: params.userAgent,
        status: params.status || 'SUCCESS',
      },
    });
  } catch (error) {
    console.error('[Audit] Failed to log:', error);
    // Nunca lança erro — o log falhou mas a requisição continua
  }
}
⚠️
Fire-and-Forget

A função captura e silencia todos os erros (com um log no console). Isso significa que se o banco de dados estiver indisponível, a requisição continua normalmente — mas o log de auditoria será perdido.

Contexto Capturado

Para cada evento de auditoria, o sistema captura automaticamente o máximo de contexto possível:

🌐

Endereço IP

request.ip — Endereço IP do cliente que fez a requisição.

📋

User-Agent

request.headers['user-agent'] — Navegador ou cliente HTTP usado.

🔗

URL e Método

request.url e request.method — Endpoint acessado.

🏢

organizationId

Extraído do header X-Organization-Id ou da API Key / JWT.

👤

userId

ID do usuário autenticado (não disponível para API Keys puramente máquina).

🔐

Permissões

Permissão requisitada e permissões reais do usuário no caso de permission.denied.

Retenção de Dados

Atualmente, o sistema não possui uma política automatizada de limpeza para logs de auditoria. Isso significa que:

  • Todos os logs são mantidos indefinidamente no PostgreSQL
  • A tabela audit_logs pode crescer continuamente
  • Consultas de longo prazo podem se tornar lentas sem manutenção de índices
  • O custo de armazenamento aumenta progressivamente

Considerações de Retenção

Para ambientes de produção, considere implementar:

📅

Retenção por Período

Apagar logs com mais de 90/180/365 dias via job agendado (pg_cron ou worker externo).

🗄️

Arquivamento

Migrar logs antigos para tabela de arquivo morto ou data warehouse.

⚠️
Risco Identificado

Sem política de retenção, a tabela audit_logs pode crescer indefinidamente, impactando performance do banco e custos de armazenamento.

Compliance e Segurança

O sistema de audit logging fornece evidências para:

  • GDPR / LGPD: Rastrear acesso a dados pessoais
  • SOX: Controles de acesso e alterações em dados financeiros
  • ISO 27001: Controle de acesso e monitoramento
  • PCI DSS: Rastreamento de alterações em dados sensíveis

Exemplos de Consulta

Logins de um Usuário (último dia)

SELECT
  id,
  userId,
  organizationId,
  action,
  ipAddress,
  userAgent,
  createdAt
FROM audit_logs
WHERE
  action = 'user.auth.success'
  AND userId = 'user_12345'
  AND createdAt >= NOW() - INTERVAL '1 day'
ORDER BY createdAt DESC;

Tentativas de Autenticação com API Key

SELECT
  a.id,
  a.organizationId,
  a.action,
  a.ipAddress,
  a.metadata->>'reason' as reason,
  a.createdAt
FROM audit_logs a
WHERE action LIKE 'apikey.auth.%'
ORDER BY createdAt DESC
LIMIT 50;

Negativas de Permissão por Organização

SELECT
  DATE(createdAt) as data,
  COUNT(*) as total_denied,
  jsonb_array_elements_text(metadata->'required') as permission_required
FROM audit_logs
WHERE
  action = 'permission.denied'
  AND organizationId = 'org_67890'
  AND createdAt >= NOW() - INTERVAL '7 days'
GROUP BY DATE(createdAt), permission_required
ORDER BY data DESC;

Logs de Erro da Última Hora

SELECT
  url,
  method,
  org.organizationName,
  usr.email as userEmail,
  metadata->>'message' as errorMessage,
  createdAt
FROM audit_logs a
LEFT JOIN organizations org ON org.id = a.organizationId
LEFT JOIN users usr ON usr.id = a.userId
WHERE action = 'error.occurred'
  AND createdAt >= NOW() - INTERVAL '1 hour'
ORDER BY createdAt DESC;

Estatísticas de Autenticação (últimos 30 dias)

SELECT
  DATE(createdAt) as data,
  COUNT(*) FILTER (WHERE status = 'SUCCESS') as sucessos,
  COUNT(*) FILTER (WHERE status = 'DENIED') as falhas
FROM audit_logs
WHERE action LIKE '%.auth.%'
  AND createdAt >= NOW() - INTERVAL '30 days'
GROUP BY DATE(createdAt)
ORDER BY data DESC;

Performance e Otimização

Fire-and-Forget

A função logAudit não aguarda a confirmação do banco para prosseguir. Isso é intencional — auditoria não deve atrasar a resposta ao usuário.

Índices Configurados

Os índices já configurados no schema Prisma facilitam consultas comuns:

  • Filtro por organizationId — útil para multi-tenancy
  • Filtro por userId — para buscar atividades de um usuário
  • Índice composto [entityType, entityId] — rastrear mudanças em entidades
  • Índice em action — agrupar por tipo de evento
  • Índice em createdAt — consultas por período

Otimizações Recomendadas

  • Partição por data: PostgreSQL particionado por mês melhora performance e manutenção
  • ETL periódico: Migrar logs antigos para tabela separada
  • Limitar consultas: Usar LIMIT em queries da dashboard de logs
