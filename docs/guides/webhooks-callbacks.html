Webhooks e Callbacks

Processamento de jobs via webhook e notificação por callback

Visão Geral

Quando um job é processado, o worker faz uma requisição HTTP para o webhookUrl configurado. Após a conclusão (sucesso ou falha final), o sistema opcionalmente faz um POST para o callbackUrl com o resultado.

Webhook Request

  • Method: configurável (GET, POST, PUT, PATCH, DELETE). Default POST.
  • Headers: Content-Type: application/json
  • Body (POST/PUT/PATCH): JSON do payload do job
  • Query (GET): payload serializado como query params (chave-valor simples)
  • Timeout: Respecta queue.timeout (default 300000ms = 5min). AbortController cancela se exceder.
// worker.ts - request options
{
  method: webhookMethod,
  headers: { 'Content-Type': 'application/json' },
  signal: controller.signal, // timeout
  body: JSON.stringify(payload) // ou query params
}

Resposta do Webhook

  • 2xx: Sucesso. Job status será COMPLETED. Response body parsed (se JSON) salvo como job.result.
  • Não-2xx: Erro. Throw exception → BullMQ retry com exponential backoff.
  • Timeout: AbortError → retry.

Callback (Notificação)

Somente se callbackUrl foi fornecido ao criar o job.

  • Método: Sempre POST
  • Headers: Content-Type: application/json
  • Body (Sucesso): 
    {
  "jobId": "...",
  "status": "COMPLETED",
  "result": {...},
  "duration": 1234,
  "attempts": 1
}
  • Body (Falha final - DEAD): 
    {
  "jobId": "...",
  "status": "DEAD",
  "error": "Timeout",
  "attempts": 3
}
  • Fire-and-forget: Callback failures são logados mas não afetam o job status.

Timeout e Retry

Cada queue configurable timeout (default 5 min). Se o worker não receber resposta dentro desse tempo, aborta e conta como falha. O BullMQ fará retry baseado em queue.maxRetries (default 3) com delay exponencial (queue.retryDelay default 5s).

Riscos

  • Webhook URL malicioso ou internal SSRF — não há validação de domínio.
  • Callback URL down — notificação perdida (sem retry).
  • Timeout muito baixo pode causar falhas desnecessárias.
  • Payloads grandes podem estourar memória ou exceder limites de request.
