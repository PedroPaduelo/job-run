Integração com Auth Service

Como o JobSync consome o serviço de identidade (Nommand Auth)

Componentes

  • Auth Service (Nommand): Provedor de identidade OAuth 2.0 + OpenID Connect
  • Frontend SDK: @nommand/auth-sdk para OAuth PKCE
  • Backend Adapters: Funções em src/lib/auth.ts para JWKS e APIs REST

Endpoints Usados

EndpointPropósito
/.well-known/jwks.jsonObter chaves públicas para verificar JWT (cache 5min)
/auth/profileBuscar dados do usuário (usado se JWT não contém email)
/organizations/Listar organizações do usuário
/organizations/{id}Buscar detalhes da organização (nome, slug, logo)
/organizations/{id}/members/meBuscar membership e roles do usuário

Base URL configurado em AUTH_SERVER_URL (backend) e VITE_AUTH_URL (frontend).

Fluxo de Sincronização de Usuário e Tenant

Backend request (Bearer JWT): 1. verifyJWT(token) → payload 2. syncUserFromAuth(token, payload) → upsert User 3. syncOrganizationFromAuth(token, orgId) → upsert Org 4. syncMembershipFromAuth(token, userId, orgId, roles) → upsert Member 5. buildAuthenticatedUser() → retorna user object com permissions

Tokens e Validação

O JWT é validado via JWKS. Vários issuers são aceitos (localhost e prod). Payload esperado:

{
  sub: userId,
  email?: string,
  firstName, lastName, displayName, avatarUrl,
  organizationId?,
  roles: string[],
  permissions?: string[],
  sessionId, scope
}

Riscos

  • Auth Service indisponível: login não funciona, mas JWKS cache permite validação por 5min.
  • Sincronização lenta: Cada request sincroniza user/org/member — pode ser custoso; pode ser cachead com TTL curto?
  • Mapeamento de roles: dependência de strings exatas (owner/admin/member/viewer).
