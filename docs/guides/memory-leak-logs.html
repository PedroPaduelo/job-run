Memory Leak in JobLogs

JobLog cresce indefinidamente e causa estouro de disco

📈 O Problema

Toda tentativa de job (incluindo retries) cria um JobLog. Não há TTL, archival ou limpeza automática.

💾
Growth Unlimited

10k jobs/dia × 3 tentativas = 30k logs/dia → ~1M logs/mês → GBs de dados

📊 Modelo JobLog (Prisma)

model JobLog {
  id        String   @id @default(cuid())
  jobId     String   @map("job_id")
  job       Job      @relation(fields: [jobId], references: [id], onDelete: Cascade)
  attempt   Int
  status    String   // 'COMPLETED', 'FAILED'
  message   String?
  duration  Int?     // ms
  metadata  Json?
  createdAt DateTime @default(now()) @map("created_at")
  @@index([jobId])
  @@index([createdAt])
}

Não há:

  • BSON/compression (PostgreSQL TOAST limited)
  • TTL automático
  • Partição por mês (↑ query perf em indexes)
  • Arquivação para cold storage

💥 Impacto

  • Disco enche → database readonly → sistema para
  • Queries job.logs cada vez mais lentas
  • Backups gigantic, restore lento
  • Custo de storage explode

✅ Recomendações

  1. Retention Policy — Deletar logs após 90 dias via cron job: 
    DELETE FROM job_logs WHERE created_at < NOW() - INTERVAL '90 days';
  2. Arquivar para S3/object storage — antes de deletar, exportar
  3. Partitioning por mês — melhor performance e DROP partition fácil
  4. Contador de logs por job — limite de 1000 logs por job, circular buffer no DB
  5. Compressão columnar — usar TimescaleDB ou particionamento com compressão
  6. Alert — monitorar tabela size: pg_size_pretty(pg_total_relation_size('job_logs'))

📈 Diagrama de Crescimento

Tempo ──────────────────────────────────────►
 │
 ├─ 1 dia: 100k logs (~500 MB)
 ├─ 30 dias: 3M logs (~15 GB)
 ├─ 90 dias: 9M logs (~45 GB)
 └─ 365 dias: 36.5M logs (~180 GB) ← Sem limpeza!

🔍 Checklist

  • JobLog tem TTL? ❌
  • JobLog é particionado? ❌
  • Há archival? ❌
  • Monitora tamanho da tabela? ❌
  • Testa crescimento em load test? ❌
