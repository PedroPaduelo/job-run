Redis Memory Optimization

Configuração de memória, eviction e TTL para BullMQ

💾 O Problema

BullMQ usa Redis para armazenar filas, jobs ativos, delayed, etc. Sem configuração adequada, Redis pode encher a memória e causar OOM ou swap.

📦
BullMQ Keys

Cada queue BullMQ cria múltiplas chaves: queue:waiting, queue:active, queue:delayed, queue:completed, queue:failed, etc.

📊 Uso de Memória por Tipo de Dado

  • queues: metadados da fila (configurações)
  • jobs ativos (active): small (jobId, data, opts)
  • jobs completed/failed:BullMQ remove após removeOnComplete / removeOnFail (default 0 = never)
  • delayed: jobs agendados permanecem até executar

🔧 Configuração atual (provável)

// src/lib/bullmq.ts
export const getQueue = (organizationId, name) => {
  return new Queue(`jobs-${organizationId}-${name}`, {
    connection: redis,
    defaultJobOptions: {
      removeOnComplete: false,  // ← default
      removeOnFail: false,
    },
  });
};

💥 Riscos

  • Se removeOnComplete = false, completed jobs ficam no Redis para sempre
  • Se thousands de jobs/dia, Redis cresce indefinidamente
  • Sem TTL, dados nunca expiram
  • maxmemory policy default = noeviction → erro ao escrever se cheio

✅ Recomendações

  1. Definir política de eviction no Redis: 
    maxmemory 2gb
maxmemory-policy allkeys-lru
  2. Configurar BullMQ TTL: 
    defaultJobOptions: {
  removeOnComplete: {
    count: 1000,    // manter últimos 1000 completed
    age: 24 * 3600  // ou remover após 24h
  },
  removeOnFail: {
    count: 100,
    age: 7 * 24 * 3600
  }
}
  3. Monitorar memória: 
    redis-cli info memory | grep used_memory_human
  4. Keyspace notifications — para saber quando chaves expiram
  5. Namespace por organizationId — já usa prefixo jobs-${orgId}-${queue}, bom para FLUSHDB por org se necessário
  6. Avoid large payloads in jobs — payload fica no Redis (ativo) e PostgreSQL (persistente). Se payload > 100KB, considerar armazenar no S3 e passar URL.

📈 Diagrama de Chaves

Redis Keyspace (prefixo: jobs:{orgId}:{queue})
│
├─ queue:waiting        ← ZSET (score = priority, timestamp)
├─ queue:active         ← SET (jobIds)
├─ queue:delayed        ← ZSET
├─ queue:completed      ← ZSET (jobIds) ← removeOnComplete limpa
├─ queue:failed         ← ZSET (jobIds) ← removeOnFail limpa
├─ job:{id}             ← HASH (job data)
│
└─ Meta chaves (lock, paused, etc)

🔍 Checklist

  • maxmemory configurado? ❌
  • maxmemory-policy definido? ❌
  • removeOnComplete/removeOnFail configurado? ❌
  • Monitoring de memória? ❌
  • TTL para chavesBullMQ? ❌
  • Payloads grandes evitados? ⚠️
