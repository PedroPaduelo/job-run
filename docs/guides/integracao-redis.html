Integração Redis — JobSync

Como o JobSync usa Redis + BullMQ, limites de persistência e riscos operacionais.

1. Uso de Redis

O JobSync usa Redis primariamente para filas BullMQ de processamento assíncrono. O Redis é o broker central de despacho de jobs e rastreamento de estado.

Características principais

  • Cliente: ioredis (configurado em src/lib/redis.ts)
  • Configuração crítica: maxRetriesPerRequest: null (retry infinito por comando — evita falhar em timeouts transitórios)
  • Conexão: singleton compartilhado em todo o backend
  • Event listeners: connect e error para logging de saúde
import Redis from 'ioredis';
const redisUrl = process.env.REDIS_URL || 'redis://localhost:6379';
export const redis = new Redis(redisUrl, { maxRetriesPerRequest: null });
redis.on('error', (err) => console.error('Redis connection error:', err));
redis.on('connect', () => console.log('Redis connected'));

2. Namespaces de Filas

Para isolamento multi-tenant, o JobSync usa o padrão:

jobs-{orgId}-{queueName}

Isso garante que filas de organizações diferentes não colidam. A função getQueue(orgId, queueName) gerencia o cache de filas por essa chave.

const key = `jobs-${orgId}-${queueName}`;
new Queue(key, { connection: redis, defaultJobOptions: { removeOnComplete: 100, removeOnFail: 1000 } });

3. Persistência Redis

Redis em-mória padrão (sem RDB/AOF configurado no código). Implicações:

  • Risco reinicialização: todos os dados em-mória são perdidos se Redis cair/cair.
  • BullMQ/JobSync: completed/failed gravados no PostgreSQL (histórico persistido). Active/delayed/waiting ficam apenas no Redis.
  • Caso Redis seja reiniciado, jobs em andamento/espera podem ser perdidos se não houver recovery via DB.
⚠️
Alerta de risco:

Sem backup ou AOF do Redis, reinicializações podem causar perda de jobs ativos/delayed. Mitigue com Redis gerenciado persistente ou replicação.

4. Pool de Conexão

Singleton em src/lib/redis.ts — todas as filas e workers compartilham a mesma instância Redis, reduzindo overhead de sockets.

maxRetriesPerRequest: null significa que o cliente tentará retry infinito em comandos Redis caso haja falhas de rede/timeout, ao invés de falhar rapidamente.

Nota: keep-alive settings devem ser ajustados conforme o deployment (Kubernetes, bare-metal, provedor gerenciado).

5. Alta Disponibilidade

Não há Redis Sentinel ou Cluster configurado. Single Redis instance = SPOF. Se Redis cair, o processamento de jobs para.

Recomendação para produção: usar Redis Cluster ou Sentinel com failover automático, ou um provedor gerenciado com alta disponibilidade.

⛔
Ponto único de falha:

Sem HA no Redis, qualquer indisponibilidade interrompe o processamento de filas.

6. Monitoramento

Comandos Redis CLI para saúde e troubleshooting:

  • redis-cli info — métricas gerais
  • redis-cli monitor — logs de comandos em tempo real (cuidado em produção)
  • redis-cli info stats — estatísticas de throughput/hits

No JobSync, o BullMQ Board exibe comprimentos de fila e jobs em estados. Adicione métricas via endpoints do backend ou Prometheus se necessário.

7. Métricas de Uso

BullMQ expõe métodos para contadores de status:

  • PENDING
  • ACTIVE
  • COMPLETED
  • FAILED
  • DELAYED
  • WAITING

O JobSync consolida essas métricas no dashboard e também persiste resultados no PostgreSQL via tabelas de logs e job history.

Coleta via service (src/services/queue.service.ts) chama bullQueue.getWaitingCount(), getActiveCount(), getDelayedCount() mais contagem de PENDING do banco.

8. Avaliação de Risco

Redis crash:

  • Jobs em memória (active/delayed/waiting) são perdidos se Redis não persistir.
  • Jobs completados/falhos permanecem no PostgreSQL — podem ser usados para reconciliação.
  • Processamento de jobs atuais é interrompido; workers podem relançar jobs se configurados.

Recomendações

  • Habilitar persistência RDB/AOF no Redis de produção.
  • Usar Redis Cluster/Sentinel para HA.
  • Implementar reconcilição de jobs a partir do DB caso Redis seja recriado.
  • Monitorar healthcheck Redis (latência, conexão)

Diagrama de Arquitetura Redis

+-----------------------+ +-------------------------+ | JobSync API |----->| | | (Fastify + Worker) | | Redis (ioredis) | +-----------------------+ | BullMQ Queues | | | jobs-{orgId}-{queue} | | +-------------------------+ | | v v +----------+ +-------------+ | PostgreSQL|◄─────────────┤ Workers | | (history, | results │ (processJob)| | logs) | +-------------+ +----------+

Redis atua como broker de enfileiramento; PostgreSQL armazena histórico e resultados finais. Worker consome do Redis e salva resultados no PostgreSQL.
