1. Processo Worker (worker.ts)

O processo worker e o coracao do sistema de processamento do JobSync. Ele e responsavel por consumir jobs das filas Redis e executa-los de forma assincrona.

Ciclo de Inicializacao

Quando o processo worker e iniciado, ele executa a seguinte sequencia:

  1. Conexao Redis: Estabelece conexao com o servidor Redis configurado via variaveis de ambiente.
  2. Busca de filas ativas: Consulta o banco de dados (PostgreSQL via Prisma) para obter todas as filas com status active = true.
  3. Criacao de Workers BullMQ: Para cada fila ativa encontrada, cria uma instancia de Worker do BullMQ.
  4. Inicio do polling: Configura um setInterval de 30 segundos para descobrir novas filas (veja Descoberta Dinamica).
// worker.ts - Inicializacao simplificada
import { Worker } from 'bullmq';
import { prisma } from './lib/prisma';
import { redisConnection } from './lib/redis';

const CONCURRENCY = parseInt(process.env.WORKER_CONCURRENCY || '10');
const workers: Map<string, Worker> = new Map();

async function startWorkers() {
  // 1. Buscar todas as filas ativas
  const queues = await prisma.queue.findMany({
    where: { active: true }
  });

  // 2. Criar um Worker para cada fila
  for (const queue of queues) {
    const queueName = `jobs-${queue.organizationId}-${queue.name}`;
    const concurrency = queue.concurrency || CONCURRENCY;

    const worker = new Worker(queueName, processJob, {
      connection: redisConnection,
      concurrency,
      limiter: queue.rateLimit
        ? { max: queue.rateLimit, duration: 60_000 }
        : undefined,
    });

    workers.set(queueName, worker);
    console.log(`Worker iniciado: ${queueName} (concurrency: ${concurrency})`);
  }
}

startWorkers();

Configuracao de Concorrencia

Cada worker possui sua propria configuracao de concorrencia, que determina quantos jobs podem ser processados simultaneamente:

Parametro Origem Valor Padrao Descricao
queue.concurrency Banco de dados (por fila) null Concorrencia especifica da fila; sobrescreve o valor global
WORKER_CONCURRENCY Variavel de ambiente 10 Concorrencia global padrao para todas as filas
ℹ
Prioridade de Concorrencia

Se a fila possui concurrency definido no banco, esse valor e usado. Caso contrario, o valor da variavel de ambiente WORKER_CONCURRENCY (padrao 10) e aplicado.

Rate Limiter

O rate limiter controla a taxa maxima de processamento de jobs por fila:

  • max: Valor de queue.rateLimit definido no banco (padrao: null = sem limite)
  • duration: Janela de tempo fixa de 60.000ms (1 minuto)
  • Se rateLimit for null, nenhum limitador e aplicado

Naming Convention das Filas Redis

Cada fila no Redis segue o padrao de nomenclatura:

jobs-{organizationId}-{queueName}

// Exemplos:
jobs-org_abc123-email-notifications
jobs-org_def456-webhook-delivery
jobs-org_abc123-data-processing

Esse namespacing garante isolamento total entre organizacoes no Redis, prevenindo colisoes e facilitando monitoramento por tenant.

2. Funcao de Processamento: processJob(job)

A funcao processJob e o handler central executado pelo BullMQ para cada job consumido da fila. Ela implementa o ciclo completo de vida de um job.

Fluxo de Execucao

processJob(job) | v +---------------------------+ | 1. Atualiza status no DB | | status = PROCESSING | +-------------+-------------+ | v +---------------------------+ | 2. HTTP POST webhookUrl | | - body: job.data | | - timeout configuravel | | - headers customizados | +-------------+-------------+ | +--------+--------+ | | Sucesso Falha (2xx) (erro/timeout) | | v v +-----------+ +-------------------+ | 3a. DB: | | 3b. throw Error | | COMPLETED | | | +-----------+ | BullMQ captura e | | | agenda retry com | v | backoff exp. | +-----------+ +--------+----------+ | 4. Chama | | | callback | Tentativas esgotadas? | (se def.) | | | +-----------+ Nao Sim | | v v [Retry] +---------+ | DB: | | DEAD | +---------+

Implementacao

// processJob - Handler do BullMQ Worker
async function processJob(job: Job) {
  const { jobId, webhookUrl, callbackUrl, headers, timeout } = job.data;

  // 1. Marcar como PROCESSING
  await prisma.job.update({
    where: { id: jobId },
    data: { status: 'PROCESSING', startedAt: new Date() }
  });

  try {
    // 2. Requisicao HTTP para o webhook
    const response = await fetch(webhookUrl, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        ...headers,
      },
      body: JSON.stringify(job.data.payload),
      signal: AbortSignal.timeout(timeout || 30_000),
    });

    if (!response.ok) {
      throw new Error(`Webhook retornou status ${response.status}`);
    }

    const result = await response.json();

    // 3a. Sucesso - Marcar como COMPLETED
    await prisma.job.update({
      where: { id: jobId },
      data: {
        status: 'COMPLETED',
        completedAt: new Date(),
        result: result,
      }
    });

    // 4. Chamar callback (se configurado)
    if (callbackUrl) {
      await fetch(callbackUrl, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ jobId, status: 'COMPLETED', result }),
      });
    }

  } catch (error) {
    // 3b. Falha - Atualizar tentativas e lancar erro
    await prisma.job.update({
      where: { id: jobId },
      data: {
        lastError: error.message,
        attempts: { increment: 1 },
      }
    });

    // Se esgotou tentativas, marcar como DEAD
    if (job.attemptsMade >= job.opts.attempts - 1) {
      await prisma.job.update({
        where: { id: jobId },
        data: { status: 'DEAD', failedAt: new Date() }
      });
      return; // Nao relanca - job considerado morto
    }

    // Relanca o erro para BullMQ agendar retry
    throw error;
  }
}

Estrategia de Retry

Quando processJob lanca um erro, o BullMQ automaticamente agenda uma nova tentativa usando backoff exponencial:

Tentativa Delay (backoff exponencial) Tempo Aproximado
1a1s * 2^0 = 1s+1 segundo
2a1s * 2^1 = 2s+2 segundos
3a1s * 2^2 = 4s+4 segundos
4a1s * 2^3 = 8s+8 segundos
5a1s * 2^4 = 16s+16 segundos
.........
maxAttempts--Status: DEAD
⚠
Jobs Mortos (DEAD)

Quando um job atinge o numero maximo de tentativas (maxAttempts), ele recebe o status DEAD. Jobs mortos nao sao mais processados automaticamente, mas podem ser retriados manualmente via API ou dashboard.

3. Descoberta Dinamica de Filas

O sistema de descoberta dinamica permite que novos workers sejam criados sem necessidade de reiniciar o processo. A cada 30 segundos, o worker verifica se ha novas filas ativas no banco de dados.

Mecanismo de Polling

// Polling para novas filas a cada 30 segundos
const POLL_INTERVAL = 30_000; // 30 segundos

setInterval(async () => {
  const activeQueues = await prisma.queue.findMany({
    where: { active: true }
  });

  for (const queue of activeQueues) {
    const queueName = `jobs-${queue.organizationId}-${queue.name}`;

    // Se ja existe um worker para essa fila, ignora
    if (workers.has(queueName)) continue;

    // Nova fila detectada - criar worker
    const concurrency = queue.concurrency || CONCURRENCY;
    const worker = new Worker(queueName, processJob, {
      connection: redisConnection,
      concurrency,
      limiter: queue.rateLimit
        ? { max: queue.rateLimit, duration: 60_000 }
        : undefined,
    });

    workers.set(queueName, worker);
    console.log(`Novo worker criado: ${queueName} (concurrency: ${concurrency})`);
  }
}, POLL_INTERVAL);

Comportamento

Cenario Acao
Nova fila ativa criada no banco Worker e criado automaticamente no proximo ciclo de polling (ate 30s)
Fila existente ja possui worker Ignorada (sem duplicacao)
Fila desativada (active = false) Worker nao e removido automaticamente. O worker continua ativo ate o processo ser reiniciado.
Fila deletada do banco Worker existente continua ativo (mas sem novos jobs). Removido no restart.
ℹ
Design Decision: Sem Remocao Automatica

Workers nao sao removidos automaticamente ao desativar uma fila. Isso evita condicoes de corrida onde jobs em processamento poderiam ser interrompidos. Para remover workers, reinicie o processo.

4. Dimensoes de Escalabilidade

O JobSync foi projetado para escalar em multiplas dimensoes, aproveitando a arquitetura distribuida do BullMQ e Redis.

Escalabilidade Horizontal

Multiplos Processos Worker

Execute multiplos processos worker.ts em maquinas diferentes, todos conectando ao mesmo Redis e mesmo PostgreSQL. O BullMQ distribui os jobs automaticamente entre os workers disponiveis.

  • Cada processo worker busca filas ativas e cria seus proprios workers BullMQ
  • O BullMQ usa locks atomicos no Redis para garantir que cada job e processado por exatamente um worker
  • Adicionar mais maquinas = mais throughput, sem alteracao de codigo
  • Todos os processos compartilham o mesmo estado (Redis + PostgreSQL)
# Executar workers em diferentes maquinas (ou containers)

# Maquina 1
REDIS_URL=redis://redis-server:6379 \
DATABASE_URL=postgresql://user:pass@db-server/jobsync \
WORKER_CONCURRENCY=10 \
node dist/worker.js

# Maquina 2
REDIS_URL=redis://redis-server:6379 \
DATABASE_URL=postgresql://user:pass@db-server/jobsync \
WORKER_CONCURRENCY=20 \
node dist/worker.js

# Maquina 3 (alta capacidade)
REDIS_URL=redis://redis-server:6379 \
DATABASE_URL=postgresql://user:pass@db-server/jobsync \
WORKER_CONCURRENCY=50 \
node dist/worker.js

Escalabilidade Vertical

Aumento de Recursos por Instancia

Aumente a capacidade de processamento de cada instancia individual:

  • Concorrencia da fila: Aumente o valor de queue.concurrency no banco para filas especificas
  • Concorrencia global: Aumente WORKER_CONCURRENCY (variavel de ambiente) para o processo inteiro
  • Recursos Redis: Aumente memoria, CPU e conexoes do servidor Redis
  • CPU/Memoria do worker: Aloque mais recursos para o container/maquina do worker

Isolamento por Fila

Controle Granular

Cada fila opera de forma independente com suas proprias configuracoes:

  • Concorrencia propria: Uma fila pode processar 5 jobs por vez enquanto outra processa 50
  • Rate limit proprio: Limite de taxa independente por fila (ex: 100 jobs/minuto para emails, 10 jobs/minuto para webhooks pesados)
  • Sem interferencia: Uma fila lenta ou congestionada nao afeta o processamento de outras filas

5. Monitoramento

O JobSync oferece multiplas formas de monitorar o estado dos workers e do processamento de jobs.

Eventos BullMQ

Cada worker emite eventos que podem ser capturados para monitoramento:

// Listeners de eventos do Worker
worker.on('completed', (job, result) => {
  console.log(`Job ${job.id} completado na fila ${job.queueName}`);
});

worker.on('failed', (job, error) => {
  console.error(`Job ${job.id} falhou na fila ${job.queueName}: ${error.message}`);
});

worker.on('error', (error) => {
  console.error(`Erro no worker: ${error.message}`);
});

Logs de Progresso

O sistema registra via console.log os seguintes eventos:

  • Inicializacao de cada worker (nome da fila, concorrencia)
  • Descoberta de novas filas (polling)
  • Inicio do processamento de cada job
  • Conclusao (sucesso ou falha) de cada job
  • Tentativas de retry e transicoes de status

Endpoint de Estatisticas

GET /api/queues/:id/stats Estatisticas de uma fila especifica

Retorna metricas em tempo real de uma fila:

{
  "queueId": "queue_abc123",
  "name": "email-notifications",
  "stats": {
    "waiting": 42,        // Jobs aguardando processamento
    "active": 8,          // Jobs sendo processados agora
    "completed": 15230,   // Total de jobs completados
    "failed": 127,        // Total de jobs que falharam
    "delayed": 3,         // Jobs agendados para retry
    "paused": 0           // Jobs em fila pausada
  }
}

6. Limites do Sistema

Entenda os limites praticos do sistema e como o backpressure e gerenciado.

Recurso Limite Fator Limitante
Jobs por fila Ilimitado Memoria disponivel no Redis
Workers por processo Variavel Numero de cores de CPU e memoria disponivel
Filas por organizacao Ilimitado Limite pratico de gerenciamento
Concorrencia por fila Configuravel CPU, memoria e capacidade do endpoint de webhook
Rate limit por fila Configuravel Definido em jobs/minuto no banco de dados

Estrategias de Backpressure

Para evitar sobrecarga do sistema, utilize as seguintes estrategias:

Controles Disponiveis

  • Concorrencia da fila: Limite quantos jobs sao processados simultaneamente por fila. Filas com webhooks pesados devem ter concorrencia baixa (ex: 2-5).
  • Rate limit: Defina um teto de jobs/minuto para proteger APIs externas de sobrecarga (ex: rateLimit: 60 = max 60 jobs por minuto).
  • Pausa de fila: Pause uma fila temporariamente via API quando detectar problemas no endpoint de destino.
  • Monitoramento: Use o endpoint /api/queues/:id/stats para detectar acumulo de jobs waiting ou delayed.

7. Alta Disponibilidade

O JobSync e projetado para ser resiliente a falhas, garantindo que nenhum job seja perdido mesmo em caso de crash.

Resiliencia a Falhas

Crash do Worker

  • Se um processo worker crashar, os jobs nao sao perdidos
  • O Redis persiste todos os jobs na fila
  • Outro processo worker (na mesma ou em outra maquina) assume o processamento
  • Jobs que estavam em PROCESSING no momento do crash sao automaticamente movidos de volta para a fila pelo BullMQ (stalled job recovery)

Supervisor de Processos

✔
Recomendacao: Use PM2

Para producao, utilize o PM2 como supervisor de processos. Ele reinicia automaticamente workers que crasham e permite gerenciar multiplas instancias.

# ecosystem.config.js (PM2)
module.exports = {
  apps: [{
    name: 'jobsync-worker',
    script: 'dist/worker.js',
    instances: 'max',           // 1 processo por core
    exec_mode: 'cluster',
    autorestart: true,
    max_restarts: 10,
    restart_delay: 5000,        // 5s entre restarts
    env: {
      NODE_ENV: 'production',
      WORKER_CONCURRENCY: '10',
      REDIS_URL: 'redis://redis-server:6379',
    },
  }]
};

# Comandos PM2
pm2 start ecosystem.config.js   # Iniciar workers
pm2 status                      # Ver status dos processos
pm2 logs jobsync-worker         # Ver logs em tempo real
pm2 reload jobsync-worker       # Zero-downtime reload
pm2 scale jobsync-worker +2     # Adicionar 2 instancias

Checklist de Producao

Redis Sentinel/Cluster Recomendado Failover automatico do Redis

8. Configuracao

Todas as opcoes de configuracao do worker e seus valores padrao.

Variaveis de Ambiente

Variavel Padrao Descricao
WORKER_CONCURRENCY 10 Numero maximo de jobs processados simultaneamente por worker (valor global)
REDIS_URL redis://localhost:6379 URL de conexao com o Redis
DATABASE_URL -- URL de conexao com o PostgreSQL (Prisma)

Configuracao por Fila (Banco de Dados)

Campo Tipo Padrao Descricao
concurrency Int? null Concorrencia especifica da fila. Se definido, sobrescreve WORKER_CONCURRENCY
rateLimit Int? null Maximo de jobs processados por minuto. null = sem limite
active Boolean true Se a fila esta ativa e deve ter um worker criado
maxAttempts Int 3 Numero maximo de tentativas antes de marcar como DEAD

Hierarquia de Configuracao

Prioridade de Concorrencia: +------------------------------------------+ | queue.concurrency (banco de dados) | <-- Maior prioridade | Valor especifico por fila | +------------------------------------------+ | | fallback (se null) v +------------------------------------------+ | WORKER_CONCURRENCY (env var) | <-- Prioridade media | Valor global para o processo | +------------------------------------------+ | | fallback (se nao definido) v +------------------------------------------+ | Padrao: 10 | <-- Menor prioridade | Valor hardcoded no codigo | +------------------------------------------+
ℹ
Dica de Configuracao

Para workloads com webhooks lentos (ex: APIs externas com alta latencia), use concorrencia alta (20-50) pois os workers ficam esperando I/O. Para workloads CPU-bound, mantenha a concorrencia baixa (2-5) para evitar saturacao da CPU.
