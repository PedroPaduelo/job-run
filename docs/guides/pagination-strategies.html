Pagination Strategies

Offset vs cursor, deep pagination, performance em listas grandes

📄 Situação Atual

JobService.list usa skip/limit (offset-based pagination):

const [jobs, total] = await Promise.all([
  prisma.job.findMany({
    where,
    orderBy: { createdAt: 'desc' },
    take: filters.limit,
    skip: filters.offset,
  }),
  prisma.job.count({ where }),
]);
⚠️
Offset Pagination Problem

Em tabelas grandes, OFFSET 100000 requer escanear 100k rows → performance degrade指数.

📊 Comparação de Estratégias

EstratégiaVantagensDesvantagens
Offset/Limit Simples, random access (vai para página 50) Slow em offset alto, instável com inserts/deletes
Cursor (keyset) Constante performance, estável Só permite next/prev, não random page
Seek Method Otimizado com WHERE id > lastId Requer ordenação por indexed column

💥 Problemas Atuais

  • orderBy: { createdAt: 'desc' } — createdAt tem índice, mas offset grande força seq scan
  • count() é expensive em tabelas grandes (sem cache)
  • Usuário pode pedir offset 100k → query lenta
  • Concorrência: jobs sendo inseridos mudam offset results (missing/duplicados)

✅ Recomendações

  1. Implementar cursor-based pagination: 
    // Ao invés de offset, passar lastCreatedAt (timestamp do último job da página anterior)
where: {
  organizationId,
  createdAt: { lt: lastCreatedAt }  // jobs anteriores ao último
}
orderBy: { createdAt: 'desc' }
take: limit
  2. Frontend mudança:
    • Remover offset do estado
    • Guardar nextCursor = último job.createdAt da página
    • Botão "Carregar mais" em vez de page numbers
  3. Cache de count — salvar total em Redis (TTL 60s) em vez de COUNT every request
  4. Infinite scroll — UX melhore, carrega dinamicamente
  5. Busca por ID específico — manter GET /jobs/:id (by primary key, instant)
  6. Export full list — se usuário precisa de todos jobs, oferecer CSV async job

📈 Diagrama

Offset Pagination (BAD):
  page 1: LIMIT 20 OFFSET 0      → fast
  page 2: LIMIT 20 OFFSET 20     → OK
  page 10: LIMIT 20 OFFSET 180   → scans 180 rows
  page 100: LIMIT 20 OFFSET 1980 → scans 1980 rows

Cursor Pagination (GOOD):
  page 1: WHERE createdAt < NOW() ORDER BY createdAt DESC LIMIT 20
  page 2: WHERE createdAt < last_cursor1 ORDER BY ... LIMIT 20
  → Sempre index seek, constante O(log n)

🔍 Checklist

  • Offset-based em uso? ✅
  • Cursor-based implementado? ❌
  • Cache de count? ❌
  • Teste de carga com offset alto? ❌
  • Frontend suporta infinite scroll? ❌
  • Export CSV como job? ❌
