Integrações Externas do JobSync

Referência das integrações externas usadas no JobSync e os riscos operacionais associados.

Visão Geral

O JobSync depende de serviços externos para autenticação, persistência, filas, observabilidade e callbacks. Abaixo documentamos cada integração, endpoints, variáveis de ambiente e riscos principais.

1. Auth Service (Nommand)

O JobSync delega a autenticação de usuários ao Auth Service (Nommand). O frontend usa a SDK OAuth PKCE para login e o backend valida JWT via JWKS.

Configuração e endpoints

  • URL base: AUTH_SERVER_URL (default http://localhost:3000)
  • JWKS: /.well-known/jwks.json
  • Endpoints consumidos:
    • GET /auth/profile
    • GET /organizations/
    • GET /organizations/{id}
    • GET /organizations/{id}/members/me

SDK

Frontend: @nommand/auth-sdk (OAuth PKCE).

Risco operacional

Auth Service downtime: impediria novos logins para usuários. A validação de tokens existentes continua funcionando enquanto o JWKS em cache estiver válido (cerca de 5 minutos).

2. PostgreSQL (Prisma)

O JobSync usa PostgreSQL como banco de dados primário para persistir dados de jobs, filas, organizações, usuários, API Keys e logs.

Configuração

  • Connection string: DATABASE_URL
  • ORM: Prisma Client
  • Schema: 8 modelos (User, Organization, OrganizationMember, Queue, Job, JobLog, ApiKey, AuditLog) com índices de performance e constraints de integridade

Risco operacional

  • Connection pool exhaustion: muitas conexões simultâneas podem saturar o banco e causar erros 500.
  • Migration drift: divergência entre schema Prisma e base de dados em produção pode quebrar queries e gerar comportamentos inesperados.

3. Redis (ioredis + BullMQ)

Redis é usado como backend para BullMQ (filas de jobs), cache de estado e coordenação entre workers.

Configuração

  • URL de conexão: REDIS_URL (default redis://localhost:6379)
  • Cliente: ioredis
  • Uso principal: filas BullMQ para jobs e workers
  • Retry: maxRetriesPerRequest: null (retry infinito no cliente Redis)

Risco operacional

Redis indisponível: o processamento de jobs é interrompido e a fila não avança. A API CRUD permanece disponível (desde que o banco PostgreSQL esteja saudável), mas nenhuma execuçãode jobs ocorrerá.

4. Sentry

O JobSync envia erros para Sentry para monitoramento de produção e rastreamento de exceções.

Configuração

  • DSN: SENTRY_DSN
  • Backend: @sentry/node
  • Frontend: @sentry/react

Risco operacional

Se o Sentry estiver indisponível, o sistema ignora erros de envio e continua funcionando. As falhas de observabilidade podem atrasar a detecção de incidentes.

5. Webhooks Externos (definidos pelo usuário)

Workers fazem chamadas para URLs de webhook definidos pelo usuário para executar tarefas externas (por exemplo, notificar outro sistema, criar registros externos, etc.).

Suporte de métodos

  • GET, POST, PUT, PATCH, DELETE

Risco operacional

  • URLs maliciosos: podem causar exposição de dados ou chamadas a endpoints não confiáveis.
  • SSRF: URLs internas podem permitir acesso a serviços internos se não houver validação adequada.
  • Endpoints lentos: chamadas longas podem atrasar o processamento de jobs ou esgotar timeouts.

6. Callbacks Externos (definidos pelo usuário)

Após conclusão ou falha de um job, o sistema realiza POST em URLs de callback configurados pelo usuário para notificar sistemas externos.

Comportamento

  • Chamadas feitas via POST
  • Execução fire-and-forget: não há retry automático em falha de callback

Risco operacional

Se o endpoint de callback estiver indisponível, a notificação será perdida. Recomendamos implementar um mecanismo idempotente e retries no consumidor do callback, ou usar webhooks adicionais para confiabilidade.
