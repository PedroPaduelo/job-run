Página de Configurações

Documentação completa da página Settings do frontend JobSync - Gerenciamento de API Keys e informações do sistema

Visão Geral

A página Settings (Configurações) do JobSync é o centro de controle para gerenciamento de API Keys e visualização de informações do sistema. Esta página permite que usuários criem, gerenciem e monitorem chaves de API para autenticação máquina-a-máquina (M2M).

Localização no Código

2. Criação de API Keys

O processo de criação de API Keys é o momento mais crítico do gerenciamento, pois a chave completa só é exibida uma única vez.

2.1 Fluxo de Criação

Fluxo de Criação de API Key =========================== 1. Usuário clica em "Create API Key" │ ▼ 2. Formulário é exibido │ ├── Name (obrigatório) ├── Description (opcional) └── Permissions (badges clicáveis) │ ▼ 3. Usuário preenche e clica "Create Key" │ ▼ 4. Chamada: createApiKey.mutateAsync() │ ├── Payload: { name, description, permissions } │ ▼ 5. Backend retorna: { key: "jsk_..." } │ ▼ 6. ✓ Alerta VERDE exibido com a chave ├── Input readonly com a chave completa ├── Botão [Copy] → copia para clipboard └── Botão [Done] → fecha o alerta │ ▼ 7. Formulário é resetado └── Estado: createdKey = null ⚠️ IMPORTANTE: A chave NUNCA mais será exibida! Se perdida, deve-se revogar e criar outra.

Comportamento de Toggle

// Função de toggle em Settings.tsx
const togglePermission = (permission: string) => {
  setNewKey((prev) => ({
    ...prev,
    permissions: prev.permissions.includes(permission)
      ? prev.permissions.filter((p) => p !== permission)
      : [...prev.permissions, permission],
  }));
};

2.4 Permissões Padrão

Quando o formulário é inicializado, as seguintes permissões já vem selecionadas por padrão:

⚠️
Validação

O campo name é obrigatório. Se o usuário tentar criar sem informar, será exibido um toast de erro: "API key name is required"

3. Mecanismo de Cópia da Chave

Um dos aspectos mais importantes da criação de API Keys é o mecanismo de cópia segura.

3.1 Exibição Única

🚨
Regra Crítica

A chave completa (result.key) é recebida do backend e armazenada no estado createdKey. Esta é a ÚNICA vez que a chave é exposta ao usuário. Após fechar o alerta, não há como recuperá-la.

3.2 Componente de Cópia

// State para controlar feedback visual
const [copiedKey, setCopiedKey] = useState(false);

// Função de cópia
const handleCopyCreatedKey = () => {
  if (createdKey) {
    navigator.clipboard.writeText(createdKey);
    setCopiedKey(true);

    // Reset após 2 segundos
    setTimeout(() => setCopiedKey(false), 2000);

    toast.success("API key copied to clipboard");
  }
};

3.3 Feedback Visual

O botão de cópia altera seu estado para fornecer feedback imediato ao usuário:

Estado Texto do Botão Estilo
Inicial Copy variant="outline"
Após clique Copied! variant="default" (ou success)

3.4 Fechamento do Alerta

const handleCloseCreatedKey = () => {
  setCreatedKey(null);      // Limpa a chave da memória
  setShowCreateKey(false);   // Fecha o formulário
};
ℹ️
Reset do Formulário

Ao fechar o alerta de chave criada, o formulário é automaticamente resetado com as permissões padrão: ["jobs:create", "jobs:read", "queues:read"]

5. Ações sobre API Keys

Duas ações podem ser realizadas sobre cada API Key: Revogar e Deletar.

5.1 Revogar (Soft Delete)

⚠️
Revoke - Soft Delete

A revogação define isActive = false. A chave permanece no sistema mas não pode mais ser usada para autenticação. Esta ação é reversível.

const handleRevokeApiKey = async (id: string, name: string) => {
  if (confirm(\`Are you sure you want to revoke the API key "\${name}"?\nIt will no longer work for authentication.\`)) {
    try {
      await revokeApiKey.mutateAsync(id);
      toast.success("API key revoked successfully");
    } catch (error) {
      toast.error("Failed to revoke API key");
    }
  }
};

5.2 Deletar (Hard Delete)

🚨
Delete - Hard Delete

A deleção remove a chave permanentemente do banco de dados. Esta ação NÃO pode ser desfeita.

const handleDeleteApiKey = async (id: string, name: string) => {
  if (confirm(\`Are you sure you want to permanently delete the API key "\${name}"?\nThis action cannot be undone.\`)) {
    try {
      await deleteApiKey.mutateAsync(id);
      toast.success("API key deleted successfully");
    } catch (error) {
      toast.error("Failed to delete API key");
    }
  }
};

5.3 Condição de Exibição

O botão de Revoke só é exibido para chaves ativas:

{key.isActive && (
  <Button variant="ghost" size="icon" onClick={() => handleRevokeApiKey(...)}>
    <AlertCircle className="h-4 w-4" />
  </Button>
)}
<Button variant="ghost" size="icon" onClick={() => handleDeleteApiKey(...)}>
  <Trash2 className="h-4 w-4" />
</Button>

7. Controle de Permissões no Frontend

Por design, a criação de API Keys deve ser restrita a usuários autorizados.

ℹ️
Implementação Futura

Segundo a especificação, o frontend deve verificar se o usuário possui a permissão apikeys:create antes de exibir o botão de criação. Esta verificação ainda não está implementada no código atual.

Lógica Esperada

// Pseudocódigo da verificação esperada
const canCreateApiKey = user.permissions.includes('apikeys:create');

return (

13. Integração com Backend

A página de Settings se integra com o backend através de mutações e queries do React Query.

13.1 Criação de API Key

const result = await createApiKey.mutateAsync({
  name: newKey.name.trim(),
  description: newKey.description.trim() || undefined,
  permissions: newKey.permissions,
});

// Resultado esperado:
// { key: "jsk_abc123..." }

if (result.key) {
  setCreatedKey(result.key);  // Exibe uma única vez
}

13.2 Revogação de API Key

await revokeApiKey.mutateAsync(id);
// POST /api/apikeys/:id/revoke
// Resultado: { isActive: false }

13.3 Deleção de API Key

await deleteApiKey.mutateAsync(id);
// DELETE /api/apikeys/:id
// Resultado: { message: "..." }
