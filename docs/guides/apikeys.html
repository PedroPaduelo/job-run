API Keys e Autenticação M2M

Sistema completo de autenticação máquina-a-máquina para integrações server-to-server

Visão Geral

O JobSync suporta dois modos de autenticação:

  • OAuth 2.0 (Usuários): Para acesso via frontend ou apps interativos
  • API Keys (M2M): Para integrações automatizadas entre serviços

Esta documentação cobre o sistema de API Keys, ideal para:

  • CI/CD pipelines que disparam jobs
  • Webhooks de outros sistemas
  • Scripts de automação
  • Integrações backend-backend

Pré-requisitos

  • Um organizationId válido ao qual a API Key será vinculada
  • Escopo de permissões adequado aos casos de uso
  • Configuração de domínio/IP se aplicável

1. Formato da Chave

As API Keys seguem um formato específico para facilitar identificação e validação:

Formato: jsk_ + 64 hex chars Exemplo: jsk_a1b2c3d4e5f67890123456789abcdef0123456789abcdef0123456789abcdef ├─ Prefixo: "jsk_" (4 caracteres) ├─ Entropia: 32 bytes aleatórios └─ Codificação: hex (64 caracteres) └─ Total: 68 caracteres Geração (Node.js): crypto.randomBytes(32).toString('hex')

Exemplo de Geração

const crypto = require('crypto');

function generateApiKey() {
  const randomBytes = crypto.randomBytes(32);
  const hex = randomBytes.toString('hex');
  return `jsk_${hex}`;
}

// Resultado: jsk_3f7a8b2c1d9e4f5a6b7c8d9e0f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9
ℹ️
Prefixos Suportados

O sistema aceita apenas chaves começando com jsk_. Isso permite validação rápida antes da consulta ao banco.

2. Ciclo de Vida da Chave

Toda API Key passa por um ciclo de vida gerenciado:

Ciclo de Vida (Lifecycle) ========================= [CREATE] │ │ 1. POST /api/apikeys │ Retorna full_key UNA VEZ ▼ [ACTIVE] ───────────────────┐ │ │ │ 2. Uso_normal │ │ ✓ ✓ ✓ │ │ │ │ 3. POST /:id/revoke │ ▼ │ [REVOKED] ◄──────────────────┘ │ │ isActive: false │ │ 4. DELETE /:id ▼ [DELETED] │ │ Removida permanentemente │ Não é mais válida Estados: • CREATE – Chave recém-criada, key Disponível • ACTIVE – Em uso, pode ser revogada • REVOKED – Desativada (soft delete), isActive=false • DELETED – Removida permanentemente (hard delete)

Importante

  • Chave completa é retornada APENAS na criação (POST). Após isso, só é possível ver a chave mascarada.
  • Revoke é reversível — você pode reativar uma chave (PATCH isActive=true)
  • Delete é permanente — não há como recuperar após DELETE

3.1 Permissões (Permissions)

Array de strings que definem o que a chave pode fazer. Valores possíveis:

PermissãoDescriçãoEscopo
jobs:createCriar novos jobsJob
jobs:readListar e ver detalhes de jobsJob
jobs:updateAtualizar jobs (metadata)Job
jobs:deleteRemover jobsJob
jobs:retryRetry de jobs falhosJob
jobs:cancelCancelar jobs pendentesJob
queues:readVer filas e estatísticasQueue
queues:createCriar novas filasQueue
queues:updateModificar configuraçõesQueue
queues:deleteRemover filasQueue
apikeys:createCriar novas API keysApiKey
apikeys:readListar API keysApiKey
apikeys:deleteDeletar API keysApiKey
📝
Permissões Padrão

Ao criar uma API Key sem especificar permissões, as seguintes são atribuídas por padrão: ['jobs:create', 'jobs:read', 'queues:read']

3.2 Restrição de Domínio ( allowedDomains )

{
  allowedDomains: ['https://meuapp.com', 'https://api.meuapp.com']
}

Lista de origens (domínios completos) autorizados a usar esta chave. Use [] ou null para permitir qualquer origem. A validação compara com o header Origin ou Referer da requisição.

3.3 Restrição de IP ( allowedIps )

{
  allowedIps: ['192.168.1.0/24', '10.0.0.5', '2001:db8::/32']
}

IPs ou CIDR blocks autorizados. Suporta IPv4 e IPv6. Lista vazia significa "any IP".

3.4 Rate Limit por Chave

{
  rateLimit: 120  // requisições por minuto (null = usar global)
}

O rate limit é aplicado por organizationId. Se a chave tem rateLimit definido, usa esse valor; caso contrário, usa o limite global da organização.

3.5 Expiração ( expiresAt )

{
  expiresAt: '2025-12-31T23:59:59Z'
}

Data ISO 8601 opcional. Após essa data, a chave é considerada expirada e não pode mais ser usada, mesmo se isActive for true.

Hook useApiKeys

// src/hooks/useApiKeys.ts
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';

export function useApiKeys(organizationId: string) {
  return {
    data: useQuery({
      queryKey: ['apikeys', organizationId],
      queryFn: () => fetch('/api/apikeys').then(r => r.json()),
    }),
    create: useMutation({
      mutationFn: (payload: CreateKeyPayload) =>
        fetch('/api/apikeys', { method: 'POST', body: JSON.stringify(payload) })
          .then(r => r.json()),
      onSuccess: () => queryClient.invalidateQueries({ queryKey: ['apikeys'] }),
    }),
    revoke: useMutation({
      mutationFn: (id: string) =>
        fetch(`/api/apikeys/${id}/revoke`, { method: 'POST' }),
    }),
    delete: useMutation({
      mutationFn: (id: string) =>
        fetch(`/api/apikeys/${id}`, { method: 'DELETE' }),
    }),
  };
}

7.4 Script Node.js

const API_KEY = process.env.JOBSYNC_API_KEY;
const ORG_ID = process.env.JOBSYNC_ORG_ID;

async function enqueueJob(data) {
  const response = await fetch('https://api.jobsync.com/api/jobs', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${API_KEY}`,
      'X-Organization-Id': ORG_ID,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      queueName: 'data-processing',
      data,
    }),
  });

  if (!response.ok) {
    throw new Error(`Job failed: ${response.status} ${await response.text()}`);
  }

  return response.json();
}

// Uso
enqueueJob({ file: 'report.csv', type: 'generate' })
  .then(result => console.log('Job criado:', result.job.id))
  .catch(err => console.error('Erro:', err.message));

7.5 Python

import requests
import os

API_KEY = os.getenv('JOBSYNC_API_KEY')
ORG_ID = os.getenv('JOBSYNC_ORG_ID')

def enqueue_job(queue_name, payload):
    url = 'https://api.jobsync.com/api/jobs'
    headers = {
        'Authorization': f'Bearer {API_KEY}',
        'X-Organization-Id': ORG_ID,
        'Content-Type': 'application/json',
    }
    data = {
        'queueName': queue_name,
        'data': payload
    }
    response = requests.post(url, json=data, headers=headers)
    response.raise_for_status()
    return response.json()

# Uso
result = enqueue_job('image-processing', {'url': 'https://example.com/image.jpg'})
print(f"Job ID: {result['job']['id']}")

8. Matriz de Riscos

Avaliação dos riscos inerentes ao sistema atual:

Risco Probabilidade Impacto Mitigação
Vazamento da API Key
Chave exposta em logs, repositórios, cliente-side		 Alta Crítico • Nunca exponha no frontend
• Use secrets managers
• Revogue imediatamente se suspeitar
Sem rotação automática
Chaves nunca expiram a menos que configurado manualmente		 Média Moderado • Configure expiresAt nas chaves
• Processo manual de rotação périodica
• Auditoria de uso para identificar chaves antigas
IP Whitelist bypass
Proxies/fowards podem alterar o IP real		 Média Moderado • Configure X-Forwarded-For corretamente
• Considere assinatura de requests (HMAC)
Permissões excessivas
Chaves com permissões além do necessário		 Média Moderado • Princípio do menor privilégio
• Auditar permissões periodicamente
• Templates de perfil por caso de uso
Race condition no rate limit
Limite pode ser contornado com requests concorrentes		 Baixa Baixo • Implementar leaky bucket com Redis atomics
• Considerar rate limit por IP + chave combinados
🚨
Melhorias Recomendadas

Considere implementar: (1) expiração automática padrão (ex: 90 dias), (2) notificação de chaves próximas ao vencimento, (3) versionamento de permissões com histórico, (4) webhook de eventos de segurança.

9. Boas Práticas

  • Não commite API Keys: Use .gitignore e variáveis de ambiente
  • Valide allowedDomains: Configure domínios específicos sempre que possível
  • Use permissões mínimas: Apenas o que a integração realmente precisa
  • Monitore usageCount: Picos de uso podem indicar vazamento
  • Revogue chaves inativas: Identifique e limpe chaves não usadas há meses
  • Crie chaves específicas: Uma chave por integração (não compartilhe)
  • Teste em staging: Crie chaves de teste com permissões restritas

10. Troubleshooting

Chave retorna 401 Unauthorized

  • Verifique se o prefixo é jsk_
  • Confirme que a chave não foi revogada (isActive)
  • Verifique se não expirou (expiresAt)
  • Confirme se o organizationId do header está correto

Chave retorna 403 Forbidden

  • Permissão insuficiente para o endpoint/operação
  • IP não autorizado (conferir allowedIps)
  • Domínio não autorizado (conferir allowedDomains e header Origin)

Chave retorna 429 Too Many Requests

  • Rate limit excedido — aguarde ou aumente o limite
  • Verifique se há múltiplas requisições concorrentes

Não consigo recuperar a chave após criá-la

  • Comportamento esperado — a chave completa só é mostrada na criação
  • Revogue a chave antiga e crie uma nova
