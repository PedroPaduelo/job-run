Sistema de Filas BullMQ

Arquitetura completa de processamento assíncrono com BullMQ, Redis e workers inteligentes

Queue Architecture

O JobSync utiliza BullMQ, uma biblioteca de filas baseada em Redis, para processamento assíncrono de jobs. A arquitetura foi desenhada para suportar múltiplas organizações com isolamento completo de recursos.

Nomenclatura de Filas

Cada fila no Redis segue um padrão de nomeação que garante isolamento multi-tenant:

┌────────────────────────────────────────────────────────────────────┐ │ NOME DA FILA NO REDIS │ │ Format: jobs-{orgId}-{queueName} │ │ │ │ Exemplo: jobs-org_abc123def456-marketing-campaigns │ │ └──────┘ └──────────────────────┘ │ │ prefixo nome da fila específico │ └────────────────────────────────────────────────────────────────────┘

Lazy Creation Pattern

As filas são criadas sob demanda (lazy loading) e armazenadas em cache em um Map para reutilização. Isso evita criar múltiplas instâncias da mesma fila e melhora performance.

📦

Arquivo: src/lib/bullmq.ts

const queues = new Map<string, Queue>();

export function getQueue(orgId: string, queueName: string): Queue {
  const key = `jobs-${orgId}-${queueName}`;

  if (!queues.has(key)) {
    queues.set(
      key,
      new Queue(key, {
        connection: redis,
        defaultJobOptions: {
          removeOnComplete: 100,   // Manter últimos 100 completados
          removeOnFail: 1000,      // Manter últimos 1000 falhos
        },
      })
    );
  }

  return queues.get(key)!;
}

Opções Padrão (Default Job Options)

OpçãoValor PadrãoDescrição
removeOnComplete100Mantém os últimos 100 jobs completados na memória do Redis. Jobs mais antigos são removidos automaticamente.
removeOnFail1000Mantém os últimos 1000 jobs falhos. Útil para debug e retry manual.
⚠️
Limpeza Manual

Como o BullMQ limpa automaticamente jobs antigos, a contagem de jobs COMPLETED/FAILED deve ser obtida do PostgreSQL, não do Redis.

Worker Architecture

Os workers são processos independentes que consomem jobs das filas BullMQ. A arquitetura suporta auto-descoberta de novas filas e graceful shutdown.

Inicialização de Workers

🚀

Arquivo: src/worker.ts

async function startWorkers() {
  // 1. Buscar todas as filas ativas do banco
  const queues = await prisma.queue.findMany({
    where: { isActive: true },
    include: { organization: true },
  });

  const workers: Worker[] = [];

  // 2. Criar um worker por fila ativa
  for (const queue of queues) {
    const queueName = `jobs-${queue.organizationId}-${queue.name}`;

    const worker = new Worker<WorkerJobData>(queueName, processJob, {
      connection: redis,
      concurrency: queue.concurrency || CONCURRENCY,
      limiter: {
        max: queue.rateLimit || 100,   // max jobs
        duration: 60000,               // por 60 segundos
      },
    });

    workers.push(worker);
  }

  console.log(`Started ${workers.length} workers`);
}

Auto-Discovery

O worker monitora o banco a cada 30 segundos para detectar novas filas ativas:

┌────────────────────────────────────────────────────────────────────┐ │ WORKER AUTO-DISCOVERY │ ├────────────────────────────────────────────────────────────────────┤ │ │ │ ┌─────────────────────────┐ │ │ │ Workers rodando │ │ │ │ para filas ativas │ │ │ └───────────┬─────────────┘ │ │ │ │ │ (a cada 30s) │ │ │ │ │ ▼ │ │ ┌─────────────────────────┐ │ │ │ Query: SELECT * FROM │ │ │ │ queues WHERE isActive=true │ │ └───────────┬─────────────┘ │ │ │ │ │ ┌─────────┴─────────────────────────┐ │ │ │ Para cada fila: │ │ │ │ Verifica se worker já existe? ─Yes─┐ │ │ └────No──────────────────────────────┘ │ │ │ │ │ │ │ ▼ │ │ │ Criar novo │ │ │ Worker │ │ │ │ │ │ (Workers ficam rodando) │ │ │ └────────────────────────────────────────────────────────────────────┘

Rate Limiter

Cada worker tem um rate limiter configurado que controla quantos jobs daquela fila podem ser processados por período:

  • limiter.max — Número máximo de jobs (padrão: 100)
  • limiter.duration — Período em ms (sempre 60000 = 1 minuto)
  • Exemplo: rateLimit: 200 = máximo de 200 jobs por minuto

Graceful Shutdown

┌────────────────────────────────────────────────────────────────────┐ │ GRACEFUL SHUTDOWN (SIGTERM/SIGINT) │ ├────────────────────────────────────────────────────────────────────┤ │ │ │ Recebe sinal SIGTERM ou SIGINT │ │ │ │ │ ▼ │ │ ┌──────────────────────┐ │ │ │ shutdown() async │ │ │ │ função executada │ │ │ └──────────┬───────────┘ │ │ │ │ │ ▼ │ │ ┌─────────────────────────────────────────────┐ │ │ │ Para cada worker: await worker.close() │ │ │ │ • Para de aceitar novos jobs │ │ │ │ • Aguarda jobs ativos terminarem │ │ │ │ • Fecha conexões Redis │ │ │ └─────────────────────────────────────────────┘ │ │ │ │ │ ▼ │ │ await prisma.$disconnect() │ │ │ │ │ ▼ │ │ process.exit(0) ← Saída limpa │ │ │ └────────────────────────────────────────────────────────────────────┘

Concorrência por Worker

A concorrência é herdada da configuração da fila no banco. Se não configurada, usa o valor do WORKER_CONCURRENCY do ambiente (padrão: 10):

const CONCURRENCY = parseInt(process.env.WORKER_CONCURRENCY || '10', 10);

// Na criação do worker:
concurrency: queue.concurrency || CONCURRENCY

Job Lifecycle

Cada job passa por um estado definido durante seu ciclo de vida. O BullMQ gerencia a transição entre estados PENDING, PROCESSING e estados finais, enquanto o banco armazena o histórico.

Retry Strategy

Jobs falhos são automaticamente重新acions de acordo com a estratégia de exponential backoff configurada na fila.

Cálculo do Delay

⏱️

Backoff Exponencial

delay = retryDelay × 2^(attempt - 1)

Onde:
- retryDelay = configuração da fila (padrão: 5000ms)
- attempt = número da tentativa atual (começa em 1)
- delay final = tempo de espera antes da próxima tentativa

Exemplos

TentativaCálculo (retryDelay=5s)Delay TotalDelay Acumulado
1 (falha)5000 × 2^05s5s
25000 × 2^110s15s
35000 × 2^220s35s
45000 × 2^340s1m 15s
5 (max=5)— (atingiu maxRetries)——

Configuração no Job

A strategy de retry é definida ao adicionar o job à fila (em job.service.ts):

await bullQueue.add('process', jobData, {
  attempts: queue.maxRetries,      ← Máximo de tentativas
  backoff: {
    type: 'exponential',
    delay: queue.retryDelay,       ← Delay base em ms
  },
  delay: scheduledAt ? ... : undefined,
});
ℹ️
Nota sobre Dead Letters

Quando attemptsMade >= maxRetries, o job é marcado como DEAD e não será mais processado automaticamente. É possível retry manual via API (POST /jobs/:id/retry).

Stats Collection

As estatísticas são coletadas de duas fontes: Redis (via BullMQ) para dados em tempo real, e PostgreSQL para dados históricos.

Fontes de Dados

🔴

Redis / BullMQ (Real-time)

  • getWaitingCount() — Jobs aguardando na fila (PENDING)
  • getActiveCount() — Jobs atualmente em processamento
  • getDelayedCount() — Jobs com retry agendado

Fonte: queue.service.ts:getStats()

🟢

PostgreSQL (Histórico)

  • COUNT COMPLETED — Total de jobs concluídos
  • COUNT FAILED + DEAD — Total de jobs falhos
dbStats = await prisma.job.groupBy({
  by: ['status'],
  where: { queueId: id },
  _count: true,
});

Implementação

async getStats(organizationId: string, id: string) {
  const queue = await prisma.queue.findFirst({
    where: { id, organizationId },
  });

  if (!queue) return null;

  const bullQueue = getQueue(organizationId, queue.name);

  // Coleta paralela
  const [waiting, active, delayed, dbStats] = await Promise.all([
    bullQueue.getWaitingCount(),
    bullQueue.getActiveCount(),
    bullQueue.getDelayedCount(),
    prisma.job.groupBy({
      by: ['status'],
      where: { queueId: id },
      _count: true,
    }),
  ]);

  const statusCounts = dbStats.reduce((acc, item) => {
    acc[item.status] = item._count;
    return acc;
  }, {});

  return {
    id: queue.id,
    name: queue.name,
    waiting,
    active,
    completed: statusCounts['COMPLETED'] || 0,
    failed: (statusCounts['FAILED'] || 0) + (statusCounts['DEAD'] || 0),
    delayed,
  };
}

Por que não usar BullMQ para completed/failed?

ℹ️
Limpeza automática

O BullMQ remove jobs antigos conforme configurado (removeOnComplete: 100, removeOnFail: 1000). Isso significa que contagens históricas seriam imprecisas. O PostgreSQL armazena todo o histórico permanentemente.

Riscos e Considerações

A arquitetura BullMQ oferece robustez, mas requer atenção a pontos críticos:

1. Perda de Conexão Redis

🔴
Impacto: Crítico

Se o Redis ficar indisponível, workers param de processar e novas filas não podem ser criadas. Jobs aguardando na fila bloquearão timeouts do cliente.

  • Mitigação: Redis Cluster com replicação, sentinel mode, alertas de health check.
  • Código: redis.on('error') loga erros. Workers tentam reconectar automaticamente.

2. Colisão de Nomes de Fila

⚠️
Impacto: Alto

Se duas organizações usarem o mesmo queueName, não há risco pois o orgId garante isolamento. O risco está em que nomes duplicados na mesma org — o sistema deve rejeitar criar filas com nomes duplicados.

  • Banco: unique constraint (organizationId, name)
  • Código: Validar com getByName() antes de criar, ou capturar erro de unique constraint.

3. Worker Starvation

🔴
Impacto: Alto

Fila com concurrency=1 mas jobs longos podem gerar backlog. Muitas filas concorrentes podem saturar workers.

  • Monitorar: Stats de waiting crescendo sem active acompanhando.
  • Ajustar: Aumentar concurrency (limite max 100) ou adicionar mais workers.
  • Rate limit: Use rateLimit para evitar sobrecarga por spikes.

4. Memory Pressure no Redis

⚠️
Impacto: Médio

Muitas filas ativas com alta concorrência geram muitas chaves no Redis. removeOnFail: 1000 ajuda mas falhas massivas podem crescer.

  • maxmemory-policy: Configurar Redis com volatile-lru ou allkeys-lru
  • Monitoring: redis-cli info memory, alertas de uso >80%
  • Limpeza manual: Queue.clean(completed, failed) para remover batches

5. Timeouts e Zombie Jobs

⚠️
Impacto: Médio

Se um worker morrer durante processamento, o job fica "preso" em PROCESSING. O BullMQ tem stampede protection que automaticamente reassigna após timeout.

  • stuckPeriod: Configurar settings.stuckPeriod (padrão: ~30s × attempts)
  • Monitor: Jobs em PROCESSING por muito tempo devem ser investigados.

6. Database vs Redis Consistency

A arquitetura usa dual-write:

  1. Job criado → PostgreSQL INSERT
  2. Job adicionado → BullMQ Queue.add()

Risco: Se uma das operações falhar, pode haver inconsistência.

  • Atualmente: Sem transação distribuída (2PC).
  • Recuperação: Jobs órfãos no Redis sem对应 PG podem ser limpos; jobs no PG sem Redis serão re-adicionados via retry manual.
