Sensitive Data in Metadata

Metadados de jobs podem conter PII ou secrets

🔐 O Problema

Job metadata é um campo JSON livre (z.record(z.unknown())). Usuários podem colocar qualquer coisa, incluindo dados sensíveis.

🛡️
PII Leak

Se metadata contém CPF, email, tokens, esses dados ficam salvos em Clear Text no banco. Qualquer pessoa com acesso ao DB vê.

📊 Schema

model Job {
  id          String
  metadata    Json?    // ← Sem restrição
  payload     Json    // ← Também é livre
  result      Json?
  ...
}

Não há:

  • Encrption at rest (PostgreSQL pgcrypto)
  • Validação de tamanho/type
  • Sanitização de PII
  • Retention policy (metadata mora junto com job)

💥 Cenários de Risco

  1. Cliente envia payload com token de terceiro (Stripe, Google) em metadata
  2. Auditoria requisita logs, e metadata com PII é extraída
  3. DB breach — attacker lê todas metadata (usuários, clientes, integrations)
  4. Backup em S3 — pg_dump com dados sensíveis

✅ Recomendações

  1. Documentar que metadata NÃO deve conter secrets — aviso na API reference
  2. Validar tamanho máximo — limite 4KB (PostgreSQL JSON max ~1GB, mas ideal limitar)
  3. Criptografar campos específicos — se necessário, usar aplicação-level encryption: 
    metadata: z.record(z.string()).refine((obj) => !containsSecret(obj), {
  message: "metadata must not contain sensitive data"
})
  4. Masking em logs — JobLog metadata deve ser sanitizada antes de salvar
  5. Retention por tipo — se job tipo "payment", reter metadata por X dias; se "log", Y dias
  6. Audit tools — script para scan de PII na tabela jobs (usando regexcpf, email, token patterns)

🔍 Checklist

  • Schema valida tamanho de metadata? ❌
  • Há política de retenção diferenciada? ❌
  • Metadata é encrypted? ❌ (app-level)
  • Logs sanitizam metadata? ❌
  • Docs avisam sobre PII? ⚠️
  • Teste de penetração valida PII leak? ❌
