Queue Detail Page

Documentação de UI e fluxo de dados da página de detalhe de fila (QueueDetail).

1. URL

A rota de detalhe da fila é:

/queues/:id

O id é o identificador da fila persistida no backend (campo queue.id).

2. Dados carregados

O componente QueueDetail usa as queries do hooks para carregar dados paralelamente:

  • useQueue(id) — detalhes da fila (nome, descrição, configuração, status ativo/pausado)
  • useQueueStats(id) — métricas em tempo real (waiting, active, completed, failed, delayed)

Além disso, a lista de jobs recentes é carregada via:

useJobs({ queue: queue?.name, limit: 10 })

As queries renderizam placeholders de loading e fallback de fila não encontrada.

3. O que é exibido

3.1 Informações principais

  • Nome da fila (queue.name)
  • Textos auxiliares e descrição de contexto
  • Status: Active / Inactive Badge
  • Botões de pausa/resume quando aplicável

3.2 Estatísticas atuais

  • Waiting
  • Active
  • Completed
  • Failed
  • Delayed

Estes valores vêm de useQueueStats e normalmente agregam dados de BullMQ + DB para exibir a visão operativa.

3.3 Configuração editável

  • Concurrency
  • Max Retries
  • Timeout
  • Rate Limit (jobs/min)

Os campos são inicializados no useEffect a partir de queue e enviados via useUpdateQueue.

3.4 Jobs recentes

O card de Recent Jobs mostra até 10 jobs filtrados por nome da fila. Cada job exibe:

  • ID parcial e link para /jobs/:id
  • Status com badge colorido
  • Tentativas (attempts/maxAttempts)
  • Data de criação em formato relativo

4. Ações de gerenciamento

QueueDetail expõe ações de mutação:

  • useUpdateQueue() — atualiza a configuração da fila (PATCH)
  • usePauseQueue() — pausa a fila
  • useResumeQueue() — retoma a fila

O botão de salvar usa o ícone Save e envia dados como:

{`{ concurrency, maxRetries, timeout, rateLimit }`}

O status de pausa e resume é controlado via queue.isPaused.

5. Tempo real / Refresh

O componente depende do cache e refetch da biblioteca de queries (TanStack Query). O backend define tempos de revalidação via constantes (ex. REFETCH_INTERVALS.QUEUE_STATS), e a UI exibe stats atualizados periodicamente.

Isso garante que o card de métricas se mantenha perto do estado real de execução do BullMQ.

6. Permissões

As permissões são aplicadas no backend. Na UI, o acesso à rota e às ações deve ser condicionado ao usuário ter permissões como:

  • queues:read para visualizar
  • queues:update para editar configurações
  • queues:pause / queues:resume para controlar execução

Mesmo que a UI mostre botões, a validação final deve ser no endpoint.

7. Navegação

Links principais na página:

  • <Link to="/queues"> — volta para a lista de filas
  • <Link to={`/jobs?queue=${queue.name}`}> — vê todos os jobs filtrados pela fila

8. Estados especiais

O QueueDetail trata os seguintes estados:

  • Loading: exibe texto de loading enquanto a query está pendente.
  • Queue não encontrada: mostra mensagem e botão de retorno.
  • Sem jobs recentes: exibe linha de tabela com "No jobs in this queue".
