JobSync Frontend

Arquitetura Frontend

Um mergulho profundo na estrutura, padrões e decisões técnicas que definem o frontend da aplicação JobSync.

1. Padrão Organizacional

O frontend do JobSync segue uma organização baseada em features, mantendo cada funcionalidade autossuficiente e isolada dentro da estrutura de pastas.

Estrutura de Pastas

  • /src/pages/ - Páginas agrupadas por feature
  • /src/lib/ - Código compartilhado
  • /src/hooks/ - Hooks customizados
  • /src/components/ - Componentes específicos

Princípios

  • Cada página contém sua própria lógica
  • Hooks encapsulam queries/mutações
  • Lib contém código reutilizável
  • Componentes genéricos ficam em @nomad/ui

A estratégia de componentes prioriza a consistência visual através da biblioteca @nomad/ui (shadcn/uiustomizada), mantendo poucos componentes reutilizáveis específicos.

Fontes de Componentes

Decisão de Design

Formulários são construídos diretamente nas páginas em vez de serem abstraídos em componentes reutilizáveis. Isso mantém a lógica de validação e transformação próxima do uso, facilitando a manutenção.

3. Estratégia de Data Fetching

O JobSync utiliza React Query (TanStack Query) para gerenciamento de estado assíncrono, implementando padrões claros para operações de leitura e escrita.

Padrões de Uso

Leitura (useQuery)

  • Use para operaç��es GET
  • Auto-refetch configurável
  • Refetch em intervalos para dados em tempo real
  • Dados cacheados automaticamente

Escrita (useMutation)

  • Use para POST/PUT/DELETE
  • Invalidate queries após sucesso
  • Rollback em caso de erro
  • Feedback visual com Toast

Invalidation

  • queryClient.invalidateQueries(['queues'])
  • Força refetch das queries afetadas
  • Garante sincronia de cache
  • Evita dados stale no UI

Optimistic Updates

  • Não implementado atualmente
  • Considerar para UX mais fluida
  • Rollback manual em erros

Exemplo: useCreateQueue

export function useCreateQueue() {
  const queryClient = useQueryClient()
  const { toast } = useToast()

  return useMutation(api.createQueue, {
    onSuccess: () => {
      // Invalida cache para forçar refetch
      queryClient.invalidateQueries({ queryKey: ['queues'] })
      toast({
        title: 'Fila criada',
        description: 'A fila foi criada com sucesso.'
      })
    },
    onError: (error) => {
      toast({
        title: 'Erro',
        description: error.message,
        variant: 'destructive'
      })
    }
  })
}

5. Tratamento de Erros

A estratégia de tratamento de erros combina notificações visuais (Toast) com rastreamento automatizado (Sentry), garantindo feedback ao usuário e visibilidade para desenvolvedores.

Camadas de Error Handling

Toast Notifications

  • Hook: useToast()
  • Feedback imediato ao usuário
  • Sucesso/erro/aviso/info
  • Implementado nos callbacks de mutation

Query Errors

  • Errors sobem para mutations
  • Capturados com try/catch no UI
  • Toast exibido no catch
  • Query retry automático configurável

Sentry

  • Integrado em lib/sentry.ts
  • Captura erros não tratados
  • Performance monitoring
  • Contexto de usuário incluído

Padrão de Implementação

// Em uma mutation hook
export function useCreateJob() {
  const queryClient = useQueryClient()
  const { toast } = useToast()

  return useMutation(api.createJob, {
    onSuccess: () => {
      queryClient.invalidateQueries(['jobs'])
      toast({ title: 'Job criado!' })
    },
    onError: (error) => {
      // Toast para o usuário
      toast({
        title: 'Erro',
        description: error.message,
        variant: 'destructive'
      })
      // Sentry captura automaticamente via global handler
    }
  })
}

// No componente
const Component = () => {
  const createJob = useCreateJob()

  const onSubmit = async (data) => {
    try {
      await createJob.mutateAsync(data)
    } catch (error) {
      // tratamento local se necessário
    }
  }
}

6. Roteamento e Autorização

O roteamento implementa controle de acesso através de um wrapper que verifica autenticação antes de renderizar páginas protegidas.

Tipos de Rotas

Tipo Rotas Acesso
Públicas /login, /callback Acesso sem autenticação
Protegidas /queues, /jobs, /settings Requer isAuthenticated

ProtectedRoute Component

// src/components/ProtectedRoute.tsx
export function ProtectedRoute({ children }: { children: ReactNode }) {
  const { isAuthenticated, isLoading } = useAuth()

  if (isLoading) {
    return <LoadingFallback />
  }

  if (!isAuthenticated) {
    // Redirect to login, preserve intended destination
    navigate('/login', { state: { from: location.pathname } })
    return null
  }

  return children
}

// Uso nas rotas
<Route
  path="/queues"
  element={
    <ProtectedRoute>
      <QueuesPage />
    </ProtectedRoute>
  }
/>

7. Abstração de API

Todos os endpoints são centralizados em um único arquivo de tipagem, eliminando duplicação e garantindo consistência entre frontend e API.

api.ts - Endpoints Centralizados

// src/lib/api.ts
export const api = {
  // Queues
  getQueues: () => ({ method: 'GET', url: '/api/v1/queues' }),
  getQueue: (id: string) => ({ method: 'GET', url: `/api/v1/queues/${id}` }),
  createQueue: (data: CreateQueueDto) => ({
    method: 'POST',
    url: '/api/v1/queues',
    body: data
  }),
  deleteQueue: (id: string) => ({
    method: 'DELETE',
    url: `/api/v1/queues/${id}`
  }),

  // Jobs
  getJobs: (params?: GetJobsParams) => ({
    method: 'GET',
    url: '/api/v1/jobs',
    params
  }),
  getJob: (id: string) => ({ method: 'GET', url: `/api/v1/jobs/${id}` }),
  createJob: (data: CreateJobDto) => ({
    method: 'POST',
    url: '/api/v1/jobs',
    body: data
  }),
  cancelJob: (id: string) => ({
    method: 'POST',
    url: `/api/v1/jobs/${id}/cancel`
  }),

  // Organizations
  listOrganizations: () => ({ method: 'GET', url: '/api/v1/orgs' }),
  getOrganization: (id: string) => ({
    method: 'GET',
    url: `/api/v1/orgs/${id}`
  }),
}

Custom Fetch Wrapper

// src/lib/fetch.ts - sem axios
async function fetchApi(
  config: RequestConfig,
  signal?: AbortSignal
): Promise<T> {
  const { token } = useAuth.getState()

  const response = await fetch(config.url, {
    method: config.method,
    headers: {
      'Content-Type': 'application/json',
      ...(token && { Authorization: `Bearer ${token}` }),
      ...config.headers
    },
    body: config.body ? JSON.stringify(config.body) : undefined,
    signal,
    credentials: 'include'
  })

  if (!response.ok) {
    const error = await response.json()
    throw new ApiError(error.message, response.status)
  }

  return response.json()
}

Nota sobre Axios

O projeto NÃO utiliza axios. A escolha pelo fetch nativo reduz bundle size e simplifica configuração, com error handling customizado para compensar a API menos rica.

Resumo da Arquitetura

O JobSync frontend emprega uma arquitetura moderna, baseada em React com TypeScript, focada em manutenibilidade e experiência do desenvolvedor.

Stack Tecnológica

  • Framework: React 18+ com TypeScript
  • Router: React Router v6
  • Data Fetching: TanStack Query v5
  • UI Library: @nomad/ui (shadcn/ui customizada)
  • Styling: Tailwind CSS
  • Forms: React Hook Form + Zod
  • Auth: SDK custom + OAuth2
  • Error Tracking: Sentry

Principais Padrões

  • Feature-based structure para organização escalável
  • React Query como camada de estado assíncrono
  • Hooks customizados para encapsulamento de lógica
  • API centralizada em lib/api.ts
  • Contexts para auth e organização
  • Toast notifications para feedback
✓
Pontos Fortes

A arquitetura favorece a manutenibilidade, testabilidade e experiência do desenvolvedor, com separação clara de responsabilidades e stack atualizada.
