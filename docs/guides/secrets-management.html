Secrets Management

Gerenciamento de .env, DSNs, integração com vault

🔐 Status Atual

Usa dotenv para carregar variáveis de ambiente de arquivo .env.

// src/server.ts
import 'dotenv/config';

// .env.example
DATABASE_URL="postgresql://..."
REDIS_HOST="..."
REDIS_PORT="6379"
SENTRY_DSN="https://..."
JWT_PUBLIC_KEY="-----BEGIN..."
API_SECRET="..."
⚠️
Sem Vault

Secrets estão em arquivo .env. Em produção, recomenda-se usar vault (HashiCorp, AWS Secrets Manager).

📋 Secrets Necessários

VariávelSensível?Notas
DATABASE_URL✅Contém usuário/senha DB
REDIS_PASSWORD✅Se Redis requer auth
JWT_PUBLIC_KEY✅Chave de validação JWT
SENTRY_DSN⚠️Não segredo, mas URL única
API_SECRET✅Se tem autenticação interna
PORT❌Não sensível
NODE_ENV❌Não sensível

💥 Riscos Atuais

  • .env no git — se commitado, expõe todos secrets
  • Hardcoded fallback — se .env não existe, código pode ter defaults perigosos
  • Rotação manual — precisa restartar app para aplicar novo secret
  • Sem audit trail — não sabe quem acessou qual secret

✅ Recomendações

  1. Adicionar .env ao .gitignore: 
    # .gitignore
.env
.env.local
.env.*.local
  2. Usar .env.example com placeholders (já tem)
  3. Vault Integration (futuro): 
    // Substituir dotenv por vault
import { Vault } from 'hashi-vault';
const vault = await Vault.fromEnv();
const dbPassword = await vault.read('secret/data/db-password');
  4. Kubernetes Secrets (se usar K8s): 
    // mount como env vars ou volumes
envFrom:
  - secretRef:
      name: jobsync-secrets
  5. Secret Rotation — ao rodar, verificar se secret mudou: 
    // Health check pode detectar secret outdated
// e warnar/alertar
  6. Encrypt .env at rest — se .env está em disco, encrypt com KMS

🔍 Código Atual

// Verificar se .env está no git
grep -q "\.env" .gitignore && echo "OK" || echo "FALTA .gitignore"

🔍 Checklist

  • .env no .gitignore? ✅
  • .env.example com placeholders? ✅
  • Vault ou K8s secrets? ❌
  • Secret rotation automation? ❌
  • Audit de secrets? ❌
  • Sem hardcoded secrets no código? ⚠️
