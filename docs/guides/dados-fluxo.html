Fluxo Completo de Dados do JobSync

Documentação end-to-end do fluxo de dados interno para criação, processamento, callbacks e frontend.

Resumo

Este guia descreve os fluxos internos do JobSync com diagramas ASCII, cada passo numerado e pontos de risco/considerações para garantir robustez operacional.

ℹ️
Importante: Essa página apresenta o fluxo de dados em português (pt-BR) e foca em design arquitetural e operações críticas.

Flow 1: Job Creation

CLIENTE → API /api/jobs → BACKEND → DB + FILA [Cliente] POST /api/jobs Payload: {queue, webhookUrl, webhookMethod, payload, callbackUrl?, priority?, scheduledAt?} | v [Fastify] Route /api/jobs - Validação Zod (createJobSchema) - Autenticação Bearer (JWT/API Key) - organizationId no request | v [JobService.create()] - Find or create queue - Cria Job no PostgreSQL (PENDING/SCHEDULED) - Add BullMQ: jobs-{orgId}-{queueName} - priority + attempts + retry/backoff | v [Resposta HTTP] {id, status, queue, createdAt}
  1. Cliente envia POST /api/jobs com {queue, webhookUrl, webhookMethod, payload, callbackUrl?, priority?, scheduledAt?}.
  2. Fastify valida com Zod (createJobSchema).
  3. Middleware auth verifica Bearer token (JWT ou API Key), seta organizationId.
  4. JobService.create() encontra ou cria fila pelo nome.
  5. Cria registro Job no PostgreSQL via Prisma (status: PENDING ou SCHEDULED).
  6. Adiciona job na fila BullMQ jobs-{orgId}-{queueName} com prioridade, attempts, exponential backoff.
  7. Retorna {id, status, queue, createdAt}.
⚠️
Ponto de risco: validação de payload e organização é crítica. Campos faltantes levam a criação incorreta de jobs e uso indevido de recursos.

Considerações: Use transações para criar job + job logs quando necessário; garantir idempotência de APIs de criação de job para clientes que reenviam requisições.

Flow 2: Job Processing (Worker)

BULLMQ → WORKER → WEBHOOK → RESULT [BullMQ Queue jobs-{orgId}-{queueName}] | v [Worker Poll] - Pop job - set status PROCESSING - startedAt, attempts++ | v [HTTP Client: webhookUrl] - GET => query params - POST/PUT/PATCH => JSON body - DELETE => sem payload ou com query - AbortController timeout | sucesso? → sim → COMPLETED + result + JobLog | ↘ callbackUrl (POST) falha? → não → retry (exponential backoff) → attempts | ↘ se max attempts => DEAD + error + failure callback
  1. Worker faz polling na fila BullMQ.
  2. Atualiza job para PROCESSING, startedAt e attempts++.
  3. Executa requisição HTTP para webhookUrl com método correto.
  4. GET envia payload como query params; POST/PUT/PATCH enviam JSON body; DELETE envia conforme contrato.
  5. AbortController aplica timeout da request.
  6. Sucesso: status COMPLETED, grava resultado, cria JobLog.
  7. Sucesso + callbackUrl: faz POST para callbackUrl com {jobId, status, result, duration, attempts}.
  8. Falha: BullMQ faz retry com exponential backoff.
  9. Se atingir tentativas máximas: status DEAD, armazena erro, envia callback de falha.
🛑
Ponto de risco: requests para webhook podem travar; retry/backoff e timeout devem ser bem calibrados para evitar acumulo de jobs travados.

Considerações: Assegure idempotência no endpoint do webhook para evitar efeitos colaterais quando o job for reprocessado após falha.

Flow 3: Bulk Job Creation

POST /api/jobs/bulk → Validação → Batch DB + BullMQ [Cliente] POST /api/jobs/bulk Payload: {queue, jobs: [{webhookUrl, payload, ...}]} | v [Backend] Validação e Auth - organizationId - permissões | v [Service] Process batches de 1000 - Prisma createMany - BullMQ addBulk | v [Resposta] {created, failed, queue}
  1. POST /api/jobs/bulk com {queue, jobs:[...]}
  2. Validação e autenticação (mesma estrutura que criação single job).
  3. Processamento em batch de até 1000 jobs.
  4. Inserção em massa via Prisma createMany + BullMQ addBulk.
  5. Retorna resumo {created, failed, queue}.
✅
Melhoria de performance: bulk reduz overhead de transações individuais, mas exige checagem de tamanho máximo e fallback em pequenos batches ao falhar.

Flow 4: Authentication Flow

JWT e API Key JWT Path: [Bearer JWT] → verifyJWT (JWKS) → syncUser → syncOrganization → syncMembership → buildAuthenticatedUser API Key Path: [Bearer API Key] → findByKey → validate(active, expiration, IP allowlist) → set organizationId + permissions
  1. JWT: chama verifyJWT via JWKS, sincronia de usuário e organização, membership carregado, create authenticated user context.
  2. API Key: busca na tabela, valida isActive, expiresAt, IP allowlist (se configurado), define organizationId e permissions no request.
⚠️
Ponto de risco: validar expiração e revogação de API Key constantemente. Considere cache TTL curto para JWKS e keys inválidas.

Considerações: Use sempre contextos de request sem efeitos colaterais; não permita dados de token do usuário em APIs de máquina (API Key). Use logs de auditoria para autenticação falha.

Flow 5: Frontend Data Flow

React → useQuery/useMutation → API → Backend [React Components] useJobs, useQueues, useStats | v [API Client] fetch com Authorization: Bearer {token} X-Organization-Id | v [Backend API] /api/jobs, /api/queues, /api/stats Mutations: useMutation(create/cancel/retry) → onSuccess invalidates queries → auto-refetch Polling: jobs list, stats overview, timeline, queue stats
  1. React usa hooks de dados (useQuery) para consumir listagens e estatísticas.
  2. API client injeta Bearer token e X-Organization-Id em todas as requisições.
  3. Mutations (create, cancel, retry) usam useMutation; no success invalidam queries e causam refetch automático.
  4. Polling configurado para jobs, stats e filas para UI em tempo quase real.
ℹ️
Consideração de UX: para grandes listas, use paginação e cache de scroll para evitar excesso de consultas a API.

Considerações Gerais de Risco por Etapa

  • Validação: sempre validar com Zod e garantir retorno de erro estruturado para debugging.
  • Autenticação/Autorização: falhas aqui expõem dados multi-tenant. Use middlewares centralizados.
  • Processamento de Jobs: timeout, retries e logs são essenciais; atualize as taxas de retry e backoff conforme SLA.
  • Callback Externo: trate respostas 4xx/5xx e evite loops de callback recursivo com deduplicação por job.
  • Failover Redis: BullMQ depende de Redis; monitore latência e memória para evitar filas paradas.
✅
Operacional: implemente health checks no worker e no backend; use dashboards e alertas para filas altas, jobs dead, erros 5xx e timeouts.
