Trilha de Auditoria - Análise Técnica

Análise detalhada do modelo AuditLog, uso no código, riscos e recomendações

1. Modelo AuditLog no Prisma

O modelo principal de auditoria define a estrutura de armazenamento de logs:

model AuditLog {
  id            String   @id @default(cuid())

  // ┌─────────────────────────────────────────────┐
  // │  CONTEXTO DE IDENTIFICAÇÃO                 │
  // └─────────────────────────────────────────────┘
  userId         String?       // Usuário que realizou ação
  user           User?         @relation(fields: [userId], onDelete: SetNull)
  organizationId String?       // Organização (tenant)
  organization   Organization? @relation(fields: [organizationId], onDelete: SetNull)

  // ┌─────────────────────────────────────────────┐
  // │  O QUE FOI FEITO                            │
  // └─────────────────────────────────────────────┘
  action     String           // Ex: "job.created", "apikey.revoked"
  entityType String?          // Ex: "Job", "Queue", "ApiKey"
  entityId   String?          // ID da entidade afetada

  // ┌─────────────────────────────────────────────┐
  // │  DETALHES                                  │
  // └─────────────────────────────────────────────┘
  metadata  Json?             // Dados extras (JSON flexível)
  ipAddress String?           // IP do cliente
  userAgent String?           // User-Agent do navegador/API

  // ┌─────────────────────────────────────────────┐
  // │  RESULTADO                                 │
  // └─────────────────────────────────────────────┘
  status AuditStatus @default(SUCCESS)  // SUCCESS | FAILURE | DENIED

  createdAt DateTime @default(now())
}

enum AuditStatus {
  SUCCESS   // Ação completada com sucesso
  FAILURE   // Erro ocorreu durante a ação
  DENIED    // Usuário não tinha permissão
}

Observações sobre o Modelo

  • userId opcional: Permite logs de ações system/API Key sem usuário
  • organizationId opcional: Ações no contexto da aplicação podem não ter org
  • metadata JSONB: Flexível mas sem validação de schema
  • SetNull em cascata: Se usuário/org for deletado, logs mantêm-se com null

5. Consultas de Auditoria

Buscar Logs por Organização (últimos N dias)

-- Período configurável
SELECT
  id,
  userId,
  action,
  entityType,
  entityId,
  metadata,
  ipAddress,
  userAgent,
  status,
  createdAt
FROM audit_logs
WHERE organizationId = 'org_12345'
  AND createdAt >= NOW() - INTERVAL '90 days'
ORDER BY createdAt DESC
LIMIT 1000;

Histórico Completo de uma Entidade

-- Rastrear todas as mudanças em um Job específico
SELECT
  action,
  userId,
  ipAddress,
  userAgent,
  metadata,
  status,
  createdAt
FROM audit_logs
WHERE entityType = 'Job'
  AND entityId = 'job_abc123'
ORDER BY createdAt ASC;

Logins por Período (Dashboard)

-- Estatística diária de autenticações
SELECT
  DATE(createdAt) as date,
  COUNT(*) FILTER (WHERE status = 'SUCCESS') as successful_logins,
  COUNT(*) FILTER (WHERE status = 'DENIED') as failed_logins,
  COUNT(DISTINCT userId) as unique_users
FROM audit_logs
WHERE action = 'user.auth.success'
  AND createdAt >= NOW() - INTERVAL '30 days'
GROUP BY DATE(createdAt)
ORDER BY date DESC;

Auditoria de Permissões Negadas

-- Identificar usuários/permissões problemáticas
SELECT
  userId,
  organizationId,
  COUNT(*) as denial_count,
  jsonb_array_elements_text(metadata->'required') as missing_permission
FROM audit_logs
WHERE action = 'permission.denied'
  AND createdAt >= NOW() - INTERVAL '7 days'
GROUP BY userId, organizationId, missing_permission
ORDER BY denial_count DESC
LIMIT 50;

6. Riscos Identificados

🔴 Risco Alto: Logs Sensíveis em metadata

Problema: O campo metadata é JSONB sem validação. Desenvolvedores podem (involuntariamente) logar:

  • Senhas ou tokens de autenticação
  • Dados pessoais (email, CPF, dados de cartão)
  • Chaves API completas
  • Payloads de jobs com dados sensíveis

Impacto: Vazamento de dados em logs permanentes; não compliance com GDPR/LGPD.

🔴 Risco Alto: Volume Cresce Indefinidamente

Problema: Sem política de retenção, a tabela cresce linearmente:

  • 1.000 req/dia → ~365 mil logs/ano
  • 10.000 req/dia → ~3.65 milhões logs/ano (~5-10GB)
  • 100.000 req/dia → ~36 milhões logs/ano (~50-100GB)

Impacto: Queries lentas, custo de storage alto, manutenção de índices onerosa.

🟡 Risco Médio: GAP de Logging

Problema: Apenas autenticação/autorização são auditados. Operações CRUD de Jobs, Queues, API Keys não possuem logs.

  • Impossível rastrear quem criou/cancelou um job
  • Sem evidência de modificações em queues
  • API key creations/revokes invisíveis na trilha

Impacto: Forensic incompleta; accountability reduzida; compliance falha.

🟡 Risco Médio: Fire-and-Forget Silencioso

Problema: logAudit() captura erros apenas no console:

try { await prisma.auditLog.create(...) } catch (e) { console.error(e) }

Impacto: Se PostgreSQL estiver down/sobrecarregado, logs são perdidos sem alerta. Não há fallback (fila Redis, arquivo local).

🟢 Risco Baixo: Índices Subótimos

Problema: Consultas comuns como WHERE organizationId = ? ORDER BY createdAt DESC usam filesort.

Sugestão: Índices compostos adicionais:

  • CREATE INDEX audit_logs_org_created ON audit_logs(organizationId, createdAt DESC);
  • CREATE INDEX audit_logs_user_created ON audit_logs(userId, createdAt DESC);

🟡 Risco Médio: Sem Particionamento

Problema: Tabela únicaGrowing = operações pesadas para DELETE/ARCHIVE.

Sugestão: Particionar por mês (PostgreSQL RANGE partition):

  • Partição ativa: audit_logs_2026_03
  • Arquivo morto: audit_logs_2025_01 (somente leitura)

7. Recomendações de Retenção e Masking

📅 Politica de Retenção Recomendada

🗑️
Logs de Sucesso (SUCCESS): Manter 90 dias na tabela ativa. Após isso, mover para tabela de arquivo morto (audit_logs_archive) ou data warehouse (S3/ClickHouse). Manter indexação reduzida.
⚠️
Logs de Falha/Denied (FAILURE/DENIED): Manter 365 dias (1 ano) — informações de segurança são mais sensíveis. Justificativa para forensic.
🔐
Logs de Erro (error.occurred): Manter 180 dias, mas redigir mensagens de erro que possam conter dados sensíveis (ex: stack traces com tokens).

🎭 Data Masking / Sanitização

Modificar logAudit() para filtrar campos sensíveis antes de salvar:

const SENSITIVE_KEYS = ['password', 'token', 'apiKey', 'secret', 'authorization'];

function sanitizeMetadata(metadata: Record): Record {
  const sanitized = { ...metadata };

  for (const key of SENSITIVE_KEYS) {
    if (sanitized[key]) {
      sanitized[key] = '[REDACTED]';
    }
  }

  // Sanitizar recursivamente em arrays/objects aninhados
  if (sanitized.payload && typeof sanitized.payload === 'object') {
    sanitized.payload = deepRedact(sanitized.payload);
  }

  return sanitized;
}

🔄 Implementação de Retenção Automática

Job agendado para limpeza/arquivamento:

// Usar pg_cron ou worker externo
// Exemplo de migration SQL:
CREATE OR REPLACE FUNCTION archive_old_audit_logs()
RETURNS void AS $$
DECLARE
  cutoff_date timestamptz := NOW() - INTERVAL '90 days';
BEGIN
  -- Mover logs antigos para tabela archive
  INSERT INTO audit_logs_archive
  SELECT * FROM audit_logs
  WHERE createdAt < cutoff_date
    AND status = 'SUCCESS';

  DELETE FROM audit_logs
  WHERE createdAt < cutoff_date
    AND status = 'SUCCESS';
END;
$$ LANGUAGE plpgsql;

📊 Estratégia de Particionamento

-- Criar particoes mensais automaticamente
CREATE TABLE audit_logs (
  id TEXT NOT NULL,
  userId TEXT,
  organizationId TEXT,
  action TEXT NOT NULL,
  entityType TEXT,
  entityId TEXT,
  metadata JSONB,
  ipAddress TEXT,
  userAgent TEXT,
  status AuditStatus NOT NULL DEFAULT 'SUCCESS',
  createdAt TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP
) PARTITION BY RANGE (createdAt);

-- Partição atual (escrita)
CREATE TABLE audit_logs_2026_03 PARTITION OF audit_logs
FOR VALUES FROM ('2026-03-01') TO ('2026-04-01');

-- Partições passadas (apenas leitura)
CREATE TABLE audit_logs_2026_02 PARTITION OF audit_logs
FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');

🚨 Alertas e Monitoramento

  • Taxa de crescimento: Alerta se tabela cresce > 10% ao mês
  • Logs faltando: Alertar se falhas na função logAudit() (Sentry já captura, mas criar métrica)
  • Conformidade: Dashboard periodicamente mostrando % de entidades auditadas

8. Compliance: Reconstrução Histórico Completo

Capacidade de Reconstrução

Estado atual: Parcial

  • ✅ Trilha de autenticação completa (login, falhas API key, permissões negadas)
  • ❌ Quem criou/cancelou um job? Não rastreado
  • ❌ Mudanças em configuração de queues? Não rastreado
  • ❌ Quem gerou/revogou uma API key? Não rastreado
  • ❌ Resultados de jobs executados? Não rastreado

Para Full Compliance (GDPR/SOX/ISO 27001):

RequisitoCobertura AtualAção Necessária
Quem acessou dados X? Não Logar entidade afetada em cada operação
Quem modificou configuração? Não Adicionar logs em endpoints de update
Quando ocorreu acesso não autorizado? Sim -
Data retention/retention policy? Não Implementar limpeza automática
Integridade dos logs (tamper-proof)? Parcial Considerar append-only tables ou hash chain
ℹ️
Recomendação Final

Para reconstruir histórico completo, é necessário:

  1. Adicionar logAudit() em todos os endpoints CRUD
  2. Incluir userId em jobService.create() (já existe campo no Job)
  3. Criar logs de transição de status de jobs (PENDING → PROCESSING, etc)
  4. Implementar política de retention com backup/archive
