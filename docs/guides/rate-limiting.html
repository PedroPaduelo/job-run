Rate Limiting

Proteção contra abuso e garantia de disponibilidade

Camadas de Rate Limit

  • Global (Fastify): 1000 req/min por organização (key = organizationId ou ip). Configurado em server.ts.
  • Por Queue (BullMQ):Configurável por fila; usa rateLimit (jobs por minuto) no worker.
  • Por API Key:Camada de rateLimit na API Key (opcional) — se definido, override global?

Implementação Global

await fastify.register(rateLimit, {
  max: 1000,
  timeWindow: '1 minute',
  keyGenerator: (request) => {
    return request.organizationId || request.ip;
  },
});

Se não houver organizationId (ex: health check), usa IP. Aplica a todas as rotas exceto health.

Rate Limit por Queue (Workers)

No worker, cada fila pode ter rateLimit. Exemplo de configuração no código:

const worker = new Worker(queueName, processJob, {
  connection: redis,
  concurrency: queue.concurrency || 10,
  limiter: {
    max: queue.rateLimit || 100,
    duration: 60000,
  },
});

Se queue.rateLimit for null, o worker não limita (só o global).

Considerações

  • Rate limit por organization garante fair usage entre tenants.
  • Limite de 1000 req/min é alto para API interna — pode ser ajustado via environment?
  • BullMQ limiter evita sobrecarregar endpoints externos quando o processamento é rápido.
  • Não há rate limit diferenciado por API Key (pode ser implementado no futuro).
