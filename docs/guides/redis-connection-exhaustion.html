Redis Connection Exhaustion

Vazamento de conexões Redis em alta carga

🔥 O Problema

O sistema usa ioredis para BullMQ. Se connections não são liberadas corretamente, o pool do Redis pode esgotar, causando.timeout e falhas em filas.

💥
Connection Leak

Cada worker abre conexão com Redis. Se worker.close() não for chamado no shutdown, conexões ficam órfãs.

📊 Como Acontece

  • src/lib/redis.ts — singleton redis com maxRetriesPerRequest
  • BullMQ Workers: new Worker(queueName, processJob, { connection: redis })
  • Redis default: DEFAULT_MAX_POOL_SIZE = 10 por instância
  • Se 100 queues × 10 workers = 1000 conexões
  • Redis por padrão: maxclients 10000 — OK, mas...

💥 Impacto

  • BullMQ não consegue enfileirar jobs (EMAXCONNECTIONS)
  • Workers novos não conectam
  • Jobs pendentes nunca processam
  • Cascata de timeouts

🔍 Código Vulnerável

// src/worker.ts — startWorkers() cria múltiplos workers
const worker = new Worker(queueName, processJob, {
  connection: redis,  // ← Todos compartilham a mesma instância?
  concurrency: queue.concurrency || CONCURRENCY,
});

BullMQ reusa a conexão Redis, então connection pool não é por worker. Porém:

  • Se um worker .close(), a conexão pode não ser liberada imediatamente
  • Se setInterval de watch (30s) cria workers novos repeatedly sem close dos antigos

✅ Recomendações

  1. Singleton Redis — Já é, bom. Não criar múltiplas instâncias.
  2. Graceful shutdown — Em shutdown(), await worker.close() executa antes de process.exit()
  3. Monitorar conexões — redis.client.listClients('normal') count
  4. Ajustar pool — redis.options.maxRetriesPerRequest = 20 para lidar com reconnect
  5. Kubernetes liveness — Se Redis cai, workers devem restartar

📈 Diagrama

Redis Server (maxclients 10000)
    ▲
    │ │ │ │  (cada worker usa 1 conexão)
    │ │ │ └── Worker[3]
    │ │ └──── Worker[2]
    │ └────── Worker[1]
    └──────── Worker[0]

Se workers são criados sem close → leak → esgota

🔍 Checklist

  • Redis singleton? ✅
  • Worker.close() no shutdown? ✅ (worker.ts linhas 233-235)
  • Watch interval cria workers duplicados? ⚠️ (linha 205-225: precisa de deduplicação)
  • Monitora conexões Redis? ❌
  • Teste de carga valida conexões? ❌
