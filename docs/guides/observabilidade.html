Health checks, métricas Prometheus, liveness vs readiness, e configuração de probes no Kubernetes.

Liveness vs Readiness: Conceitos Fundamentais

No Kubernetes, health checks são divididos em dois tipos complementares. Entender a diferença é crucial para sistemas resilientes.

Liveness Probe (Sala de Emergência)

Verifica se o processo está vivo. Se falhar, o Kubernetes reinicia o container.

  • Simples: só verifica se o processo responde
  • Não checa dependências externas (DB, Redis)
  • Deve ser ultra-rápido e confiável
  • Protege contra deadlocks e memory leaks
fastify.get('/health', async () => ({
  status: 'ok',
  uptime: process.uptime()
}));
HTTP 200 sempre se rodando

Readiness Probe (Recepção)

Verifica se o serviço está pronto para receber tráfego. Se falhar, o pod é removido do Service.

  • Checa dependências críticas (Redis, DB)
  • Controla entrada de requests no load balancer
  • Permite warm-up antes de receber tráfego
  • Pode retornar 503 temporariamente
fastify.get('/ready', async (_, reply) => {
  const redisOk = (await redis.ping()) === 'PONG';
  const dbOk = await prisma.$queryRaw`SELECT 1`;

  if (!redisOk || !dbOk) {
    return reply.status(503).send({
      status: 'not_ready',
      redis: redisOk,
      database: !!dbOk
    });
  }
  return { status: 'ready', redisOk, dbOk };
});
HTTP 200 ou 503

Configuração no Kubernetes

Configuração ideal para o JobSync considerando o comportamento atual dos endpoints.

Deployment YAML

apiVersion: apps/v1
kind: Deployment
metadata:
  name: job-sync-backend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: job-sync
  template:
    metadata:
      labels:
        app: job-sync
    spec:
      containers:
      - name: backend
        image: jobsync/backend:latest
        ports:
        - containerPort: 3001
        env:
        - name: PORT
          value: "3001"
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: jobsync-secret
              key: database-url
        - name: REDIS_URL
          valueFrom:
            secretKeyRef:
              name: jobsync-secret
              key: redis-url

        # Liveness Probe - verifica se o processo está vivo
        livenessProbe:
          httpGet:
            path: /health
            port: 3001
          initialDelaySeconds: 30    # Tempo para iniciar
          periodSeconds: 10          # Frequência
          timeoutSeconds: 3          # Timeout por request
          failureThreshold: 3        # Reinicia após 3 falhas

        # Readiness Probe - verifica dependências
        readinessProbe:
          httpGet:
            path: /ready
            port: 3001
          initialDelaySeconds: 5     # Mais rápido (já esperou liveness)
          periodSeconds: 5
          timeoutSeconds: 5
          failureThreshold: 3        # Remove do service após 3 falhas

        # Startup Probe - evita race condition no início
        startupProbe:
          httpGet:
            path: /ready
            port: 3001
          initialDelaySeconds: 0
          periodSeconds: 10
          timeoutSeconds: 5
          failureThreshold: 30       # Max 5min (30 × 10s)
          successThreshold: 1
ℹ️
Por que startupProbe?

Evita que o readinessProbe mate o container durante o startup antes do DB/Redis estarem disponíveis. O livenessProbe só ativa após o startupProbe ter sucesso.

Métricas Prometheus

Atualmente, o JobSync não possui endpoint /metrics para integração com Prometheus. Esta seção descreve como implementar.

Instalação

npm install prom-client @fastify/metrics

Código de Exemplo

import fastifyMetrics from '@fastify/metrics';
import { register, Counter, Gauge, Histogram } from 'prom-client';

// Métricas customizadas
const jobsActive = new Gauge({
  name: 'jobsync_jobs_active',
  help: 'Número de jobs ativos na fila'
});

const jobsCompleted = new Counter({
  name: 'jobsync_jobs_completed_total',
  help: 'Total de jobs completados',
  labelNames: ['queue', 'status']
});

// Registrar métricas no Fastify
await fastify.register(fastifyMetrics, {
  routeMetrics: { enabled: true },
  exposeMetrics: true,
  metricsPath: '/metrics'
});

Métricas Recomendadas

Métrica Tipo Descrição Labels

jobsync_jobs_completed_total Counter Jobs completados queue, status

Integração Prometheus

# Prometheus scrape config (prometheus.yml)

  - job_name: 'jobsync-backend'
    scrape_interval: 15s
    static_configs:
      - targets: ['job-sync-backend:3001']

✅
Alertas Recomendados
  • up{job="jobsync-backend"} == 0 → instância down
  • jobsync_jobs_failed_total / rate(jobsync_jobs_completed_total[5m]) > 0.1 → taxa de falha >10%
  • jobsync_queue_latency_seconds{quantile="0.99"} > 300 → latência P99 alta
  • http_request_duration_seconds{path="/api/"} → latency API endpoints

Cache de Health Checks (Performance)

Atualmente, /ready chama redis.ping() e SELECT 1 em cada request. Isso gera carga desnecessária. Cache de 2-5 segundos é suficiente.

Implementação Sugerida

interface HealthCache {
  timestamp: number;
  status: 'ready' | 'not_ready';
  redis: boolean;
  database: boolean;
}

const healthCache: HealthCache = {
  timestamp: 0,
  status: 'not_ready',
  redis: false,
  database: false
};

const CACHE_TTL = 3000; // 3 segundos

// Atualizar cache periodicamente
setInterval(async () => {
  try {
    const start = Date.now();
    const redisOk = await Promise.race([
      redis.ping(),
      new Promise((_, reject) => setTimeout(() => reject(new Error('timeout')), 2000))
    ]) === 'PONG';
    const dbOk = await prisma.$queryRaw`SELECT 1`;
    healthCache.status = redisOk && dbOk ? 'ready' : 'not_ready';
    healthCache.redis = redisOk;
    healthCache.database = !!dbOk;
    healthCache.timestamp = Date.now();
    fastify.log.info({ redisOk, dbOk, duration: Date.now() - start }, 'Health check cached');
  } catch (error) {
    fastify.log.error(error, 'Health check failed');
  }
}, CACHE_TTL);

fastify.get('/ready', async (_, reply) => {
  const age = Date.now() - healthCache.timestamp;
  if (age < CACHE_TTL) {
    return {
      ...healthCache,
      cacheAgeMs: age,
      cached: true
    };
  }

  // Fallback: executar na hora se cache expirou
  const redisOk = healthCache.redis;
  const dbOk = healthCache.database;
  const status = redisOk && dbOk ? 'ready' : 'not_ready';

  if (status === 'not_ready') {
    return reply.status(503).send({
      status,
      redis: redisOk,
      database: dbOk,
      cacheAgeMs: age
    });
  }

  return { status, redis: redisOk, database: dbOk, cacheAgeMs: age };
});

Health por Tenant (Multi-tenant)

Para sistemas multi-tenant, considere health checks isolados por organização. Não implementado atualmente, mas possível adicionar:

Endpoint /ready/:organizationId

GET /ready/org-123
Response: {

  "redis": true,
  "database": true,
  "quotas": {
    "jobs_active": 45,
    "jobs_limit": 100,
    "api_keys_active": 3
  }
}

Use case: Monitorar saúde específica de tenant, útil para SLAs diferenciados.

Dependências Externas: Análise

Dependência Verificado em Health? Endpoint Status
PostgreSQL (Prisma) ✅ Sim (/ready) SELECT 1 Ativo
Redis (ioredis) ✅ Sim (/ready) PING Ativo
Sentry Parcial — Ativo mas não verificado
Email (Resend) No — Não verificado
Auth Service No — Não verificado (integração JWT)
BullMQ Workers No — Apenas conectividade Redis

Checklist de Observabilidade

Health Checks

  • /health - liveness (200 sempre)
  • /ready - readiness (200/503)
  • Skippam auth middleware

Métricas Prometheus

  • Endpoint /metrics
  • Métricas HTTP (request duration)
  • Métricas custom (jobs, queues)

Melhorias Identificadas

  • Cache de health (3s TTL)
  • Timeout nos probes DB/Redis
  • startupProbe no K8s
  • Metricas por tenant

Logs & Tracing

  • Sentry ativo (erros + traces)
  • Fastify logger integrado
  • Audit log via logAudit()
