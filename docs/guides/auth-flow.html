Autenticação e Autorização (JobSync)

Visão técnica completa do fluxo OAuth2 PKCE, JWT, API Key, sincronização, permissões e auditoria.

1. OAuth 2.0 PKCE (Frontend Users)

Neste fluxo, o frontend JobSync usa @nommand/auth-sdk/react para gerir autenticação segura via OAuth 2.0 com PKCE.

Configuração do SDK

const authConfig: AuthSDKConfig = {
  baseUrl: AUTH_API_URL,
  clientId: 'job-sync',
  redirectUri: `${window.location.origin}/callback`,
  scope: 'openid profile email organizations',
  storage: 'localStorage',
  autoRefresh: true,
  onTokenRefresh: () => { ... },
  onAuthError: (error) => { ... },
};

Exportado em src/contexts/AuthContext.tsx.

Login

O método de login chama sdkAuth.authorize() (que faz PKCE + state + redirecionamento para o Auth Service):

login: async () => {
  await sdkAuth.authorize();
};

Callback

Na página /callback, o frontend chama:

await sdkAuth.handleCallback();
await orgContext.refreshOrganizations();

Após callback, o SDK armazena os tokens em localStorage (por exemplo nommand_access_token) e marca o usuário como autenticado.

Auto-refresh de token

A configuração autoRefresh: true faz refresh automático de tokens antes de expirar, reduzindo reconexões manuais.

Fluxo simplificado

Browser (JobSync) Auth Service Callback | | | 1) /login -> authorize() | | |--- PKCE request --> | | | | 2) user login + consent | |--> 3) redirect /callback?code=... |<-------------------- | 4) handleCallback() | | | | 5) token response | | 6) localStorage: nommand_access_token 7) refreshOrganizations() -> list orgs

2. Verificação de JWT (Backend)

O backend JobSync verifica tokens JWT usando JWKS assinados pelo Auth Service.

JWKS e cache

const AUTH_SERVER_URL = process.env.AUTH_SERVER_URL || 'http://localhost:3000';
const JWKS_URL = `${AUTH_SERVER_URL}/.well-known/jwks.json`;
const JWKS_CACHE_TTL = 5 * 60 * 1000; // 5 minutos

O backend usa cache em memória (5 min) para evitar fetchs constantes.

Verificação com jose

const jwks = await getJWKS();
const JWKS = jose.createLocalJWKSet(jwks);
const { payload } = await jose.jwtVerify(token, JWKS, { issuer: validIssuers });
return payload as JWTPayload;

Issuers aceitos

O backend aceita múltiplas variações de emissor (http/https, barra final, localhost, produção) para robustez.

const validIssuers = [
  baseUrl,
  baseUrl + '/',
  baseUrl.replace('https://', 'http://'),
  baseUrl.replace('http://', 'https://'),
  'https://auth.service.api.nommand.com',
  'http://auth.service.api.nommand.com',
  'https://auth.service.nommand.com',
  'http://localhost:3000',
];

Payload esperado

O payload tipado inclui:

  • sub, email, firstName, lastName
  • organizationId / org_id / organization_id
  • roles, permissions
  • sessionId, scope

Fluxo de validação JWT + sync

API request -> onRequest hook |-- Bearer token |-- verifyJWT(token) -> jose.jwtVerify |-- decoded payload (sub, email, roles, permissions, org_id...) |-- obter orgId (header X-Organization-Id, token ou /organizations) |-- buildAuthenticatedUser(token, payload, orgId) |-- request.user, request.permissions

3. Autenticação por API Key

O backend também aceita API Keys no mesmo header Authorization: Bearer <api_key>.

Validação de chave

  • Busca a API key exata em api_keys.key (tabela Prisma APIKey)
  • Verifica isActive
  • Verifica expiresAt
  • Verifica allowlist de IP (allowedIps vazia = todos permitidos)

Atualização auditável

Em caso de sucesso, atualiza:

lastUsedAt = new Date();
usageCount += 1;

Log de falhas

Falhas gravam auditoria com status DENIED e metadata apropriada:

  • inactive
  • expired
  • ip_not_allowed

Fluxo

Request -> Authorization: Bearer jsk_... -> prisma.apiKey.findUnique({key}) -> if not found -> pass para JWT -> if found: - isActive? - expiresAt? - allowedIps? -> update lastUsedAt/usageCount -> request.organizationId, apiKeyId, permissions

4. Pipeline de Sincronização de Usuário

Quando JWT é usado, o backend mantém o estado local sincronizado com o Auth Service.

Passos

  1. verifyJWT (valida assinatura e issuer)
  2. syncUserFromAuth (upsert user na tabela users)
  3. syncOrganizationFromAuth (upsert organização em organizations)
  4. syncMembershipFromAuth (upsert membership em organization_members)
  5. buildAuthenticatedUser (retorna dados de AuthenticatedUser)

Detalhes de sync

syncUserFromAuth prioritiza claims do JWT e usa API /auth/profile apenas se email está ausente.

syncOrganizationFromAuth busca /organizations/:id, ou cria fallback básico se indisponível.

syncMembershipFromAuth resolve role principal (owner/admin/viewer/member) e busca /organizations/:id/members/me para permissões precisas.

verifyJWT(token) -> payload -> orgId -> buildAuthenticatedUser(token,payload,orgId) -> syncUserFromAuth(token,payload) -> syncOrganizationFromAuth(token,orgId) -> syncMembershipFromAuth(token,userId,orgId,roles) -> permissions = payload.permissions || getPermissionsFromRoles(roles) -> return AuthenticatedUser

5. Resolução de Permissões

Permissões são resolvidas tanto pelo payload JWT quanto por derivação de roles, com suporte a curingas.

Mapeamento de roles

const ROLE_PERMISSIONS = {
  owner: ['*'],
  admin: ['jobs:create', 'jobs:read', ...],
  member: ['jobs:create','jobs:read','jobs:retry','jobs:cancel','queues:read'],
  viewer: ['jobs:read','queues:read'],
};

Funções de checagem

hasPermission(userPermissions, requiredPermission)
  -> '*' pass
  -> resource:* pass
  -> exact match pass
hasAnyPermission(...)
hasAllPermissions(...)

Wildcard

Curingas suportados:

  • * = todas as permissões
  • resource:* = todas ações no resource (ex.: jobs:*)

Uso em rotas

O middleware requirePermission(permission) chama hasPermission e, em caso de negação, grava auditoria status: DENIED e retorna 403.

6. Auditoria de Autenticação

Todos os eventos de autenticação são registrados em audit_logs via logAudit().

Campos principais:

  • userId
  • organizationId
  • action (ex.: user.auth.success, apikey.auth.failed, permission.denied)
  • status: SUCCESS, FAILURE, DENIED
  • ipAddress, userAgent, metadata
Auth attempt -> verify JWT / API Key -> if success: logAudit({action:'user.auth.success',status:'SUCCESS'}) -> if failure: logAudit({action:'apikey.auth.failed',status:'DENIED',metadata:{reason}}) -> if permission denied: logAudit({action:'permission.denied',status:'DENIED',metadata:{required,had}})
