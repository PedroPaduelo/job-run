Rate Limit Bypass

Como contornar a proteção de rate limiting

⚡ O Problema

Rate limit atual: keyGenerator: (request) => request.organizationId || request.ip. Isso significa:

  • Com API Key: rate limit por organizationId
  • Com JWT: rate limit por organizationId
  • Sem organization (ex: login): rate limit por IP
🔓
Bypass via IP Rotation

Se attacker usa múltiplos IPs (botnet, Tor), pode fugir do rate limit por IP.

📊 Rate Limit Config

await fastify.register(rateLimit, {
  max: 1000,               // requests por minuto
  timeWindow: '1 minute',
  keyGenerator: (request) => request.organizationId || request.ip,
});

Problemas:

  • organizationId não está disponível no rate limit hook (é adicionado depois no onRequest auth)
  • Se um attacker criar muitas organizações, cada uma tem seu rate limit separado
  • IP pode ser spoofado via X-Forwarded-For se proxy não configurado direito

💥 Vetores de Ataque

  1. IP Rotation — 1000 req/min por IP, usar 100 IPs = 100k req/min
  2. Organization Sprawl — criar múltiplas organizações, cada uma com 1000 req/min
  3. API Key Flood — gerar muitas API keys, cada uma com rate limit próprio? (na implementação atual, rate limit é por organizationId, não por API key)

✅ Recomendações

  1. Rate limit por API Key também — se API key tem rateLimit field, usar: 
    keyGenerator: (req) => req.apiKeyId || req.organizationId || req.ip
  2. IP + Org combo — rate limit por par (IP, organizationId) para evitar organization sprawl
  3. Global rate limit por IP — independente de org, limitar 5000 req/min por IP
  4. BlockKnownBadIPs — integração com AbuseIPDB ou fail2ban
  5. TLS Fingerprinting — detectar clientes mesmo que mudem IP (JA3 fingerprint)
  6. Monitorar anomalias — se uma org dispara de 1k para 100k req/min, alertar

request.organizationId é setado no hook onRequest logo depois, mas rateLimit roda antes? Ordem de registro de plugins importa. Se rateLimit está registrado antes do hook auth, organizationId pode ser undefined sempre.
