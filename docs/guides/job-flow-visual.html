Estados de um Job

Um job pode passar por 7 estados diferentes durante seu ciclo de vida. Cada estado tem seu propósito específico e transições bem definidas.

PENDING
Job criado, aguardando processamento
SCHEDULED
Agendado para execução futura
PROCESSING
Em execução no worker
COMPLETED
Processado com sucesso
FAILED
Falha temporária, pode ser retentado
CANCELLED
Cancelado pelo usuário antes de processar
DEAD
Esgotado número de tentativas

Falha recuperável

Falha definitiva (dead letter)

Transições de Estado

Regras de transição:
  • PENDING → PROCESSING: Worker pega da fila
  • SCHEDULED → PENDING: Timer expira (BullMQ delay)
  • SCHEDULED → CANCELLED: Usuário cancela antes do delay
  • PROCESSING → COMPLETED: Webhook retorna status 2xx
  • PROCESSING → FAILED: Webhook falha (timeout, 4xx/5xx, erro de rede)
  • FAILED → PROCESSING: Retry automático (BullMQ backoff)
  • FAILED/DEAD → PENDING: Retry manual via API
  • PENDING/SCHEDULED → CANCELLED: Usuário cancela
  • FAILED (tentativas esgotadas) → DEAD: Dead letter queue

Propagação de Erros

4.1. Tipos de Erro

Erro Causa Comportamento Tentativas restantes
Timeout AbortController após X ms (padrão: 300s) Retry automático maxAttempts - attemptsMade
HTTP 4xx Webhook retorna erro cliente (400-499) Normalmente retry (exceto 429) maxAttempts - attemptsMade
HTTP 5xx Erro servidor externo Retry automático com backoff maxAttempts - attemptsMade
Network Error DNS falha, conexão recusada Retry automático maxAttempts - attemptsMade
Rate Limit (429) Muitas requisições Retry automático (backoff lento) maxAttempts - attemptsMade

4.2. Lógica de Retry Automático

// Configuração BullMQ no worker.ts const worker = new Worker(queueName, processJob, { connection: redis, concurrency: queue.concurrency || 10, limiter: { max: queue.rateLimit || 100, duration: 60000 }, attempts: queue.maxRetries, // Padrão: 3 backoff: { type: 'exponential', delay: queue.retryDelay // Padrão: 5000ms } }); // worker.on('failed') - Quando esgota tentativas if (job.attemptsMade >= job.opts.attempts) { // Marca como DEAD no banco }
Backoff Exponencial:
  • Tentativa 1: delay de 5000ms (5s)
  • Tentativa 2: delay de 10000ms (10s)
  • Tentativa 3: delay de 20000ms (20s)
  • Tentativa 4: delay de 40000ms (40s)
  • ...
Fórmula: delay × 2^(attemptsMade - 1)

Sistema de Callback

Quando callbackUrl é fornecido na criação do job, o sistema envia notificações POST sobre mudanças de status.

Callback no COMPLETED

POST callbackUrl Content-Type: application/json { "jobId": "job_abc123", "status": "COMPLETED", "result": { ... }, // Resposta do webhook "duration": 1250, // Duracao em ms "attempts": 1 // Numero de tentativas }

Callback no DEAD

POST callbackUrl Content-Type: application/json { "jobId": "job_abc123", "status": "DEAD", "error": "Webhook failed: Request timeout", "attempts": 3, "duration": 300000 }
Importante: Falhas no callback são registradas em log mas não afetam o status do job. O job já está em seu estado final quando o callback é disparado.
