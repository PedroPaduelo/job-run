1. URL e Roteamento

A página de detalhes do job é acessada através de uma rota dinâmica que recebe o ID do job como parâmetro:

GET /jobs/:id Rota dinâmica

Implementação no React Router

// No App.tsx ou routing principal
<Route path="/jobs/:id" element={} />
i
Observação

A rota usa useParams() do React Router para extrair o parâmetro id da URL.

2. Busca de Dados

A página utiliza o hook useJob para buscar os dados do job. A requisição é configurada para refetçar automaticamente.

Hook: useJob(id)
queryKey["job", id]
queryFnjobsApi.get(id)
enabledisAuthenticated && !!id
refetchIntervalREFETCH_INTERVALS.JOB_DETAIL

Intervalo de Refetch

Configurado em src/lib/constants.ts:

export const REFETCH_INTERVALS = {
  JOBS_LIST: 30000,        // 30s para lista
  JOB_DETAIL: 5000,        // 5s para detalhe de job ativo
  QUEUES: 30000,
  QUEUE_STATS: 10000,
  STATS: 30000,
  TIMELINE: 60000,
};
✓
Atualização em Tempo Real

A cada 5 segundos a query é refeita, permitindo acompanhar o progresso do job em tempo real sem necessidade de recarregar a página.

4. Logs de Execução

Os logs são exibidos em uma lista com as tentativas de execução do job. No JobDetail.tsx, apenas os arrays logs vindos do banco de dados são utilizados.

Estrutura de um Log

LogEntry
idstring
attemptnumber
status"COMPLETED"|"FAILED"
messagestring?
durationnumber (ms)
createdAtstring (ISO)

Exibição na UI

Cada log é renderizado como um card com:

  • Badge de tentativa: Mostra "Attempt X" em verde (sucesso) ou vermelho (falha)
  • Mensagem: Texto descritivo do log
  • Duração: Tempo de execução em milissegundos
  • Timestamp: Hora da execução (HH:mm:ss)

Os logs são exibidos em ordem cronológica reversa (mais recente primeiro).

5. Ações Disponíveis

A página disponibiliza botões de ação no cabeçalho, condicionados ao status do job.

✕

Cancelar (Cancel)

Disponível quando o job está em PENDING ou SCHEDULED. Cancela a execução do job antes que inicie.

Ação Destrutiva
↻

Tentar Novamente (Retry)

Disponível quando o job está em FAILED ou DEAD. Cria uma nova tentativa executando o job novamente.

Recuperação

Implementação das Ações

const retryJob = useRetryJob();   // Hook para retry
const cancelJob = useCancelJob(); // Hook para cancel

const handleRetry = async () => {
  if (id) {
    await retryJob.mutateAsync(id);
  }
};

const handleCancel = async () => {
  if (id && confirm("Are you sure you want to cancel this job?")) {
    await cancelJob.mutateAsync(id);
  }
};

Observação sobre Permissões

No frontend atual, não há verificação explícita de permissões. Recomenda-se implementar:

  • Controle de acesso via AuthContext e roles (admin/user)
  • Habilitação/desabilitação condicional dos botões
  • O backend deve validar as permissões em todas as operações
⚠
Importante

Não existe interface de eliminação (delete) de jobs no frontend. A operação deve ser feita apenas via API ou admin tools.

6. Estados de Erro

Quando o job não é encontrado (404) ou ocorre um erro na requisição, a interface mostra mensagens específicas.

Job Não Encontrado (404)

if (!job) {
  return (
    <div className="text-center py-12">
      <p className="text-muted-foreground">
        Job not found
      </p>
      <Link to="/jobs">Back to jobs</Link>
    </div>
  );
}

Navegação: O usuário pode clicar em "Back to jobs" para retornar à lista.

7. Atualizações em Tempo Real

A página monitora ativamente o status do job através do mecanismo de polling do React Query.

↻
Polling Automático

O refetchInterval: 5000 configura uma nova requisição a cada 5 segundos enquanto a página estiver ativa e visível.

Status Progressão

Possíveis transições de status:

PENDING ──┐ │───► PROCESSING ──► COMPLETED │ │ │ ▼ │ FAILED ─► DEAD (maxAttempts) │ │ │ ▼ │ RETRY ──┐ │ │ SCHEDULED ────────────────────┘ │ ▼ CANCELLED (cancel action)

Enquanto um job estiver em processamento ou status ativo, a barra de status (Badge) atualizará automaticamente a cada 5 segundos.

9. Permissões do Frontend

No código atual, não há controle explícito de permissões. Recomenda-se a seguinte estrutura:

Permissão Controle Ações Afetadas
jobs:view Renderização da página Bloquear se não autorizado
jobs:cancel Botão Cancelar Ocultar/desabilitar
jobs:retry Botão Retry Ocultar/desabilitar

Implementação Sugerida

// AuthContext ou hook de permissões
const { can } = usePermissions();

// No render
{can('jobs:cancel') && job.status === 'PENDING' && (
  <Button onClick={handleCancel}>Cancel</Button>
)}
!
Validação Obrigatória no Backend

O frontend pode controlar a UI, mas o backend sempre deve validar permissões antes de executar cancel/retry.

12. Considerações de UX

⟳

Auto-refresh

O polling de 5s evita que o usuário precise recarregar para ver atualizações de status.

⏱

Timestamps Locais

Datas formatadas com date-fns para legibilidade: format() e formatDistanceToNow().

!

Confirmação

Cancel usa confirm() nativo para evitar ações acidentais.

⌨

Monospace para IDs

IDs técnicos usam fonte monoespaçada para facilitar leitura e cópia.
