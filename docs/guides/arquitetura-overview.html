Visão Geral

O JobSync é uma plataforma de processamento assíncrono de jobs construída com uma arquitetura moderna, escalável e multi-tenant. O sistema é dividido em duas aplicações principais:

Camadas da Aplicação

Frontend (React): Interface de usuário com rotas protegidas, useQuery/useMutation para dados, componentes reutilizáveis.

Backend API (Fastify): REST API com autenticação via JWT ou API Keys, rate limiting, CORS, validação Zod.

Serviços (Service Layer): Lógica de negócio isolada das rotas. Cada domínio tem seu service.

Repositório (Prisma): Acesso a dados via ORM. Schema define relacionamentos e constraints.

Workers (BullMQ): Processam jobs de forma assíncrona, com retry automático e timeout.

Multi-Tenancy

O sistema é multi-tenant por organização. Cada organização (tenant) tem seus dados isolados no banco e no Redis.

Identificação do Tenant

  • Header X-Organization-Id: Enviado em todas as requisições API
  • API Keys: Vinculadas a uma organização específica
  • JWT: Pode conter organizationId no payload, ou o sistema busca a primeira organização do usuário

Isolamento de Dados

Todas as queries do Prisma filtram por organizationId. O hook onRequest do Fastify valida e popula request.organizationId.

ℹ️
Isolamento Forte

O isolation é garantido em código — toda query deve filtrar por organizationId. Nunca confie apenas em constraints do banco.

Namespaces no Redis

Filas BullMQ são namespaced por organização: jobs-{orgId}-{queueName}. Isso garante que filas de organizações diferentes não colidam.

Sistema de Permissões (RBAC)

Baseado em papéis (roles) e permissões granulares. Implementado em src/lib/auth.ts.

Roles

RoleDescriçãoPermissões
OWNERDon da organizaçãoTodas (*)
ADMINAdministradorJobs: create/read/update/delete/retry/cancel, Queues:全部, ApiKeys: create/read/delete
MEMBERMembro regularJobs: create/read/retry/cancel, Queues: read
VIEWERSomente leituraJobs: read, Queues: read

Permissões por Recurso

  • jobs:create — Criar novos jobs
  • jobs:read — Ver jobs e seus logs
  • jobs:update — Atualizar jobs (raro)
  • jobs:delete — Deletar jobs
  • jobs:retry — Retriar jobs falhos
  • jobs:cancel — Cancelar jobs pendentes
  • queues:create — Criar filas
  • queues:read — Ver filas e estatísticas
  • queues:update — Modificar configurações da fila
  • queues:delete — Remover filas
  • queues:pause — Pausar/resumir filas
  • apikeys:create — Criar API Keys
  • apikeys:read — Listar API Keys
  • apikeys:delete — Deletar API Keys
  • admin:* — todas as permissões (wildcard)
  • org:settings — Configurações da organização

Verificação de Permissões

// Middleware de permissão (backend)
fastify.addHook('onRequest', async (request, reply) => {
  const permission = 'jobs:create';
  if (!hasPermission(request.permissions, permission)) {
    await logAudit({
      organizationId: request.organizationId,
      action: 'permission.denied',
      metadata: { required: permission },
      status: 'DENIED',
    });
    return reply.status(403).send({ error: 'Forbidden' });
  }
});

No frontend, use usePermissions da SDK ou o hook customizado para mostrar/esconder UI.

API Keys (Machine-to-Machine)

Para integrações server-to-server, use API Keys. Elas são:

  • Geradas como jsk_ + 32 bytes aleatórios hex
  • Vinculadas a uma organização
  • Possuem permissões específicas
  • Podem ter allowed_domains, allowed_ips, rate_limit
  • Podem ter data de expiração
  • Usadas como Bearer token na Authorization header
// Exemplo de autenticação com API Key
Authorization: Bearer jsk_abc123def456...

// Backend:
const apiKey = await prisma.apiKey.findUnique({ where: { key: token } });
if (apiKey && apiKey.isActive && !apiKey.expiresAt) {
  request.organizationId = apiKey.organizationId;
  request.permissions = apiKey.permissions;
  // Não há userId associado (machine)
}
⚠️
Importante

Apenas o primeiro uso da API Key mostra o valor completo. Após isso, guarde em local seguro — não é possível recuperá-la.

Segurança em Camadas

  • HTTPS Only: All traffic encrypted in transit
  • CORS Whitelist: Only allowed origins can call the API
  • Rate Limiting: 1000 req/min por organização, configurável por API Key
  • Input Validation: Zod schemas em todas as rotas
  • SQL Injection: Prevented by Prisma parameterized queries
  • JWT Verification: Via JWKS, com cache de 5 min
  • Sentry Monitoring: Captura exceções e contexto de segurança
  • Audit Log: Todas as ações críticas são logadas
