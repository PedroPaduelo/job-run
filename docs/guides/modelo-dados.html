Propósito da Entidade User

Representa um usuário no sistema. Os dados são sincronizados do Auth Service (provedor de identidade externo). O JobSync não gerencia senhas — apenas consome as informações de perfil e status.

Enums: UserStatus = ACTIVE | INACTIVE | SUSPENDED — define o estado da conta e habilidade de acessar o sistema.

Indexes: email (único, usado para login).

Propósito da Entidade Organization

Entidade tenant do sistema multi-tenant. Cada organização representa um cliente/empresa com seu próprio espaço isolado. Contém configurações globais (settings como JSON) e o slug é usado em URLs e referências.

Indexes: slug (único).

Enums implícitos: Nenhum — usa booleano isActive para controle.

Entidade de junção que liga User a Organization. Armazena papel (role) e permissões granulares específicas para aquele usuário naquela organização. Permissões podem ser herdadas do papel OU sobrescritas individualmente.

Enums: MemberRole = OWNER (tudo), ADMIN (gestão), MEMBER (usuário), VIEWER (só leitura).

Propósito da Entidade ApiKey

Armazena chaves de API para autenticação machine-to-machine. As chaves são geradas no formato jsk_ + 32 bytes hexadecimais. Suportam rate limit customizado, domínios/IPs autorizados e expiração.

Midias de segurança: allowedDomains (CORS por key), allowedIps (whitelist IP), rateLimit (req/h ou req/min), isActive controle, expiresAt auto-revogação.

Indexes: key (único, usado para lookup rápido de autenticação).

Propósito da Entidade Queue

Define configurações de processamento para uma fila de jobs. Configurações incluem concorrência (workers simultâneos), retry delay, max retries, timeout e rate limiting.

Configurações padrão: concurrency = 5, maxRetries = 3, retryDelay = 5000ms, timeout = 300000ms (5min).

Relacionamento: 1:N com Job — cada fila pode ter muitos jobs.

Namespaces Redis: BullMQ usa jobs-{orgId}-{queueName} como nome da fila Redis.

Propósito da Entidade Job

Entidade central do sistema. Representa uma unidade de trabalho assíncrono. Pode ser criada via API (POST /jobs) e processada por workers BullMQ.

Campos principais:

  • status — ciclo de vida do job (ver enum abaixo)
  • priority — prioridade (maior = mais rápido). Default 0.
  • attempts — número de tentativas até o momento
  • webhookUrl — URL para POST quando o job completar
  • payload — dados de entrada (JSON arbitrário)
  • tags — para filtros e queries de métricas

Enums: JobStatus = PENDING, SCHEDULED, PROCESSING, COMPLETED, FAILED, CANCELLED, DEAD (excedeu maxAttempts).

Propósito da Entidade JobLog

Armazena logs de execução de cada tentativa de job. Cada vez que um worker processa um job (inclusive retries), um log é criado com status (success/failure), duração, e mensagem de erro se houver.

Metadados: Pode armazenar contexto adicional (workerId, memUsage, etc). O campo status é livre (não enumerado) para flexibilidade.

Propósito da Entidade AuditLog

Registra todas as ações importantes para compliance e troubleshooting. Captura quem fez o quê, quando, de onde (IP) e com qual resultado.

Exemplos de ação: job.create, queue.delete, apikey.revoke, permission.denied, user.login.

Enums: AuditStatus = SUCCESS | FAILURE | DENIED.

Indexes: organizationId, userId, entityType + entityId (compound), createdAt.

Relacionamentos Detalhados

User ↔ OrganizationMember ↔ Organization

Tipo: Many-to-Many (via junção)

Como: OrganizationMember é a tabela de associação que conecta User e Organization. Armazena role e permissions específicos por par usuário-organização.

Chave: @@unique([userId, organizationId]) garante que cada usuário só tenha uma membership por organização.

Organization → Queue → Job

Tipo: One-to-Many em cascata

Como: Organization tem muitas queues. Cada queue pertence a uma organization. Queue tem muitos jobs. Job pertence a uma queue.

Deleção: Ao deletar Organization, filas e jobs são removidos (cascade no Prisma). Ao deletar Queue, jobs são removidos.

User → Job (criação)

Tipo: One-to-Many (opcional)

Como: User pode criar muitos jobs. userId em Job é opcional para jobs criados via API Key (machine).

Job → JobLog (histórico)

Tipo: One-to-Many

Como: Cada tentativa de processamento de job gera um ou mais JobLog entries. JobLog armazena o histórico completo de execução.

Organization → AuditLog

Tipo: One-to-Many

Como: Toda ação auditável gera um AuditLog ligado à organization (e opcionalmente ao userId que realizou).

Organization → ApiKey

Tipo: One-to-Many

Como: Cada organização pode ter múltiplas API Keys para autenticação M2M. A chave key é única globalmente.

Dados de entrada para o job. Schema aberto — o worker determina a estrutura esperada:

{
  "action": "send-email",
  "to": "user@example.com",
  "subject": "Welcome!",
  "templateId": "welcome-v1"

Resultado do processamento (após COMPLETED):

{
  "success": true,
  "emailId": "msg_abc123",
  "sentAt": "2026-03-14T10:30:00Z"
}

Metadatos não funcionais (tracking, debugging):

{
  "source": "webhook-stripe",
  "correlationId": "req_xyz789",
  "priorityReason": "urgent"
}

Contexto da execução (debugging avançado):

{
  "workerId": "worker-3",
  "memoryUsage": 45000000,
  "queueTime": 1204,
  "processTime": 340
}

Detalhes específicos da ação auditada:

{
  "ip": "192.168.1.100",

  "resourceId": "job_abc123",

Considerações de Performance

Indexação Estratégica

Todo campo usado em WHERE, ORDER BY ou JOIN deve ter índice. Os indexes definidos cobrem os principais padrões de query:

  • Jobs por organização e fila (dashboard)
  • Jobs por status (filtros)
  • Jobs agendados (scheduledAt)
  • Audit logs por data (logging)

JSON para Dados Flexíveis

Campos Json permitem schema-less:

  • Payload do job é arbitrário
  • Settings por organização podem evoluir sem migrations
  • Audit metadata captura contexto variável

Cuidado: JSON não é indexável (exceto GIN no Postgres). Não armazene dados que precise consultar frequentemente.

Namespaces no Redis

BullMQ usa Redis para filas. O sistema namespaced por organização evita colisão:

BullMQ queue name = `jobs-${orgId}-${queue.name}`

Isso permite múltiplas organizações no mesmo cluster Redis com isolamento.

Partição por Tenant

Todos os modelos contêm organizationId. Índices começam por esta chave, garantindo que queries sejam tenant-scoped. Para escala extrema, considere particionamento físico por tenant no Postgres.

Migrations e Evolução

Adicionando um Campo

$ npx prisma migrate dev --name add_field_to_job
$ npx prisma generate

O Prisma gera SQL que preserva dados existentes (campo NULL ou com valor default).

Alterando um Enum

$ npx prisma migrate dev --name add_status_to_job
// Editar schema: enum JobStatus { PENDING, PROCESSING, ..., ARCHIVED }
// Postgres: ALTER TYPE adiciona valor (pode precisar em produção)

Rollback

$ npx prisma migrate resolve --rolled-back <migration-name>

Sempre teste rollback em staging antes de produção.
