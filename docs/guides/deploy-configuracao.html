Etapas de Build

Backend

git clone <repo>
cd job-run
npm ci --only=production

# Build TypeScript
npm run build
# Output: dist/

Frontend

# Na raiz do projeto
npm run build
# Executa tsc -b e depois vite build
# Output: dist/ (arquivos estáticos)

npm run start
# Escuta em PORT (padrão: 3001)
# Frontend (preview local)
npm run preview
# Servidor Vite em porta 4173 (ou configurada)
# Frontend em produção (nginx)
# Servir arquivos de dist/ como estáticos

Deploy com PM2

O PM2 é crítico para gerenciamento de processos em produção. Use-o para manter a aplicação rodando com auto-restart e logs centralizados.

Comandos Básicos

# Instalar PM2 globalmente
npm install -g pm2

# Iniciar backend
pm2 start npm --name "job-sync-backend" -- run start

# Iniciar frontend (servir estáticos)
pm2 start npx --name "job-sync-frontend" -- serve dist/ -p 8080

# Iniciar workers separadamente
pm2 start npm --name "job-sync-worker-1" -- run start:worker
pm2 start npm --name "job-sync-worker-2" -- run start:worker

# Salvar configuração
pm2 save

# Configurar startup no sistema
pm2 startup
# Siga as instruções geradas

Configuração de Runtime

Backend (API)

  • PORT — Porta de escuta (padrão: 3001)
  • NODE_ENV — environment: production / development
  • DATABASE_URL — Connection string PostgreSQL
  • REDIS_URL — Connection string Redis
  • JWT_SECRET — Chave para assinar tokens
  • ALLOWED_ORIGINS — CORS allowlist
  • SENTRY_DSN — Sentry configuration
  • AUTH_SERVICE_URL — URL do serviço de autenticação

Frontend

  • VITE_API_URL — URL do backend API
  • VITE_AUTH_URL — URL do serviço de autenticação
  • VITE_APP_NAME — Nome da aplicação (opcional)

Health Checks

Endpoints para monitoramento de saúde do sistema:

/health

Retorna status geral da aplicação, incluindo conexões com banco e Redis.

GET /health
Response: { "status": "healthy", "timestamp": "..." }

/ready

Indica se a aplicação está pronta para receber tráfego. Usado em Kubernetes readinessProbe.

GET /ready
Response: { "ready": true, "timestamp": "..." }

/api/stats

Métricas internas: jobs ativos, workers, filas, database.

GET /api/stats
Response: { "jobs": {...}, "queues": {...} }

Configuração em Orquestração

# Kubernetes example
readinessProbe:
  httpGet:
    path: /ready
    port: 3001
  initialDelaySeconds: 10
  periodSeconds: 5

livenessProbe:
  httpGet:
    path: /health
    port: 3001
  initialDelaySeconds: 30
  periodSeconds: 10

Escalabilidade

Os workers podem ser executados em processos separados e até em máquinas diferentes, compartilhando o mesmo Redis + PostgreSQL:

Workers em Processos Separados

# Cada worker como instância independente
pm2 start npm --name "worker-1" -- run start:worker
pm2 start npm --name "worker-2" -- run start:worker
pm2 start npm --name "worker-3" -- run start:worker
pm2 start npm --name "worker-4" -- run start:worker

Use --instances no PM2 ecosystem para múltiplas instâncias do mesmo worker.

Scaling Horizontal

Para escalar em múltiplas máquinas:

  • Configure todos os workers para apontar ao mesmo Redis + PostgreSQL
  • BullMQ gerencia distribuição de jobs automaticamente
  • Use lock Durability no Redis para evitar duplicação

Dockerização

Multi-stage Build (Backend)

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine AS runtime
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package*.json ./
RUN npm ci --only=production
EXPOSE 3001
CMD ["node", "dist/index.js"]

Multi-stage Build (Frontend)

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine AS runtime
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/nginx.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

docker-compose.yml (Completo)

version: '3.8'

services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: jobsync
      POSTGRES_USER: jobsync
      POSTGRES_PASSWORD: strong-password
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U jobsync"]
      interval: 10s

  redis:
    image: redis:7-alpine
    command: redis-server --requirepass $$REDIS_PASSWORD$$ --appendonly yes
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s

  backend:
    build: .
    command: npm run start
    environment:
      NODE_ENV: production
      PORT: 3001
      DATABASE_URL: postgres://jobsync:strong-password@postgres:5432/jobsync
      REDIS_URL: redis://:$$REDIS_PASSWORD$$@redis:6379
      JWT_SECRET: $$JWT_SECRET$$
      AUTH_SERVICE_URL: https://auth.yourapp.com
    ports:
      - "3001:3001"
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy

  frontend:
    build: .
    command: npx serve dist/ -p 8080
    environment:
      VITE_API_URL: http://localhost:3001
    ports:
      - "8080:8080"
    depends_on:
      - backend

  worker:
    build: .
    command: npm run start:worker
    environment:
      NODE_ENV: production
      DATABASE_URL: postgres://jobsync:strong-password@postgres:5432/jobsync
      REDIS_URL: redis://:$$REDIS_PASSWORD$$@redis:6379
    deploy:
      replicas: 4
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy

  postgres_data:
  redis_data:

All-in-One Container

Para deployments simples, todos os componentes podem rodar no mesmo container via PM2:

# Dockerfile simples
FROM node:20-alpine

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Expor portas
EXPOSE 3001 4173

# Usar PM2 para gerenciar todos os processos
CMD ["pm2-runtime", "ecosystem.config.js"]

Produção — Hardening

CORS

Restrinja origens explicitamente:

ALLOWED_ORIGINS=https://app.yourapp.com,https://admin.yourapp.com

Monitoramento (Sentry)

SENTRY_DSN=https://xxxxxx@sentry.io/xxxxx
# Sentry captura erros automaticamente em produção

Redis Segurança

  • Use senha forte (requirepass)
  • Habilite persistência AOF (appendonly yes)
  • Configure bind apenas a interfaces internas
  • Evite expor Redis diretamente à internet

PostgreSQL

  • SSL recomendado para conexões externas
  • Backup automático (WAL archiving ou pg_dump)
  • Índices apropriados para consultas frequentes
  • Connection pooling (PgBouncer ou pool no app)

Node.js

  • Use NODE_ENV=production (desabilita dev warnings)
  • Configure ulimit para número de arquivos abertos
  • Habilite Core Dump para debug de crash

Sentry

Rollback

Procedimento para reverter deployments problemáticos:

Passos

  1. Identifique o commit anterior estável: git log --oneline
  2. Reverta: git revert <commit-hash> ou git checkout <stable-commit>
  3. Rebuild: npm run build
  4. Reinicie com PM2: 
    pm2 restart job-sync-backend
pm2 restart job-sync-frontend
pm2 restart all
  5. Verifique logs: pm2 logs

Recuperação Rápida sem Git

Se o build anterior ainda está no diretório:

# Parar serviços
pm2 stop all

# Restaurar dist anterior (se backupado)
cp -r dist.backup/ dist/

# Reiniciar
pm2 start all
