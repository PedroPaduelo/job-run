BullMQ Queue Tuning

Concorrência, backoff, prioridades e limitação de taxa

⚙️ Configuração Atual

// worker.ts (linhas 155-162)
const worker = new Worker(queueName, processJob, {
  connection: redis,
  concurrency: queue.concurrency || CONCURRENCY,  // default 10
  limiter: {
    max: queue.rateLimit || 100,  // jobs por minuto
    duration: 60000,
  },
});

Job options (ao adicionar job na fila):

await bullQueue.add('process', jobData, {
  priority: data.priority || 0,
  attempts: queue.maxRetries,      // default 3
  backoff: {
    type: 'exponential',
    delay: queue.retryDelay,       // default 5000ms
  },
  delay: scheduledAt ? ... : undefined,
});

🎯 Parâmetros de Tuning

ParâmetroDefaultImpacto
concurrency10+ workers paralelos → + throughput, + memória
limiter.max100/minRate limit por fila → protege downstream
attempts3+ retries → + latency, menos órfãos
backoff.delay5000msEspera entre retries → alivia serviço com falha
priority0Zero = mais alto, 100 = mais baixo
timeout300000msBullMQ abort job se passar (não afeta webhook timeout)

💥 Problemas de Configuração

  • concurrency muito alto → saturação da API de destino (webhook) e Redis overload
  • rate limit baixo → fila acumula, jobs demoram
  • backoff delay curto → retry spam em serviço caído
  • sem priorização → jobs importantes atrasam

✅ Melhores Práticas

  1. Baseline concurrency por queue:
    • Webhook externo: 5-10 (não sobrecarregar)
    • Processamento interno (DB-only): 20-50
  2. Rate limit alinhado ao destino — se API alvo permite 1000 req/min, setar queue.rateLimit = 900
  3. Backoff exponencial com jitter — BullMQ já faz exponencial, adicionar jitter manual se necessário
  4. DLQ (Dead Letter Queue) — após maxAttempts, mover para fila especial para inspeção manual
  5. Prioridades documentadas — definir: 0-30 = HIGH, 31-70 = NORMAL, 71-100 = LOW
  6. Timeouts adequados — queue.timeout deve ser maior que webhook timeout, menor que BullMQ job TTL

📈 Exemplo de Config por Tipo

Queue "email":
  concurrency: 5 (SMTP bursting issues)
  rateLimit: 100/min (SendGrid quota)
  retryDelay: 60000 (1min, email transient errors)
  timeout: 120000 (2min)

Queue "payment-webhook":
  concurrency: 2 (3rd party API rate limit)
  rateLimit: 30/min
  retryDelay: 300000 (5min, payment may be temporary)
  timeout: 300000 (5min)

Queue "internal-process":
  concurrency: 20 (no external deps)
  rateLimit: null (unlimited)
  retryDelay: 5000
  timeout: 60000 (1min)
