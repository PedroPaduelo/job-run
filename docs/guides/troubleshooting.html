Troubleshooting

Resolução de problemas comuns no JobSync

1. Worker não está processando jobs

  • Verifique se o processo worker está rodando: pm2 list ou ps aux | grep worker
  • Verifique logs: pm2 logs job-assync-backend ou worker logs específicos
  • Confirme que a queue está active e not paused no DB
  • Verifique Redis: redis-cli ping deve retornar PONG
  • Verifique conectividade com webhookUrl (firewall, TLS, auth)

2. Jobs ficam em PROCESSING e nunca completam

  • Timeout muito baixo? Aumente timeout na queue (default 300000ms)
  • Webhook URL lento ou não responde — verifique logs do worker para erros HTTP
  • Verifique se callbackUrl é acessível (se definido)
  • Se callback falha, o worker ainda marca como COMPLETED (callback falha não reverte status)

3. API retorna 401 unauthorized

  • JWT expirado? Verifique token no localStorage, chame refresh ou refaça login
  • API Key correta? Verifique se está ativa, não expirada, IP em allowlist (se definido)
  • Header X-Organization-Id está presente? O backend pode tentar buscar org do JWT, mas é melhor enviar explicitamente.

4. Rate limit errors (429)

  • Limite global: 1000 req/min por organização
  • Limite de API Key (se configurado) também conta
  • Verifique se algum cliente está fazendo polling muito agressivo
  • Considere aumentar limites em configuração se necessário

5. Jobs são criados mas não aparecem na lista

  • Query usa filtros? Remova filtros e tente novamente
  • Cache do TanStack Query pode estar desatualizado — force refresh
  • Verifique se o job realmente foi criado no banco: SELECT * FROM jobs WHERE organization_id = ?

6. Redis connection errors

  • Redis está rodando? systemctl status redis ou redis-cli ping
  • REDIS_URL correta? Default: redis://localhost:6379
  • Firewall bloqueando? Porta 6379 acessível

7. Erro 500 interno

  • Verifique logs do Sentry (se configurado)
  • Verifique logs do PM2: pm2 logs
  • Cheque se PostgreSQL está acessível: DATABASE_URL correta, credenciais válidas
  • Verifique se as migrations (prisma db push) estão aplicadas

8. CORS errors no frontend

  • ALLOWED_ORIGINS deve conter a URL do frontend (ex: http://localhost:5173)
  • Se em produção, adicione o domínio público (ex: https://app.ddw1sl.easypanel.host)
  • Desenvolvimento ignora CORS (NODE_ENV=development) — mas em produção é estrito

9. Callback não é chamado

  • Callback URL deve ser acessível a partir do worker (firewall/TLS)
  • Callback é fire-and-forget; se falhar, o erro é logado mas não reverte o job status
  • Verifique logs do worker para mensagens de callback failure
