Database Indexes

Índices PostgreSQL para performance em alta carga

📈 O Problema

Sem índices adequados, queries de listagem e contagem de jobs/queues ficam lentas em datasets grandes (milhões de rows).

🗃️
Schema Prisma Indexes

O schema já define @@index em colunas críticas. Precisamos validar se são suficientes para padrões de acesso reais.

🔍 Índices Existentes (schema.prisma)

  • User: @@index([email])
  • Organization: @@index([slug])
  • OrganizationMember: @@index([userId]), @@index([organizationId])
  • ApiKey: @@index([key]), @@index([organizationId])
  • Queue: @@unique([organizationId, name]), @@index([organizationId])
  • Job: @@index([organizationId]), @@index([queueId]), @@index([userId]), @@index([status]), @@index([scheduledAt]), @@index([createdAt]), @@index([externalId])
  • JobLog: @@index([jobId]), @@index([createdAt])
  • AuditLog: @@index([userId]), @@index([organizationId]), @@index([action]), @@index([entityType, entityId]), @@index([createdAt])

📊 Queries Mais Comuns (Análise)

// JobService.list (usage: frontend job listing)
where: { organizationId, status?, queueId?, userId?, createdAt range? }
orderBy: createdAt DESC
limit: 20, offset: 0

// StatsService.getOverview
prisma.job.count({ where: { organizationId } })                    // usa orgId idx
prisma.job.count({ where: { organizationId, createdAt >= } })    // usa orgId idx + filter
prisma.job.findMany({ where: { organizationId, status='COMPLETED', completedAt >= }, select: startedAt, completedAt })
→ usa orgId idx + status idx? Composite index melhor

// QueueService.getStats
prisma.job.groupBy({ by: ['status'], where: { queueId } })       → usa queueId idx

⚠️ Gaps Identificados

  • Composite index (organizationId, status) — usado em stats de completed/failed por org
  • Composite index (organizationId, createdAt) — usado em list com data range
  • Covering index para count queries — se count só precisa de id, usar INDEX ONLY SCAN
  • JobLog (jobId, createdAt) — já tem individual, mas composite pode ajudar

✅ Recomendações

  1. Adicionar composite indexes no Prisma: 
    @@index([organizationId, status])
@@index([organizationId, createdAt])
@@index([queueId, status])
  2. EXPLAIN ANALYZE — rodar nas queries críticas para confirmar seq scan
  3. pg_stat_statements — habilitar para identificar slow queries
  4. REINDEX CONCURRENTLY — criar índices sem lock write
  5. VACUUM ANALYZE — schedule automático (PGAutoCacle?)
  6. Partitioning por mês — se jobs > 10M, particionar por createdAt

📈 Exemplo de Composite

Query: SELECT count(*) FROM jobs WHERE organizationId = 'org1' AND status = 'COMPLETED'

SEM composite:
  → Scan idx_organizationId → filter status (muitas rows)

COM (organizationId, status):
  → Index Only Scan (muito mais rápido)

🔍 Checklist

  • Todas queries usam índices? ⚠️ (analisar EXPLAIN)
  • Composite indexes para padrões comuns? ❌
  • pg_stat_statements ativo? ❌
  • Partição planejada? ❌
  • VACUUM schedule? ❌
