Database Deadlocks

Concorrência alta pode travar transações no PostgreSQL

🔒 O Problema

PostgreSQL detecta ciclos de locks e aborta uma transação. Em alta concorrência de workers atualizando jobs, deadlocks podem ocorrer.

🔄
Deadlock Detectado

Quando dois workers tentam UPDATE/INSERT na mesma row em ordem diferente, PostgreSQL detecta ciclo e mata uma transação.

📊 Padrões de Acesso Concorrente

OperaçãoTabelaLock
prisma.job.create()jobsROW EXCLUSIVE
prisma.job.update()jobsROW EXCLUSIVE
prisma.jobLog.create()job_logsROW EXCLUSIVE
prisma.queue.update()queuesROW EXCLUSIVE

💥 Como Ocorre

  1. Worker A: UPDATE job SET status='PROCESSING' WHERE id='job1'
  2. Worker B: UPDATE job SET status='PROCESSING' WHERE id='job2'
  3. Worker A: INSERT INTO job_logs (jobId, ...) VALUES ('job1', ...)
  4. Worker B: INSERT INTO job_logs (jobId, ...) VALUES ('job2', ...)
  5. Se as transações pegarem locks em ordem diferente (job A lock, depois log A vs job B lock, depois log B), ciclo pode ser formado

📈 Exemplo Real

Worker 1:              Worker 2:
  BEGIN                 BEGIN
  SELECT job1 FOR UPDATE   SELECT job2 FOR UPDATE
  UPDATE job1              UPDATE job2
  INSERT log1               INSERT log2
  COMMIT                   COMMIT (DEADLOCK se ORDER diferente)

🛡️ Prevenção no Código

  • Schema Prisma tem @@index([queueId]), @@index([organizationId]) — OK
  • Transações são implícitas (cada query é auto-commit) — sem transação multi-step
  • prisma.$transaction() não é usado — bom, evita long-running transactions

✅ Recomendações

  1. Retry automático — BullMQ já retry em 3 tentativas (exponential backoff)
  2. Lock por organizationId — Se jobs da mesma org são seriais, usar advisory lock
  3. MONITORAR — Log de deadlocks no PostgreSQL: SELECT * FROM pg_stat_database WHERE deadlocks > 0
  4. Índices — Garantir que todos WHERE têm índice (já tem: id, queueId, organizationId, status, scheduledAt)
  5. Evitar UPDATEs em batch — JobService.update usa updateMany, mas com where por id + orgId — OK

🔍 Checklist

  • Deadlock retry配置? ✅ (BullMQ attempts + backoff)
  • Transações curtas? ✅ (auto-commit)
  • Índices suficientes? ✅
  • Monitoramento ativo? ❌
  • Teste de carga simula concorrência? ⚠️
