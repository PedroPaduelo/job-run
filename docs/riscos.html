🔥 Riscos Identificados

Auth Service — SPoF

Serviço externo de autenticação é ponto único de falha

CRÍTICO
Probabilidade: 4
Impacto: 5
Score: 20
Status: Parcial

Se o Auth Service (nommand.com) ficar indisponível, novos usuários não conseguem fazer login. Usuários existentes com JWT válido ainda podem usar (JWT cacheado).

Mitigação atual: JWT cacheado localmente (5min no Redis), syncUserFromAuth tem fallback para criar usuário básico.
Recomendação: Implementar login offline com refresh token ou ter fallback de emergência para demo mode.

Redis — SPoF Total

Redis é crítico para filas, rate limiting e scheduler

CRÍTICO
Probabilidade: 3
Impacto: 5
Score: 15
Status: Parcial

Se Redis cair: workers param, jobs não são processados, rate limiting falha-open, scheduler não funciona.

Mitigação atual: Rate limiting falha-open (allow all) para não indisponibilizar. Mas filas param.
Recomendação: Implementar Redis Sentinel ou Cluster. Monitoramento proativo. Queue dead-letter no DB como fallback.

SSRF via Webhook

Usuário malicioso pode configurar webhook para IP interno

ALTO
Probabilidade: 2
Impacto: 5
Score: 10
Status: Mitigado

Bloqueio existe mas pode haver bypass.

Mitigação: BLOCKED_HOSTS em types/index.ts bloqueia 127.x, 10.x, 172.16-31.x, 192.168.x, etc. HTTPS obrigatório em produção.

Bulk Jobs — DoS

Cliente pode enviar 1000 jobs de uma vez, sobrecarregando sistema

ALTO
Probabilidade: 3
Impacto: 4
Score: 12
Status: Parcial
Mitigação atual: Rate limiting por org (1000/min), batches de 1000 no bulk.
Recomendação: Limitar bulk a 100/jobs, adicionar delay entre batches, fila de processamento prioritária.

Callbacks Falham Silenciosamente

Callback timeout 5s, falha ignorada — cliente não sabe

MÉDIO
Probabilidade: 4
Impacto: 3
Score: 12
Status: Não Mitigado
Problema: callbackError é logado mas não notifica cliente. Callback failures não são visíveis na UI.
Recomendação: Criar tabela de callback_attempts, retry assíncrono, UI mostra status de callback.

API Keys em Texto

API Keys armazenadas sem hash — vazamento do DB expõe todas as chaves

MÉDIO
Probabilidade: 2
Impacto: 4
Score: 8
Status: Não Mitigado
Problema: Table api_keys guarda key em texto. Se DB vazar, todas as chaves são expostas.
Recomendação: Hash da API Key (tipo bcrypt ou SHA256 truncado). Cliente armazena hash, não a chave original.

Dados Crescem Indefidamente

Jobs, Logs e AuditLogs nunca são deletados — disco pode encher

MÉDIO
Probabilidade: 4
Impacto: 3
Score: 12
Status: Não Mitigado
Problema: Sem política de retenção. Jobs e logs ficam para sempre. A cada 1M jobs = ~GB de dados.
Recomendação: Implementar TTL automático: jobs completadas > 90 dias → archive + delete. audit_logs > 1 ano → aggregate + delete.

Jobs Perdidos no Crash

Se worker morre durante processamento, job pode ficar órfão

MÉDIO
Probabilidade: 2
Impacto: 4
Score: 8
Status: Parcial
Situação: BullMQ tem stalled interval mas não está configurado explicitamente.
Recomendação: Configurar stalledInterval no BullMQ, implementar job recovery que reenfileira jobs em PROCESSING há mais de X minutos.

Demo Mode no Frontend

Token JWT hardcoded em código — inseguro em produção

BAIXO
Probabilidade: 1
Impacto: 3
Score: 3
Status: Parcial
Situação: App.tsx força token demo no localStorage em inicialização.
Recomendação: Remover demo mode em produção. Implementar OAuth flow real.
