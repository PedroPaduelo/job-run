⚠️ Nota sobre Timeout do Webhook

O timeout aborta apenas a requisicao fetch. O BullMQ continuara o retry se o erro for lancado (timeout corta conexao, lancando AbortError).

6 Sistema de Callback

✅ Callback de Sucesso (COMPLETED)

POST {callbackUrl} { "jobId": "job_xxx", "status": "COMPLETED", "result": {...}, "duration": 1234, "attempts": 1 }

Enviado apos webhook retornar 2xx, antes de retornar do processJob.

❌ Callback de Falha (DEAD)

POST {callbackUrl} { "jobId": "job_xxx", "status": "DEAD", "error": "Webhook failed...", "attempts": 3 }

Enviado pelo worker failed event, quando attempts >= maxAttempts.

⚠️ Falhas no callback nao afetam o job

Erros no callback sao apenas logados. O job ja tera seu status final definido (COMPLETED/DEAD) independentemente do callback.

10 Graceful Shutdown

SIGTERM

Sinal de terminacao (docker stop, systemctl)

SIGINT

Ctrl+C no terminal

shutdown()

Funcao comum para ambos

const shutdown = async () => { console.log('Shutting down workers...'); // Fecha todos os workers gracefully for (const worker of workers) { await worker.close(); } // Disconecta Prisma await prisma.$disconnect(); process.exit(0); }; process.on('SIGTERM', shutdown); process.on('SIGINT', shutdown);
ℹ️ worker.close() aguarda jobs ativos

O BullMQ nao aceita novos jobs apos close(), mas aguarda os em execucao finalizarem antes de encerrar.

11 Gerenciamento Dinamico de Workers

Polling de Novas Filas

O worker monitora o banco a cada 30 segundos para criar workers para novas filas ativas:

// A cada 30 segundos setInterval(async () => { const currentQueues = await prisma.queue.findMany({ where: { isActive: true } }); for (const queue of currentQueues) { const queueName = `jobs-${queue.organizationId}-${queue.name}`; const exists = workers.some(w => w.name === queueName); if (!exists) { console.log(`Starting new worker: ${queueName}`); const worker = new Worker(queueName, processJob, { connection: redis, concurrency: queue.concurrency || CONCURRENCY }); workers.push(worker); } } }, 30000);
