4. Pontos Críticos de Isolamento de Dados

⚠️ Pontos de Falha Identificados

🔴 1. Query Builder - Ausência de organizationId

Todo método de serviço deve receber organizationId como parâmetro e incluí-lo no where. Falha em adicionar expõe dados de outras organizações.

Exemplo correto:

prisma.job.findMany({
  where: { organizationId, ...filters }
})

🔴 2. Update/Delete sem organizationId no WHERE

prisma.job.update({ where: { id } }) sem organizationId permite modificar jobs de outras orgs.

Solução: Usar updateMany ou incluir org no where.

✅ 3. API Key Escopo por Organização

API Keys são criadas com organizationId vinculado. No middleware (server.ts:82-102), a chave é validada contra o banco e apenas o organizationId da chave é usado.

request.organizationId = apiKey.organizationId

✅ 4. JWT Validação e Extração

O JWT é verificado com JWKS e o organizationId é extraído do payload ou do header X-Organization-Id. A validação impede tokens inválidos.

🔴 5. BullMQ Queue Names

As queues do BullMQ usam o padrão jobs-${organizationId}-${queue.name} (worker.ts:151). Isso isola filas Redis por organização, mas se um erro de nomeação ocorrer, jobs podem vazar.

✅ 6. Foreign Keys com CASCADE

Deletar uma organização remove automaticamente: ApiKeys, Queues, Jobs, OrganizationMembers, AuditLogs (SET NULL). Protege contra órfãos.

⚠️ 7. Soft Delete vs Hard Delete

isActive é usado para desativar (soft delete) queues e API keys, mas Jobs usam status. A organização em si não tem campo isActive - deletar remove tudo (hard delete via CASCADE).

🔴 8. AuditLog inclui organizationId?

SIM! O logAudit() sempre recebe organizationId do contexto. Todos os logs de ação incluem a organização.

5. Sistema de Permissões (hasPermission)

Função hasPermission() (auth.ts:385-398)

export function hasPermission(
  userPermissions: string[],
  requiredPermission: string
): boolean {
  // Admin tem tudo
  if (userPermissions.includes('*')) return true;

  // Wildcard (ex: jobs:* inclui jobs:create)
  const [resource, action] = requiredPermission.split(':');
  if (userPermissions.includes(`${resource}:*`)) return true;

  // Permissao exata
  return userPermissions.includes(requiredPermission);
}

Fluxo de Autorização nas Rotas

  1. Middleware onRequest (server.ts:71-122)
    • Autentica (API Key ou JWT)
    • Extrai organizationId
    • Define request.permissions
    • Loga audit trail
  2. Permission Check (server.ts:135-143)
    • requirePermission('jobs:create')
    • Verifica request.permissions
    • Registra denied em audit log
  3. Service Layer
    • Recebe organizationId da request
    • Inclui no where das queries Prisma
    • Não faz authorization adicional (confia no middleware)

⚠️ Vulnerabilidade Potencial

A autorização acontece antes de passar para os services, mas se uma rota esquecer de usar requirePermission(), qualquer permissão concedida no cockpit de auth terá acesso. Faltam proteções em nível de serviço.

6. Deleção de Organização e Cascade Effects

Foreign Keys com CASCADE

Tabela ON DELETE Descrição
OrganizationMember.userId CASCADE Deleta membros da organização
OrganizationMember.organizationId CASCADE Deleta membros da organização
ApiKey.organizationId CASCADE Deleta todas as API keys da org
Queue.organizationId CASCADE Deleta todas as filas da org
Job.organizationId CASCADE Deleta todos os jobs da org
Job.queueId CASCADE Deleta jobs quando queue é deletada
JobLog.jobId CASCADE Deleta logs quando job é deletado
AuditLog.userId SET NULL Mantém log, mas sem referência ao usuário
AuditLog.organizationId SET NULL Mantém log, mas sem referência à org
Job.userId SET NULL Job permanece sem criador

Não há Soft Delete de Organização

O modelo Organization não possui campo deletedAt ou isActive. Deletar uma organização causa hard delete com CASCADE - todos os dados são removidos fisicamente do banco.

Recomendação: Adicionar campos para arquivamento/restauração, preservando dados históricos para compliance.

7. Checklist de Segurança do Multi-Tenancy

  • 🔴

    TODAS as queries incluem organizationId no WHERE

    Verificar cada método findFirst, findMany, update, updateMany, delete em todos os services. Incluir organizationId é obrigatório.

  • 🔴

    Nenhuma rota pública sem autenticação

    Todas as rotas (exceto /health, /ready, /docs) passam pelo middleware que valida token e extrai organizationId.

  • 🔴

    API Keys são validadas por organizationId

    No update/delete de API Key: prisma.apiKey.findFirst({ where: { id, organizationId } }). Não usar apenas id.

  • 🟡

    AuditLog registra organizationId em todas as ações

    Verificar que logAudit() sempre inclui organizationId extraído de request.organizationId. A função já suporta空值, mas deve ser preenchida.

  • 🟢

    API Key escopo por organização (IMPLEMENTADO)

    ApiKey possui organizationId FK. No middleware (server.ts:84-102), o organizationId é extraído da chave e usado na request.

  • 🟢

    Foreign Keys com CASCADE/SET NULL (IMPLEMENTADO)

    Todas as relações tem proteção contra órfãos. Deleção de organização limpa dados relacionados.

  • 🟢

    BullMQ Queue Names incluem organizationId (IMPLEMENTADO)

    jobs-${organizationId}-${queue.name} isola filas Redis por tenant.

  • 🟡

    Rate Limiting por organização (parcial)

    fastify-rate-limit usa request.organizationId como chave (server.ts:46), mas ainda pode ser por IP. Ideal: combinar IP + org.

  • 🔴

    Worker processa jobs com validação de organizationId

    VULNERABILIDADE: O worker (worker.ts) não valida se o job pertence à organização. Se um atacante injetar um jobId de outra org no Redis, o worker processará sem verificação.

  • 🟡

    Validate queue ownership before operation

    No queueService.pause()/resume(), usar getById(organizationId, id) e checar resultado antes de executar ação (parcialmente correcto).

  • 🟢

    API Key retorna key apenas na criação

    apiKeyService.create() retorna a chave completa uma única vez. Listagem e get omitem o campo key.

  • 🟡

    Organization Sync do Auth Service

    buildAuthenticatedUser() sincroniza usuário, organização e membership. Isso garante que dados do auth service são refletidos localmente. Mas falha silenciosa se auth service indisponível?

  • 🔴

    Race conditions em create/update

    Sem transações ou locks, um usuário pode modificar um recurso se adivinhar o id. Exemplo: PATCH /queues/:id se o id pertence a outra org, updateMany retorna count === 0 (seguro), mas create pode falhar se validação搭车.

8. Resumo e Recomendações

🟢 Pontos Fortes

  • Padronização clara: organizationId sliceado no middleware
  • API Key suporta permissões granulares por tenant
  • RBAC robusto com OrganizationMember.permissions array
  • Auditoria via AuditLog com FK organizationId
  • Foreign keys protegem contra orfãos
  • BullMQ usa naming convention com orgId

🔴 Vulnerabilidades

  • Worker não valida ownership do job antes de processar
  • Falta proteção em service layer (confia 100% no middleware)
  • Pendência de soft delete para organização
  • Rate limit por IP não considera org
  • Possível race condition se ID colidir cross-tenant
  • Credenciais em variações de JWT payload inconsistente

🎯 Ações Recomendadas

  1. Worker validation: Antes de processar job, validar job.organizationId contra lista de organizações ativas.
  2. Service guards: Adicionar verificação de ownership em cada método de service como defesa em profundidade.
  3. Soft delete org: Adicionar deletedAt em Organization com restore.
  4. UUIDs: Garantir IDs usando UUIDv4 ou cuid com entropia suficiente para evitar colisões.
  5. Rate limit: Mudar para ${organizationId}:${ip}.
  6. Integration tests: Testes que tentam acessar dados cross-tenant devem falhar.
