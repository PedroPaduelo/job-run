Integracao Sentry

Arquivo: src/lib/sentry.ts

initSentry()

  • Checks SENTRY_DSN existence
  • Configure environment: NODE_ENV
  • TracesSampleRate: 10% em producao
  • beforeSend: filtra eventos no dev
// Inicializacao export function initSentry() { if (!SENTRY_DSN) { console.log('[Sentry] DSN not configured'); return; } Sentry.init({ dsn: SENTRY_DSN, environment: process.env.NODE_ENV, tracesSampleRate: NODE_ENV === 'production' ? 0.1 : 1.0, beforeSend(event) { if (NODE_ENV === 'development') { return null; // Nao envia } return event; }, }); }

captureError(error, context)

  • Adiciona contexto ao scope
  • Captura excecao no Sentry
  • Contexto inclui userId, orgId, etc.
export function captureError( error: Error, context?: Record<string, unknown> ) { if (context) { Sentry.setContext('additional', context); } Sentry.captureException(error); }

flushSentry()

  • Chamado no shutdown (SIGTERM, SIGINT)
  • Timeout: 2000ms
  • Garante envio de eventos pendentes
  • Usado antes do process.exit()
export async function flushSentry() { await Sentry.close(2000); }

setUser(user)

  • Define usuario no contexto Sentry
  • Inclui id e email
  • Null para limpar
export function setUser( user: { id: string; email?: string } | null ) { if (user) { Sentry.setUser({ id: user.id, email: user.email }); } else { Sentry.setUser(null); } }

Global Error Handler (Fastify)

Arquivo: src/server.ts (linhas 124-133)

fastify.setErrorHandler((error, request, reply) => { // 1. Capturar no Sentry com contexto captureError(error, { url: request.url, method: request.method, organizationId: request.organizationId, userId: request.userId }); // 2. Log no Fastify logger fastify.log.error(error); // 3. Log de auditoria if (request.organizationId) { await logAudit({ userId: request.userId, organizationId: request.organizationId, action: 'error.occurred', metadata: { url: request.url, method: request.method, message: error.message }, ipAddress: request.ip, userAgent: request.headers['user-agent'], status: 'FAILURE' }); } // 4. Determinar status code const statusCode = (error as any).statusCode || 500; // 5. Mensagem segura para producao const message = process.env.NODE_ENV === 'production' && statusCode === 500 ? 'Internal Server Error' : error.message; // 6. Enviar resposta reply.status(statusCode).send({ error: message }); });
Nota importante: O handler global sempre usa error.message para status 4xx (exceto 500 em production). Isso expoe mensagens de validacao em producao, mas oculta stack traces.

Graceful Shutdown

Tratamento de SIGTERM e SIGINT no servidor principal (src/server.ts) e worker.

Servidor HTTP

process.on('SIGTERM', async () => { console.log('SIGTERM received, shutting down...'); // 1. Flush eventos pendentes no Sentry await flushSentry(); // 2. Fechar conexoes Fastify await fastify.close(); // 3. Desconectar do banco de dados await prisma.$disconnect(); // 4. Sair gracefulmente process.exit(0); }); process.on('SIGINT', async () => { console.log('SIGINT received, shutting down...'); await flushSentry(); await fastify.close(); await prisma.$disconnect(); process.exit(0); });

Worker BullMQ

Arquivo: src/worker.ts (linhas 229-242)

const shutdown = async () => { console.log('Shutting down workers...'); // Fechar todos os workers for (const worker of workers) { await worker.close(); } // Desconectar do banco await prisma.$disconnect(); process.exit(0); }; process.on('SIGTERM', shutdown); process.on('SIGINT', shutdown);
Best practice implementada: O flushSentry() e chamado no shutdown, garantindo que eventos de erro pendentes sejam enviados ao Sentry antes da finalizacao.

Analise de Riscos (Security Review)

1. PII em Logs de Autenticacao

Risco

Logs de autenticacao (logAudit) podem conter PII (email, nome) nos metadados. Audit logs sao frequentemente armazenados por longos periodos.

Arquivos afetados

  • src/lib/auth.ts - logAudit (linhas 476-504)
  • src/server.ts - logAudit em error handler (linha 128)

Evidencia

// server.ts onRequest hook await logAudit({ userId: authUser.userId, // OK - nao e PII direto email: authUser.email, // PII! organizationId: orgId, action: 'user.auth.success', ipAddress: request.ip, userAgent: request.headers['user-agent'] });

Recomendacao

  • MEDIO - Remover email e nome dos audit logs
  • Manter apenas userId (nao reversivel) e orgId
  • Se necessario para debug, criptografar ou mascar

2. API Keys em Error Messages

Status: MITIGADO

OK As API keys sao tratadas como Bearer tokens e convertidas em apiKeyId que e armazenado no request. O error.message nunca inclui a API key.

Evidencia

// server.ts onRequest hook (linhas 81-97) const token = authHeader.substring(7); const apiKey = await prisma.apiKey.findUnique({ where: { key: token }, // Busca por hash include: { organization: true } }); request.apiKeyId = apiKey.id; // Armazena apenas ID

3. Sentry DSN em .env

Status: SEGURO

OK O Sentry DSN e uma chave publicaroteada. Seu vazamento nao compromete dados - apenas permite que outros enviem eventos para seu projeto Sentry.

Configuracao atual

# .env.example (linha 21) SENTRY_DSN=

Recomendacao

  • Sentry DSN e seguro em .env (nao e secret)
  • Na producao, configure rate limiting no Sentry project
  • Nao ha necessidade de .gitignore especial para este arquivo

4. Flush no Shutdown

Status: CORRETAMENTE IMPLEMENTADO

OK O flushSentry() e chamado antes do process.exit() em todos os handlers de shutdown.

Evidencia

// server.ts SIGTERM/SIGINT await flushSentry(); // 2s timeout await fastify.close();

Garantias

  • Timeout de 2000ms impede hang no shutdown
  • Eventos Sao enviados de forma assincrona
  • Falha no flush nao causa crash (try/catch implicito)

5. Stack Traces em Production

Status: CORRETAMENTE MITIGADO

OK O global error handler mascara mensagens de erro 500 em producao:

const message = process.env.NODE_ENV === 'production' && statusCode === 500 ? 'Internal Server Error' : error.message; // 4xx mantem mensagem

Erros 4xx (validation, auth, permission, not found) mantem suas mensagens por serem uteis para o cliente.

6. Contexto no Sentry: userId e orgId

Avaliacao

O sistema captura userId e organizationId como identificadores. Estes nao sao PII diretamente (sao IDs), mas podem ser usados para correlacao com outros sistemas.

captureError(error, { url: request.url, method: request.method, organizationId: request.organizationId, userId: request.userId });

Recomendacao

  • IDs sao aceitaveis - nao sao PII
  • Certificar que emails nao escapam via error.message
  • Considerar adicionar apiKeyId para rastrear chaves problematicas

7. Erros de Worker sem Sentry

Risco: MEDIO-ALTO

O src/worker.ts NAO usa Sentry. Erros de webhook falham e sao apenas logados no console e no jobLog da tabela local.

Evidencia

// worker.ts - catch block (linhas 118-134) } catch (error) { const errorMessage = error instanceof Error ? error.message : 'Unknown error'; // Apenas log no console e tabela local console.error(`Job ${jobId} failed:`, errorMessage); await prisma.jobLog.create({ ... }); throw error; // BullMQ handler retry }

Consequencias

  • Erros de webhook nao sao reportados ao Sentry
  • Sem stack trace completo
  • Dificulta debugging de falhas em producao

Recomendacao

  • CRITICO: Adicionar captureError() no catch do worker
  • Incluir contexto: jobId, queueName, webhookUrl
  • Considerar rate limiting (muitos erros de webhook podem flood Sentry)

Recomendacoes de Melhoria

1. Adicionar Sentry ao Worker

Importar e usar captureError no catch block do worker, com contexto do job:

import { captureError } from './lib/sentry.js'; // No catch block do worker } catch (error) { await captureError(error, { jobId, queueName, webhookUrl, attempt: job.attemptsMade }); throw error; }

2. Sanitizar Audit Logs

Remover PII (email, nome) do logAudit. Manter apenas identificadores nao reversiveis.

3. Add Error Codes

Adicionar campo code nas respostas de erro para internacionalizacao:

reply.status(401).send({ error: 'Invalid token', code: 'AUTH_INVALID_TOKEN' });

Padroes de Error Classes

Recomenda-se criar classes de erro customizadas para melhor categorizacao:

// Erros de validacao export class ValidationError extends Error { constructor(message: string, public cause?: Error) { super(message); this.name = 'ValidationError'; } } // Erros de permissao export class PermissionError extends Error { constructor(permission: string) { super(`Missing permission: ${permission}`); this.name = 'PermissionError'; (this as any).statusCode = 403; } } // Erros deNotFound export class NotFoundError extends Error { constructor(resource: string) { super(`${resource} not found`); this.name = 'NotFoundError'; (this as any).statusCode = 404; } }
